# 자료구조: 큐 (Queue) 시각화 기획

### 0. 기본 정보

- **대상 이름**: 큐 (Queue, FIFO Queue)
- **도메인/카테고리**: 컴퓨터 과학 기초 / 자료구조
- **대상의 본질**: 양끝에 서로 다른 역할을 부여해 "들어온 순서 = 나가는 순서" 를 기계적으로 보장하는 선형 자료구조. 새 원소는 언제나 뒤(rear)로만 붙고, 제거는 언제나 앞(front)에서만 일어난다.

---

### 1. 시그니처 행동

**뒤로만 쌓이고 앞으로만 빠진다 — 가장 오래 기다린 것이 가장 먼저 떠나는 기계.**

양 끝이 비대칭이다. 뒤쪽 게이트는 "받기만 하고 내보내지 않음", 앞쪽 게이트는 "내보내기만 하고 받지 않음". 중간에 섞이지 않으므로 입장 순서와 퇴장 순서가 정확히 일치한다.

---

### 2. 시각적 은유

- **컨베이어 벨트** (공장 조립 라인 / 공항 수하물) — 양끝 비대칭 게이트, 일방향 흐름, "내가 먼저 얹혔으면 내가 먼저 떨어진다" 의 기계적 FIFO 를 가장 직접적으로 체화. 큐의 알고리즘적·기계적 맥락과도 잘 맞는다.
- **나이테 / 숙성 그라디언트** (시간 경과의 자연 현상) — 각 원소가 벨트 위에서 얼마나 오래 기다렸는지를 "색의 숙성도" 로 번역. 단순 순서가 아닌 "시간을 견뎌낸 정도" 를 한눈에 드러낸다.
- **출입 스탬프** (극장 대기표 / 은행 번호표) — 새로 들어오는 원소에 자동 증가하는 입장 번호를 찍어준다. 이 번호가 퇴장 순서와 정확히 일치하는 것을 사용자가 직접 확인하게 해 FIFO 를 "증거" 로 각인시킨다.

---

### 3. 결정적 순간

**dequeue 가 호출된 바로 그 프레임 — 앞 게이트에서 선두가 경계를 넘어 슬라이드 아웃 되는 동시에, 나머지 전체 줄이 하나의 강체처럼 같은 속도로 한 칸 앞으로 미끄러지는 순간.**

- **전**: 컨베이어 위에 블록 n 개가 입장 번호를 단 채 일렬로 정렬. front 블록(가장 낮은 번호)이 짧은 고리 하이라이트로 "다음 떠날 것" 표시.
- **중**: front 블록이 왼쪽 게이트 바깥으로 이탈. 이탈과 **동시에** 뒤에 있던 모든 블록이 같은 오프셋만큼 왼쪽으로 병진 이동. 프레임 차이가 없어야 "한몸으로 움직임" 이 읽힌다.
- **후**: 맨 앞 자리는 원래 2 번 블록이 차지, 맨 뒷자리는 비어있다. 이탈한 front 는 경계 바깥 꼬리 영역에서 흐려지며 기록으로 남는다. 입장 카운터는 그대로(총 들어온 개수는 불변), 크기 게이지만 1 감소.

**시각적 강조 장치**
- 정지 프레임에서도 "이것이 dequeue" 임을 알아보려면: (1) 왼쪽 게이트의 OUT 화살표 섬광, (2) 경계 외부에 반투명으로 잔상이 남는 front 블록, (3) 전체 블록이 같은 방향으로 살짝 기울어진 모션 블러/트레일.
- enqueue 의 결정적 순간은 **반대편** 에서만 일어난다는 비대칭을 시청자가 한 눈에 대조할 수 있도록, enqueue 애니메이션은 오른쪽 게이트에서 위→아래 드랍 방식으로 벨트와 **직각 궤적**을 쓴다. dequeue 의 수평 이탈과 대비되도록.

### 4. 다른 대상과의 구별점

- **vs 스택 (Stack, LIFO)**: 스택은 한쪽 끝만 활성이다. 같은 자리에서 쌓고 같은 자리에서 뽑는다. 시각적으로: 스택은 한쪽이 **막혀 있는 용기**(접시/탑) 이며 활동이 한 점에 집중된다. 큐는 **양쪽이 뚫린 터널/벨트** 로, 입장과 퇴장이 서로 **다른 공간 위치** 에서 일어난다. 한 프레임만 봐도 "활동 지점이 두 개 vs 하나" 로 구분된다.

- **vs 우선순위 큐 (Priority Queue / Heap)**: PQ 는 삽입 시 힙 재정렬이 일어나 트리 구조가 요동친다. 순서는 도착이 아닌 우선도로 결정된다. 시각적으로: PQ 는 **트리** 가 꿈틀거리며 원소가 중간으로 파고든다. 큐는 삽입이 **항상 끝자리** 에 딱 얹혀지고 중간은 건드리지 않는다. 입장 번호와 퇴장 번호가 PQ 는 어긋나고, 큐는 반드시 일치한다.

- **vs 리스트 (Array / Linked List)**: 리스트는 임의 인덱스로 접근·삽입·삭제 가능. 시각적으로: 리스트는 중간 아무 곳이나 가리킬 수 있는 **인덱스 포인터/쐐기** 가 보인다. 큐는 중간부가 시각적으로 **봉인**(회색 처리, 상호작용 불가 톤) 되어 있고 오직 양끝 게이트에서만 이벤트가 일어난다.

### 5. 달성해야 할 시각 요소

- **양끝 비대칭 게이트**
  - **설명**: front(OUT) 와 rear(IN) 가 서로 다른 역할을 가진 "문" 으로 명시적으로 드러난다. 사용자가 어느 쪽이 입구이고 출구인지 한눈에 인지해야 한다.
  - **구현 힌트**: 벨트 좌측에 `◀ OUT` 라벨 + 붉은 기운(퇴장 방향 색), 우측에 `IN ▶` 라벨 + 녹/청 기운(입장 방향 색). 게이트 사이에는 좌→우 흐름을 암시하는 얕은 빗살 패턴(고정 각도)이 벨트 표면에 새겨져 있다. enqueue 시 우측 게이트만 번쩍이고, dequeue 시 좌측 게이트만 번쩍인다 — 활동의 공간 분리.

- **나이 그라디언트**
  - **설명**: 블록의 색/채도가 "입장 후 얼마나 버텼는지" 를 표현. front 에 가까울수록 "숙성", rear 에 가까울수록 "신선".
  - **구현 힌트**: 각 블록은 입장 시 채도 1.0 의 밝은 tint 로 시작. 이후 다른 블록이 enqueue 될 때마다(또는 벨트 tick 마다) 채도 -0.08 / luminance -0.05 씩 감산하여 누적. 하한 채도 0.35. dequeue 되어 꼬리 영역으로 빠지면 회색조로 수렴. 인덱스 기준이 아니라 "개체별 생명 주기" 기준이므로, 같은 위치라도 나중에 들어온 블록은 더 밝다.

- **입장 스탬프 단조증가**
  - **설명**: 각 블록 상단에 `#n` 입장 번호가 찍혀있고, 퇴장 순서의 번호가 `#1, #2, #3, …` 로 정확히 단조증가함을 사용자가 관찰할 수 있어야 한다.
  - **구현 힌트**: enqueue 이벤트 시 현재 totalEnqueued 를 카운트 업 후 그 값을 스탬프로 고정 바인딩. 블록이 이동하거나 나이가 변해도 스탬프 값은 절대 바뀌지 않음. 좌측 경계 바깥 꼬리 영역에 "막 퇴장한 번호들" 을 최근 3 개 나열해 `#1  #2  #3` 의 순서가 어긋나지 않음을 **증거** 로 노출.

- **동기 시프트 (전체가 한몸으로)**
  - **설명**: dequeue 시 front 이탈과 나머지 전체의 한 칸 전진이 **동일 프레임**에서 일어나야 한다. 각 블록이 제각기 다른 타이밍으로 움직이면 FIFO 의 "한 방향 결정성" 이 훼손된다.
  - **구현 힌트**: 한 번의 transition(모든 블록의 x 좌표를 동시에 변경)으로 묶어서 발행. 개별 블록별 delay / stagger 절대 금지. 이동 곡선은 ease-out(앞은 빠르고 뒤는 서서히 감속) 으로 "벨트가 한번 당겨졌다가 멈추는" 물리감.

- **꼬리 로그 (경계 밖 잔상)**
  - **설명**: 막 dequeue 된 블록은 왼쪽 경계 바깥에서 3 개까지 흐리게 남아있다가 뒤로 밀려 사라진다. 이 꼬리가 "과거의 순서" 를 보여주는 타임라인 역할.
  - **구현 힌트**: 경계 밖 영역 3 칸, 각 칸 opacity `0.45 / 0.25 / 0.10`. 새로운 dequeue 마다 한 칸씩 좌측으로 밀려나고 가장 오래된 것은 opacity 0 으로 사라짐. 블록 원래 색 유지하되 회색 합성으로 "과거성" 표현.

### 6. 시각 구성

**중심 시각 — 가로 컨베이어 벨트**

화면 중앙을 가로지르는 긴 직사각형 벨트. 내부에 여러 개의 블록이 오른쪽에서 들어와 왼쪽으로 퇴장하는 일방향 흐름을 따라 놓여 있다. 벨트 표면에는 방향을 암시하는 빗살 패턴 (좌→우 향하는 45° 대각선 그루브) 이 **고정된 상태로** 새겨져 있다 (불필요한 움직임은 배제). 양끝은 투명하게 열려있어 블록이 경계를 넘어 들어오고 떠나는 장면이 그대로 드러난다.

**보조 요소**

- **입장 번호 스탬프**: 각 블록 상단에 붙는 `#n` 라벨. 절대 변경되지 않는 "출생 증명서".
- **나이 채도**: 블록 색상 자체가 대기 시간을 표현.
- **양끝 게이트 라벨**: 좌측 `◀ OUT` (퇴장 방향), 우측 `IN ▶` (입장 방향). 국소 섬광으로 활동 지점을 명시.
- **입장 카운터 / 크기 게이지**: 벨트 상단 우측에 `들어온 총 개수 n_in` 과 `현재 크기 n` 을 숫자로 표시. bounded 모드에서는 `n / capacity` 로 최대치도 표시.
- **꼬리 로그**: 좌측 경계 바깥 3 칸. 직전에 떠난 블록들이 fade-out 상태로 정렬.
- **연산 로그 패널**: 벨트 하단 전폭. 시간 순서대로 `t=3  dequeue() → #2` 같은 텍스트 기록이 흐른다. 최근 5~10 줄 유지.
- **컨트롤 바**: 최하단. 재생/일시정지/단계전진/리셋 + 속도 슬라이더 + (수동 모드에서) enqueue 값 입력창 / Enqueue / Dequeue / Peek 버튼.

**요소 간 관계**

- 벨트와 양끝 게이트가 시각의 80% 를 차지. 큐 **자체** 가 주인공.
- 입장 카운터·크기 게이지는 벨트의 "현재 상태" 를 수치로 보조 (글자로만).
- 꼬리 로그는 벨트와 같은 수평 축 위에 있어 "벨트의 과거" 가 공간적으로 왼쪽, "벨트의 미래(입장 대기)" 가 오른쪽임이 자연스럽게 읽힌다.
- 연산 로그 패널은 벨트의 움직임과 텍스트가 짝지어져 "이 움직임이 어떤 연산의 결과" 인지 쌍으로 확인 가능.

### 7. 레이아웃

```
┌─────────────────────────────────────────────────────────────────────┐
│ 큐 (Queue — FIFO)                       총 입장: 7    크기: 4 / 10  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   #2 #3 #4       ┃◀OUT ┃ [#5 ][#6 ][#7 ][#8 ] ┃ IN▶┃                │
│   (꼬리 fade)    ╰──좌측 게이트─╯   중앙 벨트   ╰─우측 게이트─╯        │
│                                                                     │
├─────────────────────────────────────────────────────────────────────┤
│ 연산 로그                                                            │
│ t=0  enqueue(A)  → #1                                                │
│ t=1  enqueue(B)  → #2                                                │
│ ...                                                                  │
│ t=6  dequeue()   → #4                                                │
├─────────────────────────────────────────────────────────────────────┤
│ ▶ 재생  ⏭ 단계  ⏸ 정지  ↺ 리셋    [속도ーーー●ーー]  1x              │
│ 값: [___]  [Enqueue]  [Dequeue]  [Peek]                              │
└─────────────────────────────────────────────────────────────────────┘
```

- **상단**: 타이틀 + 수치 요약.
- **중단(가장 큰 비중)**: 왼쪽 꼬리 로그 → 좌측 게이트 → 벨트 본체 → 우측 게이트 → (우측 여백은 enqueue 드랍 예비 공간).
- **하단 1**: 연산 로그. 스크롤 가능.
- **하단 2 (컨트롤 바)**: 표준 재생 컨트롤 + 수동 연산 버튼.

방향은 **공간 = 시간** 대응이 강제된다: 왼쪽으로 갈수록 **과거**, 오른쪽으로 갈수록 **미래(아직 오지 않은 것)**. 이 축을 어기는 요소는 배치하지 않는다.

### 8. 진행과 이벤트

**주요 진행 단계**

1. **초기 상태**: 큐에 initialValues 에 지정된 블록들이 이미 배치되어 있음 (또는 빈 큐). 각 블록은 `#1, #2, …` 순으로 스탬프가 찍혀 있음.
2. **enqueue(v) 이벤트**:
   - (전 100ms) 우측 게이트 바깥 상단에 새 블록이 "떨어질 준비" 상태로 나타남. 스탬프 번호 즉시 주입.
   - (중 200ms) 블록이 위→아래로 드랍되며 벨트 끝자리에 안착. 우측 게이트 테두리에 IN 섬광.
   - (후) 크기 게이지 +1, 총 입장 +1. 연산 로그에 `enqueue(v) → #n` 한 줄 추가.
3. **dequeue() 이벤트**:
   - (전 100ms) front 블록 외곽에 짧은 고리 하이라이트 (곧 떠날 표시).
   - (중 240ms) 좌측 게이트에 OUT 섬광 + 전체 블록 **동기 좌측 시프트**. front 는 경계 바깥으로 이탈해 꼬리 로그 첫 칸에 안착. 다른 모든 블록은 한 칸 전진.
   - (후) 크기 게이지 -1. 연산 로그에 `dequeue() → #n` 한 줄 추가.
4. **peek() 이벤트**: front 블록이 2 회 pulse (확대→복귀). 연산 로그에 `peek() → #n` 기록. 구조 변화 없음.
5. **overflow** (bounded 모드에서 capacity 도달 후 enqueue): 우측 게이트가 붉게 번쩍이며 흔들림. 새 블록은 나타나지 않음. 연산 로그에 `enqueue(v) → overflow` 기록.
6. **underflow** (빈 큐에서 dequeue): 좌측 게이트가 한 번 흔들리고 에러 색으로 깜빡. 연산 로그에 `dequeue() → underflow` 기록.
7. **나이 진행 (tick)**: enqueue 또는 dequeue 이벤트가 있을 때마다, 큐에 남아있는 모든 블록의 "나이" 가 1 증가. 색 채도/luminance 가 단계적 감소.

**시각 변화 규칙**

- 모든 블록의 수평 이동은 한 프레임에 동기화 (개별 stagger 금지).
- enqueue 는 **수직 드랍** (위→아래), dequeue 는 **수평 이탈** (오른쪽→왼쪽). 궤적을 직교로 만들어 두 연산이 시각적으로 섞이지 않게 한다.
- 꼬리 로그의 이동도 dequeue 이벤트와 동기화: 새 이탈 블록이 칸 1 에 들어올 때, 기존 칸 1 은 칸 2 로, 칸 2 는 칸 3 으로, 칸 3 은 소멸한다 (모두 같은 프레임).

**사용자 제어 지점**

- **재생 / 일시정지 / 단계전진 / 리셋** (표준 컨트롤).
- **속도 슬라이더**: 0.25x ~ 4x. 드랍·시프트·꼬리 이동 애니메이션 시간 배율.
- **수동 모드**: 시나리오 자동 재생을 중단하고, 사용자가 `값 입력 + Enqueue` 나 `Dequeue / Peek` 버튼을 눌러 개별 연산을 호출 가능. 이 모드에서는 연산 로그가 사용자의 입력 기록이 된다.
- **시나리오 선택 (옵션)**: "빈 큐에서 시작", "소량 프리셋", "overflow 시연" 등 프리셋 시나리오 드롭다운.

### 9. 초기 상태와 파라미터

**초기 데이터**

```jsonc
{
  // 초기 큐 상태 (enqueue 된 순서대로)
  "initialValues": ["A", "B", "C"],

  // bounded 모드의 최대 크기. null/생략 = 무한.
  "capacity": 10,

  // 자동 재생 시나리오: enqueue / dequeue / peek 연산 목록.
  // 각 항목은 { op: 'enqueue'|'dequeue'|'peek', value?: string }.
  "scenario": [
    { "op": "enqueue", "value": "D" },
    { "op": "enqueue", "value": "E" },
    { "op": "dequeue" },
    { "op": "enqueue", "value": "F" },
    { "op": "peek" },
    { "op": "dequeue" },
    { "op": "dequeue" },
    { "op": "enqueue", "value": "G" },
    { "op": "dequeue" },
    { "op": "dequeue" }
  ]
}
```

**파라미터**

- `capacity: number | null` — 큐 최대 크기. null 이면 무한. bounded 기능(overflow 시각화)은 capacity 가 숫자일 때만 활성.
- `initialValues: string[]` — 시작 시점에 이미 큐에 있는 원소들. 순서 = enqueue 된 순서. 각 원소는 자동으로 `#1`, `#2`, … 스탬프 부여.
- `scenario: Operation[]` — 자동 재생 시 실행할 연산 목록. 수동 모드에서는 사용되지 않음.
- `features: ('bounded' | 'aging-gradient' | 'tail-log')[]` — 선택적 시각 장치 활성화.
  - `bounded`: capacity 가 주어졌을 때 overflow 시각화를 활성.
  - `aging-gradient`: 나이 그라디언트 효과 on/off (기본 on).
  - `tail-log`: 꼬리 로그 3 칸 표시 on/off (기본 on).
- `maxTailEntries: number` — 꼬리 로그 유지 개수. 기본 3.

## 부록: 기획 메모

### 왜 컨베이어 벨트를 택했나

스택은 "접시 쌓기" 라는 강한 기본 은유가 대중 인지에 고정되어 있지만, 큐는 그 정도로 단일한 은유가 정착되어 있지 않다. "줄 서기" 가 흔히 쓰이지만 이는 인간 사회적 맥락 (줄 서는 사람의 감정, 새치기 등) 이 섞여 있어 자료구조의 **기계적·결정론적 성격** 과 맞지 않는다. 컨베이어 벨트는 (1) 양끝 게이트의 역할이 구조적으로 분리, (2) 기계적 결정론 (벨트가 움직이면 위의 물건은 반드시 같이 움직임), (3) 공항 수하물·공장 조립 라인 등 대부분의 학습자가 이미 본 이미지, 이 세 가지를 모두 만족한다.

### 나이 그라디언트가 왜 필요한가

큐를 단순 "인덱스 줄" 로만 그리면 FIFO 의 본질 중 "시간이 흐른다" 가 보이지 않는다. front 가 "가장 오래된" 이라는 사실은 인덱스가 아니라 **대기 시간** 에서 나오는 것인데, 이를 색으로 번역하면 학습자가 "이 친구는 오래 기다린 친구구나" 를 즉시 체감한다. 이는 BFS 의 queue-display 같은 보조 뷰에서는 굳이 필요없지만, 큐 **자체** 를 주인공으로 내세울 때는 핵심 장치다.

### 다른 대상과 혼동되지 않는 핵심 시그널

한 프레임 정지 상태에서 "이건 큐다" 를 알아볼 수 있게 하려면 다음 세 요소가 동시에 보여야 한다: (1) 좌우 양끝에 **다른** 역할의 게이트, (2) 블록에 단조증가 **입장 번호** 스탬프, (3) 왼쪽 경계 밖 **꼬리 로그** 의 번호가 큐 내부 맨 앞 번호와 연속 (`#1 #2 #3 | #4 #5 #6`). 이 세 가지가 갖춰지면 스택·PQ·리스트 어느 것과도 혼동될 수 없다.

### 축소·보류 판단

- **원형 큐 (Circular Queue) 변형**: 이 기획은 선형 FIFO 큐만 대상으로 한다. 원형 큐는 별도 facet(`datastructure-circular-queue`) 로 분리한다. 컨베이어 벨트 은유가 자연스럽게 원형 루프로 확장 가능.
- **양쪽에서 넣고 빼는 deque**: 마찬가지로 별도 facet. 이 기획의 "양끝 비대칭" 이 deque 에서는 "양끝 대칭" 으로 반전되므로 혼동 방지를 위해 별도 설계 필요.
- **우선순위 큐**: 개념이 완전히 다르므로 별도 facet. 힙 트리 시각화가 주인공이 되어야 한다.
- **capacity 기본값**: 무한 큐가 교육적으로 더 단순해 기본 시나리오는 capacity = null 로 가되, 샘플 파라미터에는 capacity = 10 을 넣어 bounded 모드의 존재를 드러낸다.
- **벨트 표면 움직임 애니메이션**: "벨트가 천천히 흐르는 패턴" 은 아름답지만 사용자의 주의를 블록 대신 배경으로 끌고 갈 위험이 있다. 기획 본문에서는 **고정 빗살 패턴** 으로 제한 — 방향만 암시하고 움직이지는 않게 한다. (구현 단계에서 사용자 테스트 후 재검토 가능.)

### 구현 단계로 넘길 판단 보류 사항

- 연산 로그 영역의 고정 높이 vs 자동 확장 — 작은 화면에서는 3~5 줄만 노출하고 스크롤.
- 수동 모드의 UI 위치 — 컨트롤 바에 통합 vs 별도 패널. 기획에서는 통합으로 제시했으나 복잡해지면 분리 가능.
- `scenario` 포맷이 기존 FACET IR / FacetJson 구조와 어떻게 정합할지 — 구현 단계에서 FACET 의 알고리즘 개념 (`AlgorithmFn` + `ctx.emit`) 으로 변환 검토.
- 벨트 폭과 블록 개수의 동적 관계 — capacity 가 큰 경우 블록이 너무 작아지는 문제. 최소 블록 폭 하한 설정 또는 horizontal scroll 검토 (BFS 기획에서 줌/패닝을 배제한 것과 동일 논리로 가로 스크롤도 비선호, 대신 블록을 더 작게 축소하되 스탬프 가독성 유지하는 하한을 설정).
