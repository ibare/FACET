## 0. 기본 정보

- 이름: LRU 캐시 (Least Recently Used Cache, lru-cache)
- 도메인: system-design
- type: dataStructure
- 진행 모델: 입력 반응형 (`get(key)` / `put(key, value)` 외부 호출에 반응해 상태 전이)
- 본질 한 문장: 키-값 저장소 위에 "최근에 사용된 순서" 라는 추상을, hash map 의 키 좌표와 doubly linked list 의 순서 좌표가 같은 노드를 공유하는 방식으로 새겨, 모든 호출이 그 노드를 MRU 끝으로 끌어올리고 capacity 초과 시 LRU 끝을 떨어내는 자료구조다.

## 1. 시그니처 행동

호출 한 번이 들어오면 해당 키의 노드가 list 위 자기 자리에서 풀려나 MRU 끝으로 끌려 올라가고, 만약 이 호출이 새 키의 put 이며 이미 capacity 가 꽉 차 있었다면 같은 순간 LRU 끝의 노드가 list 와 hash 양쪽에서 동시에 사라진다.

## 2. 시각적 은유

표준 메타포 두 좌표계 공유를 정교화한다.

- **위쪽 영역 — hash map**: 키 슬롯 격자. 각 슬롯은 "key" 라벨과 함께 아래쪽 list 의 한 노드로 향하는 항시적 연결선을 가진다. hash 의 값이 곧 list 노드 포인터라는 invariant 가 화면 내내 유지된다.
- **아래쪽 영역 — doubly linked list**: 가로로 늘어선 노드 박스. **왼쪽 끝이 LRU, 오른쪽 끝이 MRU** 로 화면 내내 일관 유지 (R3 / R6 관행 채택). 양 끝의 head/tail dummy 가 옅은 톤으로 상시 보이며 경계를 표시한다.
- **두 좌표계의 결합**: 세로축 = 두 자료구조 공존, 가로축 = 최근성 (왼쪽 LRU ↔ 오른쪽 MRU). 같은 노드가 위에서는 "key 좌표" 로, 아래에서는 "순서 좌표" 로 동시에 호명된다.
- **capacity 게이지**: list 영역 위쪽에 슬롯 수만큼의 칸이 그려진 카운터. 채워진 슬롯과 빈 슬롯이 시각적으로 구분되어, 여유/꽉 참/초과의 세 단계를 즉시 읽힌다.

창의적 비유는 일체 도입하지 않는다. hash + doubly linked list 의 표준 도식만 디테일과 일관성으로 강화한다.

## 3. 결정적 순간

**결정적 순간 A — promotion (hit 또는 기존 키 갱신)**
- 직전: 해당 노드가 list 중간 어딘가에 위치, hash 슬롯에서 연결선이 그 노드로 연결됨.
- 진행: (1) hash 키 슬롯이 점등되며 "lookup" 화살표가 list 노드를 가리킨다 → (2) 노드가 양쪽 이웃과의 연결을 끊고 (unlink) 살짝 떠오른다 → (3) 부드러운 호 (arc) 를 그리며 MRU 끝으로 이동해 재삽입된다.
- 강조 장치: 이동 중 노드의 윤곽 강조, 좌표 변환을 보여주는 잔상 (이전 위치 → 새 위치 의 호), 위쪽 hash 연결선이 함께 따라가며 끊기지 않음. **"읽기인데 쓰기" 라는 부수효과가 한눈에 잡히도록 get 과 put 의 promotion 단계를 동일 시각 언어로 처리한다.**

**결정적 순간 B — eviction (capacity 초과 put)**
- 직전: capacity 게이지가 꽉 참, list 가 capacity 만큼 노드로 채워짐.
- 진행: (1) 새 키 put 이 들어오면 capacity 게이지에 "+1 시도" 가 표시되며 한 박자 멈춤 → (2) **LRU 끝 노드가 아래로 떨어져 사라지고 동시에 위쪽 hash 슬롯도 비워진다** (양쪽 영역에서 동기 소멸) → (3) 새 노드가 MRU 끝에 삽입되며 새 hash 슬롯이 점등.
- 강조 장치: 떨어지는 노드와 새로 들어오는 노드를 같은 프레임에 배치해 "꽉 차서 누가 떨어진다" 의 긴장이 단일 프레임으로 압축된다. capacity 게이지의 여유 1 → 꽉 참 → 초과 시도 의 세 단계 임계가 색/숫자로 분리.

## 4. 다른 대상과의 구별점

- **FIFO 캐시**: 노드는 들어온 순서로만 정렬되고 promotion 호가 없다. get 호출이 list 를 흔들지 않는다. LRU 의 결정적 순간 A (get 의 부수효과로 인한 노드 이동) 가 FIFO 에는 없다.
- **LFU 캐시**: 정렬축이 "최근성" 이 아니라 "빈도" 다. 노드 옆에 빈도 카운터가 붙고 promotion 은 카운트 증가에 따른 재정렬이지 끝으로 끌어올리는 단순 동작이 아니다. LRU 의 가로축 일관성 (왼쪽 LRU → 오른쪽 MRU) 이 LFU 에서는 다중 버킷 그림으로 분해된다.
- **일반 doubly linked list**: hash map 영역 자체가 없다. 위쪽 좌표계가 부재하므로 두 좌표계 결합이라는 본질이 사라진다.
- **일반 hash map**: 아래쪽 list 영역이 없다. "최근성" 축이 부재하므로 promotion / eviction 의 시각 언어 자체가 정의되지 않는다.

## 5. 달성해야 할 시각 요소

1. **두 좌표계 공유 (Dual Coordinates)** — 설명: hash 영역의 키 슬롯과 list 영역의 노드가 같은 객체임이 항시 유지되는 invariant. 시각 표현: 위쪽 hash 슬롯 → 아래쪽 list 노드로 향하는 항시 가는 연결선. 노드가 promotion 으로 이동해도 연결선은 끊기지 않고 함께 휘어진다.
2. **MRU↔LRU 가로축 (Recency Axis)** — 설명: 왼쪽 끝 LRU / 오른쪽 끝 MRU 의 공간 규약과 list 의 정렬 의미. 시각 표현: list 영역 아래에 가로 축선, 좌측에 "LRU" 우측에 "MRU" 라벨 상시 표시. 노드 위치 자체가 최근성 인덱스를 의미한다.
3. **promotion 호 (Promotion Arc)** — 설명: 호출된 노드가 unlink → MRU 끝 재삽입으로 이동하는 두 단계 변형. get 의 쓰기성 부수효과를 시각화. 시각 표현: 노드의 이전 위치에서 MRU 끝까지의 부드러운 호 궤적, hash → list 의 lookup 화살표 점등이 promotion 의 첫 단계로 결합된다.
4. **capacity 임계 게이지 (Capacity Threshold)** — 설명: 여유 / 꽉 참 / 초과 시도 의 3단계 임계를 분리. 시각 표현: list 위쪽에 슬롯 수만큼의 칸 게이지. 채움 정도와 색조 (여유=중성, 꽉 참=경계, 초과=강조) 의 세 단계가 결정적 순간 B 의 한 박자 멈춤과 동기화.
5. **호출 트레이스 (Call Trace)** — 설명: 지금까지의 get/put 호출 이력 누적. 학습자가 현재 list 배열이 어떤 호출 history 의 결과인지 역추적 가능. 시각 표현: 화면 우측의 좁은 세로 스트립에 호출 인덱스 + `get(k)` / `put(k,v)` 텍스트가 위에서 아래로 누적, 가장 최근 호출이 강조.

## 6. 시각 구성

- **중심 시각**: 화면 중앙을 차지하는 두 좌표계 결합 도식. 위쪽 hash map 격자 + 아래쪽 doubly linked list + 둘을 잇는 항시 연결선. 가로축 LRU↔MRU 라벨이 list 영역 아래에 상시 표시. capacity 게이지가 list 위쪽에 가로로 얹힌다.
- **보조 요소 1 — 호출 트레이스 스트립**: 화면 우측의 좁은 세로 패널. 호출 history 누적.
- **보조 요소 2 — 입력 컨트롤**: 화면 하단의 좁은 가로 바. 키 선택 / get / put 액션 버튼 + 값 입력. 시각화의 일부라기보다 입력 반응형의 트리거 인터페이스.
- **요소 간 관계**: 입력 컨트롤이 호출을 발행 → 트레이스 스트립에 한 줄 추가 → 중심 도식에서 promotion 또는 eviction 결정적 순간 재생 → capacity 게이지가 갱신. 모든 변화가 한 호출 단위에 동기화된다.

## 7. 레이아웃

기본 폭 640~720px, 확장 880px 안에서 깨지지 않도록 단일 중심 + 보조 1~2개 한도를 지킨다.

```
┌────────────────────────────────────────────────┬──────────┐
│  [capacity 게이지: ■■■□  3/4]                   │  호출    │
│                                                 │  트레이스│
│  hash map (위쪽 영역, 키 슬롯 격자)               │          │
│   ┌──┐ ┌──┐ ┌──┐ ┌──┐                           │  i=5     │
│   │k1│ │k2│ │k3│ │  │                           │  put(k4) │
│   └┬─┘ └┬─┘ └┬─┘ └──┘                           │          │
│    │    │    │   (연결선이 list 노드로 항시 연결) │  i=4     │
│  doubly linked list (아래쪽 영역)                 │  get(k2) │
│   [head]─[n1]─[n2]─[n3]─[tail]                  │          │
│             ↑ promotion arc                     │  ...     │
│  ◀ LRU                              MRU ▶       │          │
├────────────────────────────────────────────────┴──────────┤
│  [key 선택 ▼]  [get]  [put value: __]                      │
└─────────────────────────────────────────────────────────────┘
```

- 폭 640~720px: 중심 도식 + 우측 트레이스 스트립 + 하단 컨트롤 바. capacity = 4 까지 list 노드 4개가 가로로 여유 있게 배치된다.
- 확장 880px: 노드 박스와 hash 슬롯 폭이 자연스럽게 늘고 promotion arc 의 곡률이 더 시원해진다. 보조 패널 추가는 없다.
- 좌우 4~6 패널 분할은 사용하지 않는다. 단일 중심 + 보조 1 (트레이스) + 입력 바 (컨트롤) 의 명확한 위계.

## 8. 진행과 인터랙션

입력 반응형이므로 자율 시간 진행은 없다. 외부 입력이 시각 변화를 유발한다.

- **사용자가 키를 선택하고 get 클릭 → hit 인 경우**: hash 슬롯 점등 → lookup 화살표 → 해당 노드 unlink → promotion arc → MRU 끝 재삽입. 트레이스 스트립에 `get(k) → v` 한 줄 추가. capacity 게이지는 변화 없음.
- **사용자가 키를 선택하고 get 클릭 → miss 인 경우**: hash 슬롯 영역에서 슬롯 부재가 짧게 표시 → list 변화 없음. 트레이스 스트립에 `get(k) → null` 추가.
- **기존 키에 put**: hit 와 동일한 promotion 호 + 노드 안의 값 갱신. capacity 게이지 변화 없음.
- **새 키에 put, capacity 여유 있음**: 새 노드가 MRU 끝에 직접 삽입, 새 hash 슬롯이 점등, capacity 게이지 +1.
- **새 키에 put, capacity 초과 (결정적 순간 B)**: capacity 게이지가 "초과 시도" 단계로 한 박자 멈춤 → LRU 끝 노드가 list 와 hash 양쪽에서 동기 소멸 → 새 노드가 MRU 끝에 삽입.
- **트레이스 스트립의 한 줄 클릭** (보조): 해당 호출 시점의 list 상태를 강조 표시 (역추적용 회상). 자율 재생은 없음.

## 9. 초기 상태와 파라미터

- **capacity**: 4 (게이지 4 칸. 여유/꽉 참/초과의 3단계 임계가 자연스럽게 드러나는 최소치).
- **초기 키-값 시퀀스**: 데모 진입 시 다음 호출이 이미 적용된 상태로 시작한다.
  1. `put(k1, v1)`
  2. `put(k2, v2)`
  3. `put(k3, v3)`
  4. `get(k1)` → promotion 으로 k1 이 MRU 끝으로 이동
- **초기 list 배열** (왼쪽 LRU → 오른쪽 MRU): `[head]─[k2]─[k3]─[k1]─[tail]`. capacity 게이지 3/4 (여유 1).
- **초기 hash 슬롯**: k1, k2, k3 세 슬롯이 각각 list 의 해당 노드로 연결.
- **초기 트레이스 스트립**: 위 4개 호출이 i=1..4 로 누적, i=4 줄이 강조.
- **다음 호출 예시 (학습자가 즉시 시도하기 좋은 시나리오)**: `put(k4, v4)` → 여유 → 꽉 참 진입, `put(k5, v5)` → 결정적 순간 B (k2 가 LRU 로서 축출), `get(k3)` → promotion 호.

## 10. 시각화 안 텍스트

> LRU 캐시는 용량 제한이 있는 키-값 저장소에서 "가장 최근에 사용된 순서" 를 일급 정보로 유지한다. 위쪽 hash map 은 키 → 노드의 빠른 조회를, 아래쪽 doubly linked list 는 사용 순서를 담당하며, **두 영역은 같은 노드를 공유**한다. get 도 단순 조회가 아니라 노드를 MRU 끝으로 끌어올리는 쓰기성 행위라는 점에 주목하자.
>
> 학습자는 화면에서 다음을 본다: 호출 한 번이 발행될 때마다 해당 노드가 MRU 끝으로 끌려 올라가는 promotion 호, capacity 게이지가 꽉 찬 뒤 새 키가 들어올 때 LRU 끝 노드가 양쪽 영역에서 동시에 사라지는 eviction 의 결정적 프레임, 그리고 우측 호출 트레이스가 누적되며 현재 list 배열이 어떤 호출 history 의 결과인지 역추적되는 흐름.
