# 네트워크 계층 IP 라우팅 — 공통 강화 기획 (2A)

## 0. 기본 정보

- **대상 이름**: 네트워크 계층 IP 라우팅 (hop-by-hop forwarding)
- **도메인/카테고리**: 컴퓨터 네트워킹 / 네트워크 계층(L3) / 패킷 포워딩
- **대상의 본질**: 패킷이 한 라우터에서 다음 라우터로 한 칸씩 던져지며 목적지에 도달한다. 각 라우터는 자기 안의 라우팅 테이블에서 목적지 IP 와 가장 길게 일치하는 prefix 규칙을 골라 next hop 을 결정하고, 그 결정은 하나의 라우터 안에서만 유효하다.
- **이 기획의 접근**: "공통 강화". 학습자가 IP 라우팅 자료에서 이미 익숙해진 표준 시각 언어(노드+엣지 그래프, 움직이는 패킷, 라우팅 테이블 표, 결정 펄스, 헤더 패널)를 골격으로 채택하되, 표준이 흐릿하게 두는 결정의 "근거"를 디테일로 채운다.
- **참조한 레퍼런스**:
  1. **Kurose & Ross 동반 애니메이션** — 라우팅 테이블 옆 표 + step 진행의 정석 골격을 그대로 차용.
  2. **Cisco Packet Tracer** — Simulation Mode 에서 라우터 도착 시 incoming/outgoing 으로 색이 갈리는 hop 단위 멈춤 패턴.
  3. **Practical Networking — Subnetting/Routing 시리즈** — 32 비트 IP 위에 마스크 경계를 빨간 세로선으로 명시하는 비트 단위 정렬 디테일을 라우팅 테이블 행 highlight 와 결합.

## 1. 시그니처 행동

> **패킷이 라우터 위에서 잠깐 멈추면, 옆의 라우팅 테이블에서 가장 긴 prefix 가 매치되는 한 행이 빛나고, 그 행이 가리키는 출구 링크 색으로 패킷이 물든 뒤 다음 라우터로 떠난다.**

이 한 문장이 매 hop 마다 반복되는 것이 IP 라우팅의 시각적 정체성이다.

## 2. 시각적 은유

보고서의 공통 메타포 — **"노드+엣지 그래프 위를 움직이는 봉투"** — 를 채택한다. 다듬는 점은 두 가지.

- **봉투(패킷)는 단순한 점이 아니라 "수명 게이지가 붙은 카드"**. 카드 앞면에 dst IP 가 큰 글씨로, 옆구리에 TTL 게이지가 hop 마다 한 칸씩 줄어든다.
- **라우터는 단순한 동그라미가 아니라 "표를 들고 있는 게이트"**. 라우터 동그라미 옆에 작은 표 아이콘 그림자가 항상 따라다녀, "이 라우터는 자기만의 표를 가진다" 는 사실을 시각화 안에 늘 노출한다. 선택된 라우터에서만 표가 사이드 패널로 펼쳐진다.

## 3. 결정적 순간

각 hop 에서 일어나는 **3-스텝 micro-animation**. 표준 시각 언어의 골격을 그대로 따르되, 각 스텝의 정보 밀도를 높인다.

1. **도착 펄스 (≈ 200ms)** — 패킷이 라우터에 닿으면 라우터가 한 번 부풀었다 가라앉는다. 동시에 사이드 표가 활성화 라우터의 것으로 교체된다(fade swap).
2. **비트 매칭 펼침 (≈ 600ms)** — 사이드 표의 모든 prefix 행이 동시에 dst IP 의 비트 옆에 정렬된다. 매치되는 비트는 초록, 안 맞는 비트는 회색으로 표시되고, **매치 길이를 가로 막대 길이로 환산**해 행 우측에 그린다. 가장 긴 막대가 한 박자 늦게 골드로 변하며 "이 행이 이긴다" 를 선언.
3. **출구 색 입힘 + 출발 (≈ 300ms)** — 선택된 행의 outgoing 인터페이스 색이 패킷 카드 테두리에 입혀지고, 같은 색의 링크가 굵어진다. 패킷이 그 링크 위로 미끄러져 나간다. 떠나는 순간 TTL 게이지가 한 칸 줄어든다.

이 1.1 초의 micro-animation 이 매 hop 마다 동일한 리듬으로 반복되어, 학습자가 "라우팅이란 이 박자다" 를 신체로 익힌다.

## 4. 다른 대상과의 구별점

| 가까운 대상 | 시각적 차이 |
|---|---|
| **L2 스위칭 (MAC 학습)** | 스위치는 prefix 매칭이 아니라 **정확한 MAC 일치**. 비트 매칭 펼침이 없고 표가 단순한 key→port 사전. 또한 학습으로 표가 자라는 별도 시각화가 필요. |
| **DNS 조회** | DNS 는 한 번의 조회로 IP 주소 한 개를 받아오는 lookup. hop-by-hop 의 박자 자체가 없고, 트리 내려가기(.→com→example.com) 가 시각의 중심. 패킷이 "이동" 하지 않는다. |
| **TCP 세그먼트 흐름** | TCP 는 같은 두 끝점 사이의 양방향 흐름과 시퀀스 번호/윈도우. 라우터를 지나는 점은 같지만, 시각의 주인공은 토폴로지가 아니라 두 끝점 사이의 **시퀀스 다이어그램**(시간축 X, 두 호스트 Y). |
| **지하철 환승 안내** | 출발 시 전체 경로가 정해진다. IP 라우팅 본 데모는 **출발 라우터가 끝까지 알지 못한다** 는 사실을 명시적으로 보인다 — 첫 라우터의 표에는 도착지가 "어디로 가야 한다" 고만 적혀 있고, 그 다음 라우터의 결정은 그 시점에 비로소 일어난다. |

특히 지하철 메타포와의 차별화는 시각 안에서 적극적으로 드러낸다 (8 절 진행 단계 마지막에 "출발 라우터가 본 경로 vs 실제 경로" 비교 컷).

## 5. 달성해야 할 시각 요소

1. **토폴로지 그래프** — 라우터 5~7 개와 호스트 2 개(src, dst) 의 노드+엣지 그래프. 라우터마다 고유한 짧은 이름(R1~R6) 과 인터페이스 색(라우터당 2~3 색).
   - 구현 힌트: `packages/core` 의 force-layout 을 쓰지 않고 **고정 좌표** 로 배치(임베드 폭 안정성). 호스트는 좌단/우단 끝, 라우터는 그 사이에 격자형.

2. **수명 카드 패킷** — `dst IP` 글씨 + `TTL` 가로 게이지(8칸, hop 마다 1칸 감소) + 출구 색 테두리. 카드 폭 ≈ 100px.
   - 구현 힌트: SVG `<g>` 한 단위로 묶고, 링크 위 슬라이드는 SMIL 대신 `transform: translate` 보간.

3. **활성 라우터의 라우팅 테이블 패널** — 4~6 행. 각 행: `prefix / mask` · `next hop` · `out iface`. lookup 시 행 옆 비트 정렬 영역 펼침.
   - 구현 힌트: 비트 정렬은 32 칸 격자로 그리되, 임베드 폭 절약을 위해 **점-사분 표기 옥텟별 8 비트 묶음에 옅은 구분선**. 매치 비트 초록, 비매치 회색, prefix 길이까지를 빨간 세로선.

4. **매치 길이 막대(Match Bar)** — 표의 각 행 우측 끝에 prefix 매치 비트 수를 가로 막대로 환산. 가장 긴 막대는 골드.
   - 구현 힌트: 0~32 비트를 0~64px 폭으로 매핑. 동일 prefix 길이 후보가 여럿이면 표의 위에 있는 행이 이기고, 배경에 옅은 동률 표시.

5. **헤더 inspect 미니 패널** — 패킷 카드를 클릭(또는 hover) 하면 src IP / dst IP / TTL 현재값 / 지나온 라우터 리스트가 펼쳐진다.
   - 구현 힌트: 카드 위에 부유하는 작은 카드(140×80px). 라우팅 테이블 패널과 동시에 펼쳐도 임베드 폭에 충돌하지 않는 위치(패킷 위쪽).

## 6. 시각 구성

- **중심 시각**: 토폴로지 그래프 위를 미끄러지는 수명 카드 패킷.
- **보조 요소들**: ① 활성 라우터의 라우팅 테이블 패널 ② 패킷 헤더 inspect ③ 진행 컨트롤 바(play/pause/step, hop 카운터) ④ 짧은 캡션 영역(2~4문장 개념 설명, 6 절 인용 텍스트 또는 그 단축).
- **요소 간 관계**:
  - 패킷이 라우터에 도착 → 그 라우터의 표가 패널에 swap 되어 펼쳐짐.
  - 패널의 한 행이 골드로 변함 → 그 행의 out iface 색이 패킷 카드 테두리와 링크에 동기화.
  - 패킷이 떠남 → TTL 게이지 한 칸 감소 → 캡션이 "R3 가 10.0.2.0/24 행을 골랐다" 로 갱신.
- **임베드 폭 안에 모두 들어가야 함**: 720px 가로 안에서 토폴로지 영역(폭 360~400px) 과 라우팅 테이블 패널(폭 280~320px) 이 좌우로 나란히, 그 위에 캡션 영역(전폭), 그 아래에 컨트롤 바(전폭). 헤더 inspect 는 절대좌표로 패킷 위에 부유하므로 가로 분할에 영향 없음. **분할은 좌우 2 영역까지만**.

## 7. 레이아웃

640~720px 가로(확장 시 880px) 기준 4 단 적층.

```
┌──────────────────────────────────────────────┐  720px
│  [캡션] 2~4문장 개념 설명. dst IP / TTL 강조.   │  ~70px
├────────────────────────┬─────────────────────┤
│                        │                     │
│   [토폴로지 그래프]    │   [라우팅 테이블]    │
│   R1─R2─R3─R5─dst      │   prefix│next│iface │  ~320px
│       │   │            │   ─────────────────  │
│   src─R4─R6           │   비트 정렬 + 막대  │
│                        │                     │
├────────────────────────┴─────────────────────┤
│  [컨트롤] ◀ ⏯ ▶  hop 3/6   speed: ×1         │  ~40px
└──────────────────────────────────────────────┘
```

- **모바일/협소 임베드(≤ 540px)**: 좌우 분할이 위아래로 적층된다. 토폴로지가 위(높이 ~280px), 표가 아래(높이 ~260px). 캡션과 컨트롤은 그대로 전폭. 라우팅 테이블 패널이 활성화될 때 토폴로지 위에 부드럽게 절반 덮는 sheet 로 변환되는 옵션도 가능(8 절 후속 확장).
- **확장 폭(880px)**: 토폴로지 영역이 ~480px 로 늘어, 라우터 6 개를 더 여유롭게 배치. 표 패널은 비트 정렬 영역이 32 칸을 한 줄에 다 보일 만큼 넓어진다.

## 8. 진행과 이벤트

기본 시나리오: src 호스트에서 dst 호스트(예: `203.0.113.42`) 로 패킷 1 개를 보낸다. 4~5 hop 경로.

| 단계 | 시각 변화 | 사용자 제어 |
|---|---|---|
| **0. 초기** | 토폴로지 정적, 패킷 카드가 src 옆 대기. 캡션: "패킷은 한 라우터에서 다음 라우터로 한 칸씩 던져진다." | Play 누름 |
| **1. src→R1 슬라이드** | 패킷이 첫 링크 위를 미끄러진다. TTL 8 그대로. | Pause 가능 |
| **2. R1 도착 펄스** | R1 부풀음, 표 패널이 R1 의 표로 swap. | — |
| **3. R1 비트 매칭** | 표의 모든 행이 dst IP 와 비트 정렬, 매치 막대 그어짐, 가장 긴 행 골드. | Step 단위로 멈춤 가능 |
| **4. R1→R3 출발** | 골드 행의 iface 색이 패킷+링크에 입혀지며 R3 으로 떠남. TTL 7. | — |
| **5~7. R3, R5 에서 같은 박자 반복** | 매 hop 마다 (2)~(4) 가 반복. 캡션이 매 hop 마다 "Rk 는 prefix P 를 골랐다" 로 갱신. | — |
| **8. dst 도착** | 패킷이 dst 호스트에 흡수, 토폴로지 위에 지나온 경로가 잔상으로 남음(옅은 골드 곡선). | — |
| **9. 회고 컷** | 캡션이 "출발 시 src 가 본 경로는 R1 까지뿐이었다" 로 바뀌고, R1 의 표만 강조 — 지하철 메타포와의 차이를 명시. | "다시 보기" 누름 가능 |

**사용자 제어 지점**:
- Play / Pause / Step / Reset.
- hop 슬라이더 — 임의 hop 으로 점프(라우팅 테이블 패널이 그 hop 의 라우터 것으로 즉시 swap).
- dst IP 입력 — 학습자가 dst 를 바꿔 같은 토폴로지에서 경로가 어떻게 갈라지는지 확인. (4 개 정도의 프리셋 dst 제공: 같은 prefix 안 / 다른 prefix / 기본 경로(default route) 적용 / 일치 없음(폐기) 케이스.)
- 표 행 hover — 그 행이 골랐을 가상의 next hop 을 토폴로지 위에 점선 화살표로 미리보기.

**선택적 확장**: TTL 0 시연 모드 — TTL 을 1~2 로 시작해 도중에 폐기되는 케이스. 폐기 시 패킷 카드가 회색으로 변하고 ICMP Time Exceeded 메시지 작은 아이콘이 src 로 돌아간다.

## 9. 초기 상태와 파라미터

- **토폴로지 (고정)**: 라우터 6 개(R1~R6) + src + dst. R1, R3, R5 가 주 경로. R2, R4, R6 가 우회/대안. 모든 링크는 동등 비용.
- **라우팅 테이블 (각 라우터마다 4~6 행)**:
  - 각 라우터의 표는 의도적으로 겹치는 prefix 를 포함 — 예: `203.0.113.0/24` 와 `203.0.0.0/16` 를 같은 표에 두어 longest prefix match 가 의미 있게 작동.
  - 모든 라우터에 `0.0.0.0/0` (default route) 를 마지막 행으로 둠.
- **패킷 초기 상태**: src=`192.0.2.10`, dst=`203.0.113.42`, TTL=8, 헤더 카드 펼침 상태 false.
- **재생 파라미터**: 기본 속도 ×1 = hop 당 1.1 초(도착 0.2 + 매칭 0.6 + 출발 0.3). ×0.5 / ×1 / ×2 / ×4 토글.
- **시드/난수**: 없음. 결정론적 진행. 같은 dst 입력에 같은 경로 보장.
- **랜덤 dst 버튼**: 4 프리셋에서 순환.

## 10. 레퍼런스 대비 차별점

### 공통 요소를 어떻게 더 잘 다루는가
- **라우팅 테이블 행 highlight 의 "왜 그 행인가"**: 표준(Kurose, Packet Tracer)은 단순히 행 배경색만 바꾼다. 본 데모는 **모든 행을 동시에 dst IP 와 비트 정렬해 매치 길이를 가로 막대로 시각화** — 학습자는 "왜 다른 행이 아니라 이 행인지" 를 한 화면에서 직관적으로 본다.
- **결정 펄스의 박자**: 표준은 하이라이트가 즉발적이다. 본 데모는 **도착 펄스 → 비트 매칭 펼침 → 출구 색 입힘 → 출발** 의 4 단을 1.1 초 안에 명확한 시간 분리로 둔다. 매 hop 마다 같은 박자가 반복되어 학습자가 리듬을 신체로 익힌다.
- **TTL 의 시각적 실재**: 표준(Wireshark)은 헤더 트리 안의 숫자 한 자리. 본 데모는 **패킷 카드 옆구리의 8칸 가로 게이지** 로, hop 마다 한 칸이 어두워진다. 패킷의 "수명" 이 시각적으로 줄어드는 감각을 준다.

### 표준이 놓치는 어떤 측면을 추가하는가
- **표 hover 미리보기**: 학습자가 다른 행을 hover 하면 그 행이 가상으로 골랐을 next hop 이 토폴로지 위에 점선으로 미리보기된다. "왜 이 행이 이겼는가" 의 반사실(counterfactual) 을 직접 만져볼 수 있다. 표준 학술 도구에는 없다.
- **회고 컷 (단계 9)**: hop-by-hop 의 본질 — "출발 라우터가 끝까지 알지 못한다" — 을 데모 끝에 명시적으로 강조. 지하철 환승 안내 메타포와 정면으로 차별화.
- **dst 입력 인터랙션**: 같은 토폴로지 위에서 dst 만 바꿔 경로가 어떻게 갈라지는지 즉각 본다. 표준 동반 애니메이션은 시나리오가 고정.

### 학습자가 "표준 도구의 그 시각화보다 낫다" 고 느낄 부분
- Kurose 의 표 highlight 는 "그 행이 이겼다" 만 보여 주는 반면, 본 데모는 **모든 후보의 매치 길이를 동시에 막대로 보여 주고 가장 긴 것이 시각적으로 이긴다**. longest prefix match 라는 이름이 시각으로 즉시 정당화된다.
- Packet Tracer 의 inspect 창은 별도 modal. 본 데모는 **패킷 자체에 TTL 게이지를 박아** inspect 없이도 핵심 헤더가 항상 보인다.
- BGPlay 의 시간축 스크럽은 AS 단위. 본 데모는 **hop 슬라이더로 한 패킷의 hop 단위 점프** — 한 라우터 한 라우터의 결정을 학습자가 직접 골라 본다.

## 11. 글 흐름 안에서의 위치

**위치**: 네트워크 계층(L3) 챕터의 **원리 설명** 섹션 도입부. IP 주소 / prefix / 라우터 같은 용어가 짧게 소개된 직후, "그래서 패킷이 실제로 어떻게 가는가" 를 보이는 첫 번째 본격 데모로 등장.

이 데모 다음에 ② 라우팅 테이블의 데이터 구조(trie/radix), ③ 라우팅 프로토콜(OSPF, BGP) 같은 주제가 이어진다. 본 데모는 그 모든 후속 주제의 **공통 베이스라인 시각** — "그래서 결국 패킷은 hop-by-hop 으로 던져진다" 는 사실을 인이 박이게 만드는 자리.

**시각화 안에 포함될 짧은 설명 텍스트** (캡션 영역, 720×70px 안):

> 인터넷의 패킷은 한 라우터에서 다음 라우터로 한 칸씩 던져진다. 각 라우터는 패킷의 **목적지 IP** 만 보고 자기 안의 **라우팅 테이블** 에서 가장 잘 맞는 규칙(가장 긴 prefix 가 매치되는 행, **Longest Prefix Match**) 을 골라 다음 라우터로 보낸다. 출발 라우터는 끝까지의 경로를 알지 못한다 — 매 hop 마다 그 라우터가 자기만의 결정을 새로 내린다. 패킷의 **TTL** 은 hop 마다 1 씩 줄어, 길을 잃은 패킷이 영원히 떠도는 일을 막는다.

**툴팁 용어 풀이** (표 행 / 패킷 카드 hover 시):
- prefix → "IP 주소의 앞부분 몇 비트. 우편번호 앞자리와 비슷."
- next hop → "곧바로 넘겨 줄 옆 라우터."
- TTL → "패킷의 수명. 매 hop 마다 1 줄고 0 이면 폐기."
- 0.0.0.0/0 → "기본 경로. 다른 행이 다 안 맞을 때 쓰는 fallback."

## 부록: 기획 메모

### 자가 점검
- **시그니처 행동의 고유성**: "패킷이 라우터 위에서 잠깐 멈춘다 → 비트 정렬로 가장 긴 prefix 행이 골드 → 출구 색 입힘 → 떠난다" 는 박자는 L2 스위칭(MAC 정확 매치, 비트 정렬 없음) 이나 DNS 조회(이동 없음) 와 혼동되지 않는다. ✓
- **레퍼런스 대비 차별점**: 비트 정렬 + 매치 막대, TTL 게이지, hover 미리보기, 회고 컷의 4 가지가 표준 학술 도구에 모두 존재하지 않는다. ✓
- **임베드 폭 안전성**: 좌우 2 분할(360/320) + 캡션/컨트롤 전폭, 모바일에서 세로 적층. 좌우 4~6 분할 없음. ✓
- **개념 미숙지 학습자 진입성**: 캡션 영역에 4 문장 개념 설명 + 핵심 용어 4 개의 hover 툴팁. 데모를 처음 보는 학습자도 따라올 수 있다. ✓
- **초심자 이해도**: 박자가 명확하고 한 hop 이 1.1 초로 일정하다. dst 프리셋 4 종이 longest prefix match / default route / 폐기를 모두 커버. ✓

### FACET 자산 매핑(참고용)
- 토폴로지 그래프 layout: `packages/core` 의 layout 자산 — 단, 본 기획은 **고정 좌표** 채택. force-layout 의 흔들림이 임베드 환경에서 산만함을 유발.
- 패킷 카드 / TTL 게이지: 새 view 컴포넌트(`view-packet-card`) 가 자연스럽다. SVG group + transform 보간.
- 라우팅 테이블 표: 기존 표 view 패턴 재사용 + 비트 정렬 영역만 신규.
- 매치 막대: 단순 `<rect>` width 매핑. 신규 view 까지 가지 않아도 됨.

### 2B(새로운 방식) 와의 협업 포인트
- 본 기획은 표준 골격을 안정화한다. 2B 가 trie/radix 라우팅 테이블 같은 새로운 시각을 제안하면, **본 기획의 사이드 패널을 trie 뷰로 swap 하는 모드** 를 옵션으로 추가할 수 있다 — 박자(도착 → 매칭 펼침 → 출구 색 → 출발) 와 토폴로지 골격은 그대로, 매칭 펼침 단계만 표 ↔ trie 로 토글.
- 본 기획의 비트 정렬 영역과 2B 의 비트 격자 시각화는 같은 시각 어휘(매치 비트 초록, prefix 경계 빨간 세로선) 를 공유하면 학습자 인지 부담이 준다.
