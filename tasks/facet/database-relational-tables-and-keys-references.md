## 2. 레퍼런스 요약

### 1) Codd 원전 논문 (1970)
- 출처: Communications of the ACM. URL 위 참고
- 한 줄 정체성: 관계모델의 기초. 시각 도식보다 표의 추상 정의가 우선.
- 시각 객체: 이름이 붙은 도메인 위의 n-ary 관계를 평면 표 (행이 튜플, 열이 도메인) 로 표시.
- 시간 표현: 정적. 스냅샷 한 장.
- 메타포: 수학적 집합. 표는 집합의 표기 도구일 뿐 본질이 아님.
- 강조점: 키는 "튜플 식별자" 라는 책임. 외래키는 다른 관계의 키를 참조하는 역할로 따로 호명.
- 인상 디테일: 키 컬럼에 화살표 / 굵기 등 그래픽 강조가 거의 없고, 도메인 이름과 텍스트 정의가 키 식별을 대신함.
- 약점: 추상도가 높아 학습 초입의 직관에 도움이 적음. "왜 한 행이 유일한가" 의 즉각성 부재.

### 2) Wikipedia — Relational model
- 한 줄 정체성: Codd 모델의 표준 해설. Customer / Order / Invoice 3-릴레이션 도식이 표준 예시.
- 시각 객체: 사각 격자 테이블 3개 + 키 텍스트 강조 (굵게 + 밑줄 = 후보키, 비굵게 + 밑줄 = 외래키).
- 시간 표현: 정적.
- 메타포: 격자 (열 머리 + 행). 관계선은 그리지 않고 컬럼명 일치로 참조 암시.
- 강조점: 후보키 / 기본키 / 대체키 / 외래키의 텍스트 표기 관습.
- 인상 디테일: 외래키 셀 값과 참조 대상 셀 값이 같은 글자임을 같은 페이지에서 확인 가능 — "참조" 의 의미를 값 일치로 보여줌.
- 약점: 관계선 없는 격자 만으로는 두 테이블 사이의 연결 방향이 한눈에 들어오지 않음.

### 3) Silberschatz / Korth / Sudarshan — Database System Concepts 7판 Ch.2
- 한 줄 정체성: 대학 표준 교재. 관계 = 테이블, 튜플 = 행, 속성 = 열의 정식 정의 + 스키마 vs 인스턴스 구분.
- 시각 객체: instructor / department 등 명명된 격자 테이블. 헤더에 속성명, 본문에 튜플.
- 시간 표현: 정적 스냅샷. 다만 "instance" 라는 표현으로 시점성을 암시.
- 메타포: 집합으로서의 행 모음 + 표 격자.
- 강조점: 슈퍼키 → 후보키 → 기본키의 포함 위계, 외래키 제약을 referential integrity 로 형식화.
- 인상 디테일: ID 컬럼이 두 테이블에 동시에 등장하며, "동일 값이 양쪽에 있어야 한다" 가 제약의 본질로 서술됨.
- 약점: 키 위계의 차이를 텍스트로만 설명. 시각적으로 슈퍼키와 후보키 차이를 보여주지 않음.

### 4) freeCodeCamp — Crow's Foot Notation
- 한 줄 정체성: 입문용 ERD 표기 튜토리얼. Crow's Foot 의 카디널리티 기호 해독을 정면에서 다룸.
- 시각 객체: 엔티티 사각형 + 관계선 + 선 끝의 동그라미 / 막대 / 까마귀 발 기호.
- 시간 표현: 정적 (구조 다이어그램).
- 메타포: "한쪽 끝이 몇 개와 매칭될 수 있는가" 를 말단 기호로 압축.
- 강조점: 0 / 1 / N 의 두 자리 표기. 안쪽 (최소) + 바깥쪽 (최대).
- 인상 디테일: 관계선이 두 사각형을 잇고, 양 끝에 비대칭 기호가 붙어 비대칭 관계를 한 눈에 보임.
- 약점: 키 자체 (PK / FK) 보다 카디널리티에 시선이 쏠림. PK 표기 관습은 텍스트로만.

### 5) Creately — Chen Notation 가이드
- 한 줄 정체성: 1976년 Peter Chen 표기. 관계를 1급 시민으로 도식화.
- 시각 객체: 엔티티 사각형, 관계 마름모, 속성 타원, 키 속성은 밑줄. 약한 엔티티는 이중 사각형, 부분키는 점선 밑줄.
- 시간 표현: 정적.
- 메타포: 객체 (사각형) / 관계 (마름모) / 성질 (타원) 의 자연언어적 분리.
- 강조점: 기본키 = 밑줄. 약한 엔티티 / 부분키의 시각 표현이 명시적.
- 인상 디테일: 속성을 외부 타원으로 늘어뜨려 "한 행 = 속성 묶음" 의 분해를 가시화.
- 약점: 큰 스키마에서 타원이 흩어져 가독성이 낮음. 실무 도구에서는 거의 안 씀.

### 6) Lucidchart — ER Diagram Symbols and Notation
- 한 줄 정체성: Chen / Crow's Foot / Bachman / IDEF1X 등 표기법을 한 자리에 비교.
- 시각 객체: 표기법별 엔티티 / 관계 / 속성 / 키 기호의 매트릭스.
- 시간 표현: 정적.
- 메타포: 표기법은 다양하지만 가리키는 본질 (엔티티 + 관계 + 키) 은 같음.
- 강조점: 표기 차이의 학습 비용을 낮추기 위한 비교표.
- 인상 디테일: 같은 1:N 관계를 4가지 표기로 나란히 보여줘 표기와 의미의 분리를 학습.
- 약점: 비교표 자체가 학습 초보에게는 정보 과잉이 될 수 있음.

### 7) dbdiagram.io — DBML 문법 / 다이어그램
- 한 줄 정체성: 텍스트 DSL → 자동 다이어그램. 동적 ERD 도구의 대표.
- 시각 객체: 카드형 테이블 (헤더에 테이블명, 본문에 컬럼 목록), 키 아이콘 (열쇠), FK 는 컬럼 사이 곡선.
- 시간 표현: 동적 (편집 시 즉시 반영). 다만 진행 모델은 여전히 구조 스냅샷.
- 메타포: 카드 + 곡선. 테이블이 노드, FK 가 엣지인 그래프.
- 강조점: PK 옆 열쇠 아이콘, FK 양 끝 점에 1 / N 표기, hover 시 관련 곡선 강조.
- 인상 디테일: 컬럼 줄에 마우스를 얹으면 그 컬럼이 참조 / 피참조에 어떻게 엮였는지 곡선이 진해짐.
- 약점: 후보키 / 대체키의 시각 구분이 약함 (PK 만 강조).

### 8) MySQL Workbench — EER Color Key
- 한 줄 정체성: 상용 GUI 도구의 컬러 코딩 표준. 동적 도구.
- 시각 객체: 카드형 테이블 + 컬럼 좌측 키 / 다이아몬드 / 마름모 아이콘 + 색상 (PK 노랑 키, 인덱스 다른 색, NN 등).
- 시간 표현: 동적 (편집 / hover 시 관계선 highlight). 구조 자체는 정적.
- 메타포: 컬럼 단위로 의미 (PK / FK / NN / UQ) 가 좌측 아이콘 슬롯에 누적.
- 강조점: 색 + 아이콘으로 컬럼 역할을 한 눈에 분류.
- 인상 디테일: 한 테이블에 포커스를 주면 연결된 다른 테이블의 관계선이 색상으로 살아남.
- 약점: 색·아이콘 규약을 외워야 의미가 보임. 색맹 접근성 약함.

### 9) GeeksforGeeks — Relational Model and Codd Rules
- 한 줄 정체성: 대학 보충 교재 톤의 해설. 정의 + 작은 도식 + 예제 표.
- 시각 객체: 단순 격자 테이블. PK 컬럼은 굵게, FK 는 화살표로 다른 테이블 PK 를 가리킴.
- 시간 표현: 정적.
- 메타포: 표 + 화살표. 화살표 방향 = 참조 방향.
- 강조점: 후보키 → 기본키 선택의 텍스트 설명, 대체키의 위치 명시.
- 인상 디테일: STUDENT(roll) → COURSE(course_id) 같은 작은 두 테이블로 PK / FK 의 짝을 빠르게 보여줌.
- 약점: 그림 품질 낮음, 카디널리티 / 무결성 위반 사례 시각화 없음.

### 10) Datarian (한국어) — 여러 종류의 Key 이해
- 한 줄 정체성: 한국어 데이터 모델링 입문. 슈퍼키 → 후보키 → 기본키 → 대체키 → 외래키의 위계를 설명.
- 시각 객체: 학번 / 주민번호가 함께 있는 학생 테이블 같은 친숙한 예시. 키 종류별 텍스트 라벨.
- 시간 표현: 정적.
- 메타포: 키 위계의 깔때기 (모든 슈퍼키 ⊃ 후보키 ⊃ 기본키).
- 강조점: 후보키 중 하나가 기본키, 나머지가 대체키라는 분기점.
- 인상 디테일: "유일성 + 최소성" 두 조건이 후보키 정의의 축임을 강조.
- 약점: 그림 의존도가 낮아 시각적 인상은 약함.

### 11) 위키백과 (한국어) — 외래 키
- 한 줄 정체성: 외래키의 형식 정의 + 참조 무결성 + 행위 (CASCADE / RESTRICT 등) 한국어 표준 해설.
- 시각 객체: 두 테이블 사이의 참조를 텍스트와 작은 표로 설명.
- 시간 표현: 정적.
- 메타포: 한 테이블의 컬럼이 다른 테이블의 키를 "가리킨다".
- 강조점: 외래키가 거부하는 순간 (참조 대상 부재 / 삭제) 의 동작 규칙.
- 인상 디테일: 참조 무결성 위반이 발생하는 시나리오를 명문화 — FACET 가 "거부 순간" 을 시각화할 단서.
- 약점: 시각 도식보다 텍스트 정의 중심.

## 3. 공통 요소

거의 모든 레퍼런스가 공유하는 시각 언어.

- **사각 격자 테이블**: 헤더 = 컬럼명, 본문 = 행. 테이블에는 이름표가 붙는다.
- **컬럼 단위 키 표기**: PK 는 굵게 + 밑줄 또는 좌측 열쇠 아이콘. FK 는 비굵은 밑줄 / 별도 아이콘 / 다른 색.
- **두 테이블 + 짝지어진 컬럼**: 거의 모든 입문 자료가 "테이블 두 개 + 한쪽 PK + 다른쪽 FK" 의 최소 구성으로 시작.
- **참조 표시**: 컬럼명 일치 (Codd / Wikipedia) → 화살표 (GeeksforGeeks) → 곡선 + 카디널리티 기호 (Crow's Foot / dbdiagram.io / MySQL Workbench) 로 구체화.
- **키 위계의 텍스트 정의**: 슈퍼키 ⊃ 후보키 ⊃ 기본키, 그리고 후보키 - 기본키 = 대체키.
- **참조 무결성**: FK 값은 참조 대상 PK 값에 존재해야 한다 — 거의 모든 자료의 공통 명제.

## 4. 공백

레퍼런스가 잘 다루지 않거나 시각적으로 약하게 처리하는 측면.

- **"무엇이 한 행을 유일하게 만드는가" 의 즉시성**: 키 컬럼을 굵게 표시할 뿐, 같은 값이 두 행에 들어가면 안 된다는 제약의 시각적 충돌을 보여주는 자료가 거의 없음. 후보 행을 들이밀었을 때 "충돌 / 통과" 가 즉시 보이지 않음.
- **FK 제약이 거부하는 순간**: 위키백과 한국어판이 텍스트로 언급할 뿐, 시각적으로 "참조 대상 없는 FK 값" 이 빨갛게 거부되는 장면을 보여주는 정적 자료는 사실상 없음.
- **후보키와 대체키의 시각 구분**: 대부분 PK 만 강조 (밑줄 / 굵게). 같은 굵기로 후보키 둘을 나란히 보여주고 그 중 하나가 기본키로 "선택" 되는 시각 분기를 거의 다루지 않음.
- **복합키 (composite key)**: dbdiagram.io 같은 일부 도구가 지원을 언급하나, "두 컬럼이 함께여야 행을 유일하게 만든다" 는 시각적 묶음 (괄호 / 밴드) 표현은 약함.
- **키 부재의 모호성**: 키가 없을 때 같은 값을 가진 행이 구분되지 않는 모호 상태를 시각화한 자료가 거의 없음. 학습자가 "왜 키가 필요한가" 를 통감할 도식이 비어 있음.
- **PK ↔ FK 의 쌍대성**: PK 가 "정의하는 쪽" 이고 FK 가 "참조하는 쪽" 이라는 비대칭이 화살표 한 방향으로만 표현되며, 두 역할이 한 컬럼에서 어떻게 다르게 작동하는지 (예: PK 는 NULL 불가, FK 는 가능) 의 대비는 약함.

## 5. 대상 개념 요약

테이블은 같은 형태의 행을 모은 격자이고, 각 행은 어떤 컬럼 묶음 — 키 — 에 의해 유일하게 식별된다. 후보키 중 하나가 기본키로 선택되어 그 테이블의 대표 식별자가 되며, 다른 테이블의 외래키는 이 기본키 값을 참조해 두 테이블 사이의 연결을 만든다. 키와 참조는 단순한 표기가 아니라 제약이다 — 같은 키 값을 가진 두 행은 공존할 수 없고, 참조 대상 없는 외래키 값도 받아들여지지 않는다. 이 두 제약이 동시에 살아있을 때만 격자들이 일관된 한 구조로 묶인다.
