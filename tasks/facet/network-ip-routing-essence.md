# IP 라우팅 핵심 추출

> 입력 — 분류: type=systemBehavior, 진행 모델=메시지 시퀀스형, 도메인=network, 핵심 학습 질문="한 패킷이 어떻게 각 라우터의 라우팅 테이블 조회와 next-hop 결정만으로 출발지에서 목적지까지 hop-by-hop 으로 길을 찾아가는가". 레퍼런스 10건 (Kurose-Ross LPM 인터랙티브, Cloudflare Learning, Cisco Packet Tracer, INET/OMNeT++ RoutingTableVisualizer, simulations4all, BGPlay, Traceroute Mapper 군, Opte/KMCD, ByteByteGo + Practical Networking, NETMANIAS + 첼시팬 + Velog) 의 공통 시각 언어와 10개 공백을 종합. 베이스의 라우팅 면은 **데이터 평면 (forwarding) 만** 으로 고정한다 — 컨트롤 평면(라우팅 프로토콜에 의한 테이블 형성) 은 공백 후보로 미루고, 베이스에서 라우팅 테이블은 "이미 주어진 것" 으로 정적·구분된 톤으로 그린다. 베이스 토폴로지는 **호스트 A — R1 — R2 — R3 — 호스트 B 의 4~5 노드 선형 + 분기 1~2 개의 작은 망형** 으로 고정한다 — 임베드 폭에 적합하고 hop-by-hop 운동·LPM·TTL 세 본질을 동시에 운반할 수 있는 최소 무대다. 결정 근거: 단일 패킷 + 데이터 평면 + 작은 토폴로지는 학습 부하가 가장 낮으면서 "지역적 결정 + LPM + 수명 + 분산성" 네 본질을 동시에 운반할 수 있으며, 컨트롤 평면을 함께 그리면 "한 패킷이 어떻게 길을 찾아가는가" 의 학습 질문 자체가 흐려진다.

## 1. 반드시 다뤄야 할 본질

학습자가 빠뜨리면 시각화가 IP 라우팅을 거짓으로 보여주게 되는 것.

1. **패킷이 hop-by-hop 으로 라우터 사이를 미끄러진다 — 한 걸음씩의 운동.** 패킷은 출발지 호스트에서 목적지 호스트로 직행하지 않고, 라우터를 차례로 거치며 한 hop 씩만 전진한다는 사실이 운동 자체로 강제되어야 한다. 패킷이 한 라우터에서 다음 라우터로 이동하는 사건이 시간 위에 끊긴 단위로 펼쳐지고, 그 사이에는 "라우터 안에서의 멈춤" 이 명시되어야 한다. 직선으로 한 번에 도달하는 그림은 패턴의 정체를 거짓으로 만든다.
2. **각 라우터가 자기 라우팅 테이블만 보고 next-hop 을 결정한다 — 결정의 지역성.** 한 hop 의 결정은 그 라우터 박스 안에서, 그 라우터의 표 한 장만으로 이루어진다는 사실이 위치만으로 강제되어야 한다. 다른 라우터의 표는 그 라우터의 결정에 관여하지 않는다 — 결정의 입력이 "현재 라우터의 표 + 패킷 헤더의 목적지 IP" 두 개뿐이라는 사실이 시각으로 못 박혀야 한다. 패킷이 미리 정해진 길을 따라가는 것처럼 보이게 그리면 hop-by-hop 분산의 본질이 사라진다.
3. **longest prefix match — 가장 긴 일치가 next-hop 을 정한다.** 라우팅 테이블의 여러 행 중 패킷의 목적지 IP 와 prefix 가 일치하는 것이 여러 개일 수 있고, 그중 가장 긴 (가장 구체적인) 일치가 채택된다는 사실이 시각적 사건으로 보여야 한다. 단일 일치가 즉시 채택되는 것이 아니라 후보들이 함께 비교되고 그중 하나가 이긴다는 비교 운동이 결정의 핵이며, 이 비교가 보이지 않으면 학습자는 "표를 맨 위에서부터 읽으면 된다" 는 오해를 가지게 된다.
4. **TTL 이 매 hop 1 씩 줄고, 0 이면 패킷이 폐기된다 — 패킷의 수명.** 패킷 헤더에 박힌 TTL 숫자가 라우터를 통과할 때마다 한 칸씩 줄어드는 운동이 모든 hop 마다 사건으로 발화하고, 0 에 도달하는 순간 패킷이 사라지는 폐기 사건이 시각으로 펼쳐져야 한다. TTL 은 텍스트 부속이 아니라 패킷의 정체성에 박힌 운동량이며, 실패 시나리오 (목적지에 닿지 못함) 가 베이스에서 누락되면 라우팅의 정직한 그림이 거짓이 된다.
5. **출발지 호스트와 목적지 호스트가 네트워크의 끝점 — 라우터는 중간 노드일 뿐.** 패킷의 시작과 끝은 호스트이지 라우터가 아니라는 사실이 노드의 시각적 위상 차이로 명시되어야 한다. 좌측 끝의 출발지 호스트, 우측 끝의 목적지 호스트가 라우터와 다른 아이콘·다른 위상으로 그려져 두 끝점의 정체성이 즉시 구분되어야 한다. 라우터들의 사슬만 그리면 "패킷이 어디서 시작해 어디서 끝나는가" 의 본질이 사라진다.

## 2. 강화해야 할 요소

표준 레퍼런스가 약하게 다루는 것을 이 시각화의 차별점으로 끌어올린다. 레퍼런스 보고서의 10개 공백 중 베이스에 적합한 5개를 선별한다.

1. **longest prefix match 의 운동형 시각 — 비트 단위 매칭이 한 컷에 펼쳐진다.** 라우터의 결정 순간에 라우팅 테이블의 여러 prefix 행이 동시에 노출되고, 패킷의 목적지 IP 의 비트가 각 행의 prefix 와 한 비트씩 일치 검사되며, 일치 길이가 시각적으로 차오른다. 가장 긴 일치 행이 점진적으로 강조되고 짧은 일치 행은 순서대로 흐려지며, 최종 채택된 한 행이 next-hop 으로 발화한다. R1 의 단발 lookup 도, R3·R4·R10 의 표 텍스트도 보여주지 못한 "왜 longest 가 이기는가" 의 비트 비교 운동을 베이스의 정체성으로 끌어올린다.
2. **TTL 게이지의 사건성 — 봉투에 박힌 숫자가 매 hop 카운트다운한다.** 패킷 봉투 위에 TTL 게이지·숫자 카운터가 항상 보이고, 라우터를 통과하는 매 hop 마다 한 칸씩 줄어드는 운동이 사건으로 발화한다. 임계 (낮은 값) 근방에서 색이 변하며 "곧 사라질 수 있다" 는 긴장이 시각으로 운반되고, 0 에 도달하는 순간 패킷이 정지·폐기되는 사건이 텍스트가 아닌 운동으로 펼쳐진다. 표준 레퍼런스가 TTL 을 헤더 텍스트 한 줄로 처리한 한계를 사건성 있는 게이지로 끌어올린다.
3. **hop-by-hop 결정의 독립성 — 라우터의 시야 한계를 시각으로 강제한다.** 패킷이 도착한 라우터의 표만 펼쳐 보이고, 다른 라우터의 표 영역은 어둡게·접혀서 가려진다. 학습자가 한 시점에 볼 수 있는 결정 정보는 "지금 라우터의 표 한 장" 뿐이며, 다음 hop 의 결정은 다음 라우터에 도착해야 비로소 펼쳐진다. "패킷이 미리 정해진 길을 따라간다" 는 표준 도식의 무의식적 인상을 깨고, 길은 결정의 연속으로 사후에 형성된다는 사실이 시야 제한 운동으로 못 박힌다. 어느 라우터도 전역 토폴로지를 모른다는 사실이 그림 자체의 위상으로 강제된다.
4. **패킷 정체성 보존과 헤더 펼침 — 한 패킷이 끝까지 추적된다.** 패킷이 색·번호·아이콘으로 고유한 정체성을 갖고, 모든 hop 을 가로지르는 동안 그 정체성이 유지된다. 라우터에 도착한 순간 헤더가 펼쳐져 src·dst·TTL 세 필드가 표로 보이고, 다음 hop 으로 떠날 때 TTL 만 줄어들고 src·dst 는 그대로라는 비대칭이 시각으로 운반된다. R3 의 헤더 펼침과 R9 의 표 비교가 정적이었던 영역을 "한 패킷의 시간선" 위에 동기화된 펼침으로 끌어올려, "L3 추상의 안정성 + TTL 의 운동성" 두 사실이 같은 운동에서 동시에 보이게 한다.
5. **데이터 평면과 컨트롤 평면의 시각 분리 — 표는 "주어진 것" 으로 정적 톤.** 베이스는 forwarding 만 다루고, 라우팅 테이블은 라우터마다 정적·구분된 톤(예: 흐린 배경, 잠긴 자물쇠 마크 같은 보조 표지) 으로 그려져 "이 시각화 안에서 표는 변하지 않는다" 가 위상으로 강제된다. 표가 어떻게 만들어지는가 (RIP·OSPF·BGP) 는 시각에서 빠져 있다는 사실 자체가 명시적이며, 학습자에게 "지금 보고 있는 것은 데이터 평면" 이라는 자각을 시각이 운반한다. R6 (BGPlay) 가 컨트롤 평면만, R3·R5 가 데이터 평면만 다루느라 두 평면이 한 도식 위에서 분리되지 못한 공백을 베이스의 위상 결정으로 메운다.

## 3. 공백 활용 후보

레퍼런스의 공백 중 흥미로우나 베이스에는 채택하지 않는 것. 미래 창의 사이클의 출발점.

1. **컨트롤 평면 (RIP / OSPF / BGP) 의 테이블 형성 운동.** 라우팅 테이블이 어떻게 만들어지고 갱신되는가 — 이웃 라우터끼리 정보가 교환되며 표의 행이 추가·갱신되는 운동을 별도 시각으로 끌어올린다. 베이스의 정적 표가 "어떻게 그렇게 되었는가" 의 답이 별도 사이클의 정체성이 된다.
2. **다중 경로 / asymmetric routing — 같은 출발-도착 사이 두 패킷이 다른 길.** 같은 src·dst 의 두 패킷이 시점에 따라 다른 hop 시퀀스로 도달하는 운동, 또는 왕복의 가는 길과 오는 길이 다른 사실을 시각으로 강제하는 별도 사이클. R7 traceroute 에서 우연히 발견되는 사실을 의도적 시각으로 끌어올린다.
3. **링크 장애와 재수렴.** 한 링크가 끊어지는 사건이 컨트롤 평면을 움직이게 하고, 표가 일제히 갱신되며 새 경로로 패킷이 우회하기 시작하는 운동. 데이터 평면의 행동과 컨트롤 평면의 시간성을 한 화면에서 결합하는 별도 시각.
4. **실 인터넷 좌표 + AS 표시 — 추상 토폴로지에서 실물 인프라로의 외연.** 추상적인 R1·R2·R3 박스가 아니라 도시·대륙 좌표 위의 AS 와 해저 케이블 위에서 패킷이 운동하는 시각. R7·R8 의 풍경적 시각을 학습용으로 끌어들이는 별도 사이클.
5. **NAT 와의 결합 — 사설 IP / 공인 IP 의 변환 운동.** 라우터가 단순 forwarding 이 아니라 src 주소를 바꾸어 보내는 운동, 응답이 돌아올 때 역변환되는 사실을 한 시각에 운반하는 외연 확장.
6. **패킷 단편화 — MTU 를 넘는 패킷이 쪼개진다.** 한 패킷이 라우터에서 여러 조각으로 나뉘어 다른 hop 으로 흩어지고 목적지에서 재조립되는 운동. 베이스의 단일 패킷 정체성을 흐리지 않기 위해 분리.
7. **큐잉 지연과 정체 — 라우터 안에 패킷이 쌓인다.** 다중 패킷이 동시에 도착해 라우터 큐에 누적되고, 큐 길이가 정체로 시각화되며 손실 사건이 발화하는 운동. R5 가 부분적으로 시도한 영역을 더 깊게 끌어올리는 별도 사이클.
8. **IPv4 vs IPv6 prefix 의 모양 차이.** 32비트 vs 128비트 prefix 의 시각적 길이 대비, 같은 LPM 알고리즘이 두 주소 체계 위에서 어떻게 다르게 보이는가 의 비교 시각.
9. **traceroute 의 응답 추적 — TTL 폐기를 의도적으로 활용한다.** 출발지가 일부러 작은 TTL 로 패킷을 보내 각 hop 의 ICMP TTL Expired 응답을 수집하는 운동. 베이스의 "TTL 은 실패 사건" 을 뒤집어 "TTL 을 도구로 쓴다" 의 메타 시각.
10. **다중 패킷의 동시 운동 — 같은 토폴로지 위 N 패킷이 서로 다른 길.** 베이스의 단일 패킷을 N 으로 확장해, 라우터 내부의 동시 처리·서로 다른 정체성·서로 다른 hop 시퀀스가 한 시간축 위에서 동시 운동하는 별도 시각.
