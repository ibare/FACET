# CDN 분류

## 1. 대상 정보

- 이름: CDN (Content Delivery Network, 콘텐츠 전송망)
- 영역: 시스템 설계 / 캐싱
- 상위 분류: 시스템 설계의 캐싱 (지리적으로 분산된 엣지 캐시가 본질)
- 도메인: system-design-caching
- 도메인 정당화: 상위 결정이 "시스템 설계의 캐싱" 이므로 도메인 슬러그도 그 영역 이름을 그대로 따른다. 단순한 `caching` 보다는 시스템 설계 맥락임을 드러내는 `system-design-caching` 이 다른 캐싱 사례 (LRU, write-through 등) 와 같은 영역으로 묶인다는 점을 명확히 한다.

## 2. type

systemBehavior

- 근거: CDN 은 단일 자료구조도, 단일 알고리즘도 아니다. 클라이언트 / 엣지 노드 / 오리진 서버라는 여러 주체가 지리적으로 분산된 상태로 상호작용하며, 요청의 hit / miss / fill / 만료 라는 역할 분담된 행위가 시스템 전체에서 관찰되는 정체성이다. 자료구조 (LRU 처럼 단일 인스턴스의 상태 변형) 보다 시스템 차원의 행위 패턴이 학습 대상이다.

## 3. 진행 모델

입력 반응형

- 근거: CDN 은 자율적인 시간 진행 (algorithm 처럼 알아서 다음 단계로 이동) 이나 정해진 메시지 시퀀스 (protocol 처럼 handshake 의 순서가 핵심) 가 본질이 아니다. 클라이언트가 발행하는 요청 (특정 엣지 노드에 대한 콘텐츠 요청) 이 트리거이며, 엣지 노드는 그 요청에 반응해 캐시 hit 을 즉시 응답하거나, miss 시 오리진에서 가져와 채우고 응답한다. TTL 만료 같은 시간 요소가 있으나 이는 보조적이며, 학습의 중심축은 "요청에 어떻게 반응하는가" 이다.

## 4. 핵심 학습 질문

지리적으로 분산된 여러 엣지 캐시가 각자 클라이언트 요청에 반응할 때, 어떤 요청은 가까운 엣지에서 즉시 응답되고 어떤 요청은 오리진까지 거슬러 올라가야 하는가, 그리고 그 차이가 사용자 체감 지연과 오리진 부하에 어떤 형태로 드러나는가?
