## 2. 레퍼런스 요약

### R1. Cloudflare — CDN Reference Architecture

- 출처: Cloudflare Developers (공식 레퍼런스 아키텍처 문서)
- URL: https://developers.cloudflare.com/reference-architecture/architectures/cdn/
- 한 줄 정체성: "CDN 없는 직선" 에서 시작해 엣지·지역·중앙 캐시·Cache Reserve 까지 한 다이어그램씩 층을 더해가며 진화시키는 사업자의 정공법 도식.
- 시각 객체: 클라이언트 아이콘, 엣지 노드 다발, 상위 캐시 노드, 오리진 서버, 그리고 그 사이를 잇는 화살표. 클라이언트마다 색을 달리해 두 요청 흐름이 같은 엣지에서 갈리는 모습을 한 화면에 겹친다.
- 시간 표현: 정적 다이어그램 8개의 시리즈. 각 그림이 한 단계의 "추가 결정" 을 의미하는 진화 서사.
- 메타포: 깔때기 / 다층 필터 — 요청이 가까운 층에서 잡히면 좋고, 못 잡히면 다음 층으로 흘러간다.
- 강조: 두 클라이언트 비교에서 한쪽은 오리진까지 가고 한쪽은 상위 캐시에서 멈춘다는 사건. 95% 이상이 지역 응답으로 끝난다는 결말.
- 인상적 디테일: 같은 데이터 흐름에 "Argo Smart Routing" 을 끼우면 화살표 경로 자체가 휘어 빨라진다는 라우팅 지능의 시각화.
- 약점: 정적이라 캐시 미스가 미스로 연쇄되는 시간 차이가 운동으로 안 보임. 지리감 (어느 도시 엣지인지) 도 추상화돼 사라진다.

### R2. Cloudflare — Global Network 페이지

- 출처: cloudflare.com/network (사업자 마케팅 + 데이터 시각화)
- URL: https://www.cloudflare.com/network/
- 한 줄 정체성: 세계지도 위에 100개국 이상의 데이터센터 위치를 점으로 찍어 "어디에든 가깝다" 를 한 화면 인상으로 박는 표준적 CDN 지도.
- 시각 객체: 메르카토르 세계지도 + 도시 점 (별표는 AI 추론 가능 위치) + 대륙별 도시 목록.
- 시간 표현: 정적. 다만 "레이턴시 테스트 실행" 버튼이 사용자가 자기 위치에서 가장 가까운 엣지까지의 시간을 직접 측정하게 한다.
- 메타포: 별자리 / 별이 박힌 지구 — 점 하나하나가 "여기서도 응답할 수 있다".
- 강조: 점의 밀도. 13,000+ 상호연결, 200+ 도시 같은 숫자와 점이 1:1 로 묶인다.
- 인상적 디테일: 사용자 위치 기반 레이턴시 측정 — 추상 숫자가 자기 ms 로 환산되는 순간.
- 약점: 캐시 동작 (히트/미스, TTL, 무효화) 은 전혀 보이지 않음. 위치 분포만 자랑하는 정적 마케팅 맵.

### R3. Akamai — Internet Observatory / Visualizing Akamai

- 출처: akamai.com (사업자 공식 시각화 플랫폼)
- URL: https://www.akamai.com/uk/en/resources/visualizing-akamai/internet-observatory/ , https://www.akamai.com/internet-station/traffic-map
- 한 줄 정체성: Akamai 가 처리하는 전 세계 트래픽 (전체 웹 트래픽의 15~30%) 을 실시간 색·굵기로 지도 위에 그려 "지금 인터넷이 흐르는 모습" 자체를 보여주는 라이브 시각화.
- 시각 객체: 세계지도 + 도시별 트래픽 강도를 색·반경으로, 대륙간 흐름을 두께가 변하는 호 (arc) 로 표시.
- 시간 표현: 실시간 — 시간대마다 색·굵기가 호흡한다. 공격 트래픽, 가장 느린 도시, 가장 붐비는 지역 같은 라이브 지표.
- 메타포: 지구의 맥박 / 혈류 — 데이터가 대륙 사이를 펄스로 흐른다.
- 강조: "전체" 의 호흡. 어느 한 사용자가 아니라 글로벌 집합체가 어떻게 출렁이는가.
- 인상적 디테일: 공격 (DDoS) 트래픽 색이 평시 트래픽 위에 덧칠되며 평소와 다른 흐름을 즉시 드러낸다.
- 약점: 캐시 메커니즘 자체와는 거리가 있음. CDN 의 작동 원리보다는 결과로 나타난 트래픽 풍경에 가깝다.

### R4. Steve Song — Global Map of CDNs and Cloud Services

- 출처: cdns.nsrc.org (오픈 데이터 프로젝트, NSRC 호스팅)
- URL: https://cdns.nsrc.org/
- 한 줄 정체성: Cloudflare·Akamai·Google·AWS·Fastly 등 여러 사업자의 PoP 위치를 한 지도 위에 사업자별 색으로 겹쳐, "전 세계가 누구의 엣지에 덮여 있는가" 를 비교하는 객관 지도.
- 시각 객체: 세계지도 + 사업자별 색 점 + 토글 가능한 레이어. 같은 도시에 여러 색이 겹치면 다중 사업자 PoP.
- 시간 표현: 정적 (분기·연 단위 갱신).
- 메타포: 영역 지도 / 통신 식민지 — 색 점이 지구를 어떻게 분할 점령했는가.
- 강조: 커버리지 격차 — 북미·유럽은 색이 빽빽하고, 아프리카·중앙아시아는 듬성하다는 시각적 대비.
- 인상적 디테일: 사업자 토글로 한 회사 점만 켜고 끄면, "이 회사는 어디까지 닿는가" 의 영역이 즉시 드러난다.
- 약점: 동작은 안 보이고 분포만 보인다. 한 요청이 어떻게 흘러가는지에 대한 서사는 없다.

### R5. ByteByteGo (Alex Xu) — How Does CDN Work?

- 출처: bytebytego.com (시스템 설계 인기 해설 채널)
- URL: https://blog.bytebytego.com/p/how-does-cdn-work , https://bytebytego.com/guides/how-does-cnd-work/
- 한 줄 정체성: "뉴욕의 Bob 이 런던 사이트를 부른다" 는 한 인물 시나리오로 CDN 의 11단계 흐름을 번호 매긴 화살표로 그려, 시스템 설계 인터뷰 자료의 사실상 표준이 된 도식.
- 시각 객체: 사람 아이콘 (Bob), 클라이언트 박스, 엣지 CDN 서버, 지역 CDN, 중앙 CDN, 오리진 서버, DNS 서버. 각 사이의 화살표에 1~11 번호.
- 시간 표현: 번호로 펼쳐진 정적 시퀀스. 한 그림에 모든 단계가 동시에 보이지만 번호가 시간 순서를 인도.
- 메타포: 한 사람의 여정 — Bob 의 한 클릭이 어디까지 갔다 오는가.
- 강조: 캐시 미스가 어떻게 상위 계층으로 "올라가는가" — 화살표가 위쪽으로 한 단씩 길어지는 시각.
- 인상적 디테일: 같은 그림 안에서 캐시 히트 (짧은 왕복) 와 캐시 미스 (오리진까지 길어진 왕복) 가 다른 색 화살표로 동시에 비교된다.
- 약점: 정적 단일 그림이라 시간 차이가 운동으로 안 보임. 한 사용자만 등장해 "여러 지역 동시 트래픽" 인 본질이 약해진다.

### R6. GeeksforGeeks — Designing Content Delivery Network (CDN) in System Design

- 출처: geeksforgeeks.org (대중 CS 학습 매체)
- URL: https://www.geeksforgeeks.org/system-design/designing-content-delivery-network-cdn-system-design/
- 한 줄 정체성: PoP, 엣지 서버, 오리진, DNS 라우팅을 박스·화살표 다이어그램과 시간복잡도 / 수치 지표 표 양쪽으로 분해하는 시스템 설계 입문서.
- 시각 객체: 박스 (서버), 굵은 화살표 (요청·응답), 점선 (DNS 조회), 다층 트리 모양으로 엣지·지역·중앙·오리진을 위로 쌓는다.
- 시간 표현: 정적 다이어그램 + 별도 표 (TTL, 캐시 히트율 95~99%, RTT 절감치).
- 메타포: 트리 / 계층 — 엣지가 잎, 오리진이 뿌리.
- 강조: "DNS 가 어떻게 가까운 PoP 를 고르는가" 의 전치 단계.
- 인상적 디테일: 캐시 히트율을 90% 이상으로 두는 "건강한 CDN" 과 50% 의 "병든 CDN" 의 수치 대비.
- 약점: 그림 자체가 단조롭다. TTL 만료, 무효화 같은 시간축 사건은 글로만.

### R7. Hello Interview — Caching for System Design Interviews

- 출처: hellointerview.com (시스템 설계 인터뷰 학습 플랫폼)
- URL: https://www.hellointerview.com/learn/system-design/core-concepts/caching
- 한 줄 정체성: CDN 을 "지리적으로 흩어진 read-through 캐시" 로 정의하고, 일반 캐시·분산 캐시·CDN 세 가지를 같은 어휘 (히트율, 일관성, 무효화) 로 비교 정렬하는 학습용 도식.
- 시각 객체: 같은 추상 박스 (Cache) 가 위치만 바뀌어 그려진다 — 앞단 캐시, 백엔드 옆 캐시, 사용자 옆 (CDN). 화살표는 read-through 표준 패턴 (Cache → 미스 → Origin → 채움).
- 시간 표현: 정적 비교 도식.
- 메타포: "캐시는 어디에 두느냐의 게임" — 위치만 바뀌는 같은 박스.
- 강조: CDN 이 지리적 위치라는 한 차원만 추가된 캐시라는 통일된 시각.
- 인상적 디테일: 무효화·TTL·stale-while-revalidate 같은 캐시 정책 어휘를 CDN 에 그대로 적용해 어휘를 통일.
- 약점: 그림이 단순하다. 분산성·라우팅·anycast 같은 CDN 고유 본질은 캐싱 일반에 흡수돼 흐릿해진다.

### R8. Wikipedia — Content Delivery Network

- 출처: en.wikipedia.org (표준 백과)
- URL: https://en.wikipedia.org/wiki/Content_delivery_network
- 한 줄 정체성: CDN 의 학문적 정의·역사·구조·요청 라우팅 (DNS 기반, anycast 기반) 을 정적 도식 한두 장과 본문으로 정리한 표준 진입점.
- 시각 객체: 단일 다이어그램 — 사용자가 ISP 를 지나 가장 가까운 surrogate 서버로 연결되고, 서버는 오리진과 동기화된다. 모식적 박스·선.
- 시간 표현: 정적.
- 메타포: 없음. 표준 학술 도식.
- 강조: surrogate (대리 서버) 라는 용어와 그 위치. 요청 라우팅 알고리즘 (DNS, anycast, HTTP redirect) 의 분류.
- 인상적 디테일: anycast 와 DNS 라우팅이 같은 "가까운 곳 고르기" 를 다른 층에서 푼다는 분류적 통찰.
- 약점: 그림 한두 장. 거의 본문 텍스트.

### R9. Daou IDC — CDN 이란?

- 출처: blog.daouidc.com (한국 IDC 사업자 공식 블로그)
- URL: https://blog.daouidc.com/blog/cdn-easy-explained
- 한 줄 정체성: "CDN 안 쓸 때 vs 쓸 때" 두 그림을 나란히 놓고, 넷플릭스 시청 시나리오로 CDN 의 필요를 말로 풀어주는 한국어 입문 해설.
- 시각 객체: 비교 일러스트 — 좌측은 미국 본사 한 점에서 전 세계로 뻗는 긴 화살표 다발, 우측은 각 대륙 안에 거점 + 짧은 화살표.
- 시간 표현: 정적 두 컷 비교.
- 메타포: 본사 ↔ 지점 — 모든 손님이 본사로 몰리는 그림 vs 동네 지점에서 처리되는 그림.
- 강조: "오리진에 트래픽 집중 → 시스템 다운" 의 위험을 본사 다발 화살표로 시각화.
- 인상적 디테일: 넷플릭스라는 친숙한 사례로 추상 기술의 체감을 바꾼다.
- 약점: 두 컷 정적. 캐시 동작·라우팅·계층 구조 같은 내부는 그림에 없다.

### R10. NAVER Cloud — 더 빠른 콘텐츠 전송을 위한 CDN 활용 팁

- 출처: medium.com/naver-cloud-platform (한국 클라우드 사업자 기술 블로그)
- URL: https://medium.com/naver-cloud-platform/%EB%8D%94-%EB%B9%A0%EB%A5%B8-%EC%BD%98%ED%85%90%EC%B8%A0-%EC%A0%84%EC%86%A1%EC%9D%84-%EC%9C%84%ED%95%9C-cdn-%ED%99%9C%EC%9A%A9-%ED%8C%81-91f488f202d8
- 한 줄 정체성: 캐시 히트·미스 두 흐름을 "원본 거치는가 / 안 거치는가" 의 두 별도 다이어그램으로 분리해, 운영 관점에서 캐시 적중률을 끌어올리는 처방까지 묶은 한국어 실무 자료.
- 시각 객체: 사용자 → 캐시 서버 → (히트면 즉시 응답 / 미스면 원본 → 캐시 적재 → 응답) 의 두 분기 다이어그램.
- 시간 표현: 정적 두 컷.
- 메타포: 없음. 운영 다이어그램.
- 강조: TTL, Purge (강제 무효화), 캐시 적중률 — 그림보다 글로 전개.
- 인상적 디테일: 캐시 미스 다이어그램에서 "캐시에 채우는 단계" 를 별도 화살표로 명시 — 미스의 비용에 "다음을 위한 적재" 까지 들어간다는 점.
- 약점: 두 분기를 한 그림으로 합치지 않아 "같은 사건의 두 결말" 이라는 통합 인상이 약함.

## 3. 공통 요소

거의 모든 레퍼런스가 공유하는 시각 언어:

1. **세계지도 + 점** — 사업자 자료 (R2, R3, R4) 의 표준. 점 하나 = 엣지 PoP. 점 밀도가 곧 신뢰성·근접성의 시각적 자랑.
2. **클라이언트 → 엣지 → (지역) → (중앙) → 오리진의 다층 흐름도** — 시스템 설계 자료 (R1, R5, R6) 의 표준. 위로 갈수록 멀고, 가능하면 위까지 안 가는 게 좋다.
3. **번호 매긴 화살표** — 한 그림에 시간 순서를 압축할 때 1~11 같은 번호로 시퀀스를 인도 (R5, R6).
4. **캐시 히트 = 짧은 왕복 / 캐시 미스 = 긴 왕복** — 같은 요청에서 두 결말의 화살표 길이 차이로 표현 (R1, R5, R10).
5. **DNS 가 가까운 엣지를 고른다는 전치 단계** — 그림 좌측 또는 상단에 DNS 박스를 별도로 두고 점선 화살표로 연결 (R5, R6, R8).
6. **클라이언트 아이콘 한 명** — 시스템 설계 자료에서 "Bob", "User", "사용자" 단수로 추상화. 한 사람의 여정으로 시스템을 설명.
7. **"안 쓸 때 vs 쓸 때" 비교 두 컷** — 한국어·입문 자료의 단골 (R9). 좌측엔 본사로 몰리는 화살표 다발, 우측엔 지역마다 짧은 화살표.
8. **수치 지표 짝** — 캐시 히트율 95%+, RTT < 20ms 같은 숫자가 그림 옆에 표로 붙는다 (R1, R6).

학습자에게 형성된 기대:
- 가까운 엣지가 답하면 빠르고, 오리진까지 가면 느리다.
- CDN 은 "사용자 옆에 둔 캐시" 다.
- DNS 가 가장 가까운 엣지로 보내준다.

## 4. 공백

기존 레퍼런스에서 약하거나 비어 있는 영역:

1. **여러 클라이언트의 동시성** — 거의 모든 그림이 한 사용자만 등장시킨다. 실제 CDN 의 본질은 "수많은 요청이 같은 엣지로 동시에 쏟아지고, 한 번 채워진 캐시가 그 뒤 N명을 모두 빠르게 응답한다" 는 집합 효과인데, 두 번째·세 번째 사용자가 같은 엣지에서 즉시 답을 받는 가속 운동이 운동성으로 표현된 사례가 적다.
2. **시간 차이의 체감** — 캐시 히트는 ms, 미스는 수백 ms 라는 차이가 막대 길이·숫자로만 보이고, "히트 응답이 화면에 뚝 떨어지고 미스 응답은 한참 뒤 도착하는" 시간 차 자체가 운동으로 보이는 사례가 거의 없다.
3. **TTL 만료의 순간** — TTL 이 다 되어 캐시 항목이 사라지고, 다음 요청이 다시 오리진까지 가야 하는 사건이 시각화된 자료 부재. 대부분 글로만 설명.
4. **무효화 (purge) 의 전파** — 오리진이 콘텐츠를 갱신해 모든 엣지에 무효화 신호를 흩뿌리는 사건이 시각화되지 않음. CDN 운영의 가장 빈번한 골칫거리인데도.
5. **다층 캐시의 캐스케이드** — R1 의 시리즈가 가장 가깝지만, 여전히 정적. "엣지 미스 → 지역 미스 → 중앙 미스 → 오리진 적중 → 거꾸로 모든 층에 적재" 의 왕복 흐름이 한 운동으로 그려진 자료 거의 없음.
6. **anycast 라우팅의 동적 전환** — Cloudflare 자료가 "fastest path" 라고 말하지만, 한 엣지가 죽었을 때 다음 엣지로 BGP 가 트래픽을 즉시 옮기는 사건의 운동 시각화는 부재.
7. **오리진 부하의 변화** — CDN 도입 전후로 오리진이 받는 요청 수가 어떻게 줄어드는지 — "오리진의 호흡 강도" 가 그림으로 보이는 사례 적음. 숫자 표로만.
8. **지리감과 흐름의 결합** — 사업자 지도 (점) 와 시스템 설계 흐름도 (박스·화살표) 가 분리되어 있다. 같은 화면에 "지구 위 점들 + 한 요청이 어느 점을 거쳐가는가" 가 결합된 자료 드물다.
9. **캐시 콘텐츠의 다양성** — 인기 콘텐츠 (히트율 높음) 와 비인기 콘텐츠 (미스 많음) 가 같은 엣지에서 어떻게 다르게 굴러가는지의 비교 시각화 부재. 콘텐츠는 익명의 단일 객체로 그려진다.
10. **거리·근접의 진짜 의미** — 그림은 평면 지도라 "가깝다" 가 직선 거리처럼 보이지만, 실제 근접은 BGP 토폴로지·ISP 피어링의 결과다. 네트워크적 가까움이 지리적 가까움과 다를 수 있다는 사실이 거의 시각화되지 않음.

## 5. 대상 개념 요약

> CDN 은 한 곳에 있던 콘텐츠를 전 세계 여러 도시의 엣지 서버에 미리·요청 시 복사해 두고, 어떤 클라이언트가 무언가를 요청하면 가장 가까운 엣지가 먼저 받는다. 그 엣지가 그 콘텐츠를 이미 가지고 있으면 (히트) 짧은 왕복으로 즉시 응답하고, 없으면 (미스) 상위 계층 캐시 또는 오리진까지 거슬러 올라가 가져와 채운 뒤 응답하며, 그 결과 다음 요청부터는 같은 엣지가 곧장 답한다. 같은 요청이라도 어디에서, 언제 발행됐는가에 따라 짧은 응답이 되기도 하고 긴 응답이 되기도 하며, 그 차이가 모여 사용자 체감 지연을 만들고 오리진의 부하를 결정한다. 분류 보고서가 짚었듯, 이 시스템 차원의 행위 패턴 — 분산된 엣지가 각자 클라이언트 요청에 반응해 히트·미스를 분기시키는 — 이 학습 대상이며, 단일 캐시 알고리즘이나 자료구조로 환원되지 않는 systemBehavior 다.

학습자가 시각화와 함께 읽을 짧은 캡션:
- "가까운 엣지가 답하면 빠르고, 멀어질수록 길어진다." (지리적 캐싱의 본질)
- "히트는 즉답, 미스는 위로 거슬러 올라가는 여정." (두 결말)
- "같은 콘텐츠를 두 번째 사람이 부르면 이미 엣지에 있다." (집합 효과)
- "TTL 이 다 되면 다시 오리진까지 가야 한다." (시간축의 자명한 사건)
- "엣지가 N개여도 한 요청은 그중 하나만 만난다." (라우팅 결정)

용어 풀어쓰기:
- 엣지 (Edge) → "사용자 옆 도시에 놓인 작은 캐시 서버"
- 오리진 (Origin) → "원본을 가진 본진. 가능하면 안 가게 하고 싶은 곳"
- 히트 (Hit) → "엣지가 이미 그 답을 가지고 있어 곧장 돌려준 사건"
- 미스 (Miss) → "엣지에 답이 없어 윗 계층까지 다녀와야 하는 사건"
- TTL → "엣지가 그 답을 유효하다고 믿는 기한. 지나면 다시 받아와야 한다"
- PoP → "한 도시의 엣지 서버 묶음. 지도 위 점 하나"
- Anycast → "전 세계 같은 주소를 여러 곳에서 외쳐, 가까운 엣지가 자동으로 답하게 만드는 라우팅"
- 무효화 (Purge) → "오리진이 답을 갱신했으니 모든 엣지에게 옛 답을 버리라고 알리는 사건"
