# Big-O 시간복잡도 시각화 사전 분류

## 1. 대상 정보

- **이름**: Big-O notation (점근적 시간복잡도)
- **도메인**: concept (개념·수학) — 알고리즘 분석 이론
- **설명**: 입력 크기 n 이 커질 때 알고리즘의 실행 시간(또는 자원 사용)이 어떻게 증가하는지를 함수의 상한으로 표현하는 점근 표기법. O(1), O(log n), O(n), O(n log n), O(n^2), O(2^n), O(n!) 등 성장률 클래스의 비교가 학습의 핵심.

## 2. type 분류

- **결정**: `concept`
- **근거**:
  - 코드로 실행되는 절차가 아니라 **함수의 성장률에 대한 정의·동치류·상한 관계**를 나타내는 추상 개념.
  - 단계적 결정 절차(procedure) 도 아니고, 메시지 시퀀스(protocol) 도 아니며, 자료구조의 형태(dataStructure) 도 아니다.
  - "이게 무엇인가? / 왜 중요한가?" 가 본질 — 정의·정체성·범위를 보여주는 대상.

## 3. 전략 결정 (A/B)

- **결정**: **B타입 (새로운 방식)**
- **근거**:
  - 핵심 학습 질문이 "왜 알고리즘 복잡도가 이렇게 분류되는가?", "왜 n^2 는 n log n 보다 본질적으로 나쁜가?", "왜 입력이 커질수록 차이가 폭발하는가?" 라는 **성격·직관**의 영역.
  - 표준 시각 언어(여러 곡선이 겹쳐진 그래프)는 이미 어디서나 보지만, 이는 "결과의 정량 비교" 에 머무르며 "왜 이렇게 갈라지는지" 의 직관을 주지 못한다.
  - 일상 비유(전화번호부 펼치기, 줄 서기, 도미노)와의 연결이 강력 → B타입 가중.
  - 정확한 단계별 동작이 아닌 **존재 방식·운명**을 보여줘야 한다.
  - 수식(점근 정의)과 1:1 대응은 가능하나, 학습 가치는 정량 검증보다 **체감되는 격차**에 있다.
- **참고**: A타입(표준 그래프)은 이미 포화 상태. 차별화 여지는 B타입에 있다.

## 4. IR 보유 여부

- **IR 없음**.
- 근거: 실행 가능한 알고리즘이 아니라 알고리즘의 분석 결과를 표현하는 개념. transpiler 대상 아님.

## 5. 핵심 학습 질문

1. 같은 작업을 다른 복잡도 클래스로 풀면 입력이 커질 때 어떤 일이 벌어지는가? (격차의 폭발)
2. O(log n) 이 왜 "거의 공짜" 처럼 느껴지고, O(2^n) 이 왜 "사실상 불가능" 인가?
3. 상수 차이(2n vs 100n)는 왜 무시되고, 차수 차이(n vs n^2)는 왜 결정적인가?
4. 점근 상한이라는 개념 — 작은 n 에서는 거꾸로 보일 수 있다는 사실의 직관.
5. 같은 클래스(예: O(n log n)) 안에 있다는 것이 왜 "사실상 같은 부류" 로 취급되는가?

## 6. 후속 단계 가이드

- **Stage 1 (레퍼런스 수집)**: "Big-O 시각화" 의 기존 표현(겹친 곡선 그래프, 표 비교, n=10/100/1000 시간 추정 표)을 광범위하게 수집. 공통 시각 요소를 식별해 **회피 대상 목록**으로 정리.
- **Stage 2 (B타입 기획)**:
  - 메타포 후보 탐구 — 시간/거리/연료/공간 점유 등 일상 자원의 소진을 통한 "복잡도의 운명" 표현.
  - 클래스 간 격차를 **체감**시키는 장치(같은 작업, 같은 시작점, 다른 경로의 도달 시점) 우선.
  - 정확한 곡선 그래프는 보조에 머물고, 본질을 새 시각 언어로 옮길 것.
- **유의**: type=concept 이므로 algorithm 의 단계 재생(emit/projector)이 본질이 아니다. 정적·반응형 시각화 또는 사용자 입력(n 슬라이더 등)에 의한 비교 인터랙션이 자연스럽다. View / Runner 구조를 algorithm 재생 가정 없이 설계해야 한다.
