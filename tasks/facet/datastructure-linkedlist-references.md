## 2. 레퍼런스 요약

### R1. VisuAlgo — Linked List (Single, Doubly), Stack, Queue, Deque

- 출처: visualgo.net (싱가포르 NUS Steven Halim 그룹, DSA 시각화 표준)
- URL: https://visualgo.net/en/list
- 한 줄 정체성: 노드를 직사각형 정점(vertex), 링크를 방향 간선으로 그려 "그래프의 가장 단순한 형태" 로 연결 리스트를 정의한 표준 인터랙티브.
- 시각 객체: 값 박스 + 다음 정점을 가리키는 화살표. 좌→우 가로 배치(또는 위→아래). 단일은 단방향, 이중은 양방향 화살표.
- 시간 표현: Step / Play / 속도 조절. 삽입은 새 노드가 화면 밖에서 등장 → 화살표 두 개가 순차로 다시 묶임. 삭제는 들어오는 화살표 하나가 다음 노드로 점프하며 노드가 페이드아웃.
- 메타포: 데이터 구조 추상화 자체. 구체 비유 없음. "head 만 알면 전체에 닿는다" 는 ADT 약속.
- 강조: head/tail 라벨이 항상 떠 있고, "다음 한 칸 너머는 알 수 없다" 는 순차 접근의 한계.
- 인상적 디테일: 삽입 시 "먼저 새 노드의 next 를 잇고, 그 다음 이전 노드의 next 를 갈아끼운다" 는 순서가 화살표 두 개의 시간차로 직접 보인다.
- 약점: 노드들이 화면상 균등 간격으로 정렬돼 있어 "메모리 어디든 흩어져 있다" 는 핵심이 시각적으로 드러나지 않음.

### R2. USFCA Galles — Linked List

- 출처: cs.usfca.edu/~galles/visualization (DSA 시각화 고전. 90년대 자바 애플릿 → 자바스크립트 포팅)
- URL: https://www.cs.usfca.edu/~galles/visualization/StackLL.html , https://www.cs.usfca.edu/~galles/visualization/QueueLL.html
- 한 줄 정체성: 연결 리스트의 head/tail/next 를 명시한 가장 정확한 교과서적 도식. "이 그림이 곧 정의" 라 부를 수 있는 무미건조한 클래식.
- 시각 객체: 값 박스(상단) + next 화살표(하단). 박스 안 좌측은 데이터, 우측은 next 슬롯. 마지막 노드의 next 는 NULL 로 명시.
- 시간 표현: 버튼 한 번 = 한 연산. 새 노드는 화면 측면에 분리돼 나타난 뒤 화살표가 천천히 그어지며 합류.
- 메타포: 없음. 순수 추상 도식.
- 강조: 노드 내부의 "데이터 칸 + next 칸" 분리 구조와, NULL 종료라는 명시적 끝.
- 인상적 디테일: 새 노드를 임시 위치에 띄워두고 화살표를 돌려 끼우는 동작이 "포인터 재배선" 의 직접 시각화.
- 약점: UI 가 평면적이고 인상이 약함. 한 화면에 7~10개 이상 들어가면 화살표가 교차하며 가독성 붕괴.

### R3. DSA Visualizer — Singly / Doubly / Circular Linked List

- 출처: dsavisualizer.in (인도 교육 플랫폼)
- URL: https://www.dsavisualizer.in/visualizer/linkedList/types/singly , https://www.dsavisualizer.in/visualizer/linkedList/types/doubly , https://www.dsavisualizer.in/visualizer/linkedList/types/circular
- 한 줄 정체성: 단일·이중·원형 세 변종을 같은 UI 로 토글하며 보여주고, 코드 패널·퀴즈가 결합된 학습 패키지.
- 시각 객체: 가로 박스 + 화살표. 이중은 위·아래 두 줄 화살표. 원형은 마지막→첫 화살표가 위쪽으로 큰 호를 그림.
- 시간 표현: Step / Play / Reset. 한 단계 = 화살표 하나의 변화 + 코드 라인 하이라이트.
- 메타포: 추상 박스 + 코드 트레이스. 일상 비유는 거의 없음.
- 강조: 세 변종 사이의 차이가 "화살표 개수와 닫힘/열림" 한 가지 차원으로 환원됨을 시각적으로 보여줌.
- 인상적 디테일: 같은 입력 시퀀스를 세 변종에 흘리면 결과 형태가 달라지는 비교 학습 가능.
- 약점: 광고/배너로 시야가 분산. 메모리 산포 관점은 없고 노드가 줄 맞춰 정렬됨.

### R4. vonvista Linked List Visualizer

- 출처: vonvista.github.io/Linked-List (개인 학습용 데모, GitHub Pages)
- URL: https://vonvista.github.io/Linked-List/
- 한 줄 정체성: "기본 / tail 포인터 있음 / 스택 / 큐 / 이중" 다섯 모드를 한 화면에서 토글하는 미니 변종 비교 도구.
- 시각 객체: 가로 노드 박스 + 화살표. 모드 전환 시 같은 데이터에 대해 head/tail/prev 라벨이 가감됨.
- 시간 표현: 즉시 애니메이션. 단계 컨트롤 약함.
- 메타포: 없음.
- 강조: "한 자료구조의 작은 추가가 어떤 변종을 만드는가" — tail 포인터 하나, prev 화살표 하나가 ADT 를 다른 종으로 바꾼다는 사실.
- 인상적 디테일: 변종 토글로 "최소 변경 단위" 가 선명해짐.
- 약점: 운동감 부족, 노드 수 제한 명확하지 않음. 시각적 완성도는 학습 데모 수준.

### R6. basecs (Vaidehi Joshi) — What's a Linked List, Anyway?

- 출처: medium.com/basecs (CS 입문 일러스트 시리즈)
- URL: https://medium.com/basecs/whats-a-linked-list-anyway-part-1-d8b7e6508b9d
- 한 줄 정체성: 손그림 일러스트로 노드 = 명함, 링크 = 화살표 손글씨 라벨로 표현해, ADT 를 친숙한 종이 메모처럼 그린 해설.
- 시각 객체: 둥근 모서리 노드 박스(데이터 + next 칸) + 손글씨 화살표 + null 표식의 X 그림. 메모리 영역을 흩어진 카드들 위에 겹쳐 표시.
- 시간 표현: 정적 스틸 다수. "이전 상태 → 다음 상태" 두 컷을 나란히 배치.
- 메타포: "노드는 자기 데이터와 이웃 한 명만 안다(A node only knows about what data it contains, and who its neighbor is)".
- 강조: 노드의 자기 인식 한계 — "내 다음만 안다" 가 곧 순차 접근의 본질.
- 인상적 디테일: 메모리 산포를 카드들이 비뚤게 흩어진 그림으로 보여주고, 그 위에 손글씨 화살표만 잇는다. 배열 그림(균등 칸) 과의 시각 대비가 명확.
- 약점: 정적 일러스트라 "잇고 끊기" 의 운동감은 별도 영상에서만.

### R8. Ryan Haining — Stop using Train Cars to Describe Linked Lists

- 출처: ryanhaining.medium.com (현직 엔지니어 비판 글)
- URL: https://ryanhaining.medium.com/stop-using-train-cars-to-describe-linked-lists-394a74c249b1

## 3. 공통 요소

거의 모든 레퍼런스가 공유하는 시각 언어:

1. **두 칸 박스 노드** — 노드를 좌(데이터) 우(next) 두 칸으로 분할한 직사각형으로 그린다. 이중 연결은 칸이 셋(prev·data·next). (R1, R2, R3, R6, R7, R9, R10)
2. **방향 화살표 = 포인터** — 한 노드의 next 칸에서 시작해 다음 노드 본체로 들어가는 단일 화살표. 단일은 단방향, 이중은 양방향 두 줄. (전 레퍼런스)
3. **head/tail 라벨** — 첫 노드 위에 head, 마지막 노드 옆에 tail 텍스트. 머리만 알면 전체에 닿는다는 ADT 약속의 시각적 닻. (R1, R2, R3, R4, R6, R9)
4. **NULL 종료 표식** — 마지막 next 는 X, ⏚, NULL 글자, 사선 등으로 명시. "여기서 끝" 을 시각 사건으로 만든다. (R2, R6, R7, R9)
5. **수평 좌→우 배치** — 노드를 수평으로 늘어놓는 관습. 변종이 둘레로 닫히면 원형, 위아래 두 줄로 묶이면 이중. (R1, R3, R7, R9, R10)
6. **삽입/삭제의 "화살표 두 개 재배선"** — 한 단계 = 화살표 끊기 + 화살표 잇기. 강조색(보통 빨강) 으로 "방금 바뀐 한 화살표" 만 부각. (R2, R3, R6, R10)
7. **변종 비교를 같은 데이터로** — 단일/이중/원형을 같은 다섯 값에 적용해 화살표 개수·방향 차이만 부각. (R3, R4, R7, R9)
8. **버튼 입력 반응형 인터랙션** — Insert/Delete/Search 버튼 + 인덱스/값 입력. 한 클릭 = 한 연산. (R1, R2, R3, R4)

학습자에게 형성된 기대:
- 노드는 자기 데이터와 "다음 한 명" 만 안다.
- head 에서 출발해 화살표를 따라가는 것 외엔 도달 방법이 없다.
- 삽입/삭제는 노드를 "옮기는" 게 아니라 "화살표를 갈아끼우는" 것이다.

## 4. 공백

기존 레퍼런스에서 약하거나 비어있는 영역:

1. **메모리 산포의 시각화** — 거의 모든 도구가 노드를 균등 간격으로 정렬한다. 실제로는 "여기저기 흩어진 셀들" 인데, 산포된 카드 위에 화살표만 잇는 그림은 R6 (basecs) 와 R8 (Haining 의 비판 글) 에서만 명시된다. 운동형으로 "노드가 메모리 어디서나 태어날 수 있다" 를 보여준 사례 부재.
2. **포인터 재배선의 시간 순서** — "먼저 새 노드의 next 를 잇고, 그 다음 이전 노드의 next 를 갈아낀다" 는 안전 순서가 화살표 두 개의 시간차로 명확히 강조된 사례가 적다(R1 이 가장 가까우나, 학습자는 두 화살표가 동시에 변하는 것으로 오해하기 쉬움).
3. **head 의존성의 위험** — head 를 잃으면 리스트 전체에 닿을 수 없다는 사실이 시각적으로 약하다. head 라벨이 떨어져 나가면 어떻게 되는지의 운동을 보여준 도구 없음.
4. **순차 접근 비용의 체감** — 인덱스 N 접근이 화살표를 N번 따라가는 일이라는 사실이 운동으로 강조되지 않음. R1 의 search 애니메이션이 가장 가깝지만 여전히 짧음. "뒤로 갈수록 점점 멀어진다" 의 시각적 압박감 부재.
5. **배열과의 정면 대비** — 같은 데이터에 같은 연산(중간 삽입)을 가했을 때 배열은 N개를 밀어내고, 연결 리스트는 화살표 두 개만 갈아낀다는 시각 대비를 한 화면에서 운동으로 보여준 사례 없음. 정적 비교만 다수.
6. **변종 사이의 트레이드오프 시각** — 단일→이중 추가 시 메모리(prev 칸) 가 늘고 양방향 운동을 얻는다는 균형이 시각화되지 않음. 변종이 그저 "화살표 한 줄 더" 로만 그려짐.
7. **링크의 양보 가능성** — 링크는 갈아 끼울 수 있다는 사실이 자연스럽게 나오는 반면, 데이터(노드 안 값) 는 보통 고정으로 그려진다. "값은 그대로, 위치만 바뀐다" 는 비교적 표현됐지만 그 역(노드는 그대로, 이웃이 바뀐다) 의 강조는 약함.
8. **"잇고 끊는 행위 자체" 의 메타적 시각화** — 분류 보고서의 핵심 학습 질문은 "어떻게 잇고 끊는 행위만으로 순서가 재구성되는가" 인데, 대부분 도구는 결과 상태(이은 후/끊은 후) 만 보여주고 "끊긴 손가락이 다른 노드를 가리키러 가는 운동" 을 사건화한 사례가 적다.
9. **기차 비유의 함정 인지** — R8 만이 가장 인기 있는 비유가 본질을 가릴 수 있음을 지적한다. 학습 자료 대부분은 비유를 무비판적으로 채택. 시각화 설계가 "어떤 비유를 쓰지 않을지" 결정하는 단계가 부족.

## 5. 대상 개념 요약

> 연결 리스트는 노드들이 메모리 어디에든 흩어져 있어도 각자가 "내 다음은 너" 라고 가리키는 단 하나의 링크만으로 한 줄 순서를 만든다. 어떤 위치에 닿으려면 head 에서 시작해 화살표를 한 칸씩 따라가야 하고, 어떤 자리든 삽입·삭제는 노드를 옮기는 일이 아니라 두세 개의 화살표를 끊고 다시 잇는 일이다. 이 한 가지 메커니즘 — 링크 재배선 — 이 단일·이중·원형 모든 변종의 공통 본질이며, 배열의 "연속 메모리 + 위치 계산" 패러다임과 정반대편에 선 자료구조 철학을 대표한다.

학습자가 시각화와 함께 읽을 짧은 캡션:
- "노드는 자기 다음 한 명만 안다." (지역 인식)
- "머리에서 출발하지 않으면 어디에도 닿을 수 없다." (head 의존)
- "옮기지 않는다. 다시 잇는다." (포인터 재배선)
- "흩어져 있어도 한 줄이다." (메모리 산포 vs 논리적 순서)
- "끊는 화살표 하나, 잇는 화살표 둘 — 그게 삽입이다." (재배선 회계)

용어 풀어쓰기:
- node → "값 한 개와 다음을 가리키는 손가락 하나를 가진 작은 카드"
- head → "리스트로 들어가는 단 하나의 입구"
- next → "내 다음 카드를 가리키는 손가락. 갈아 끼울 수 있다"
- NULL → "여기서 끝, 더는 다음이 없음"
- 삽입(insert) → "새 카드의 손가락을 먼저 다음으로 묶고, 이전 카드의 손가락을 새 카드로 옮겨 끼운다"
- 삭제(delete) → "이전 카드의 손가락을, 사라질 카드의 다음으로 곧장 옮겨 끼운다"
- 순회(traverse) → "head 에서 시작해 손가락이 가리키는 곳을 따라 한 칸씩 걷는다"
