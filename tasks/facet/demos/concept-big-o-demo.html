...앞 본문 텍스트가 여기 있다고 가정한다. 알고리즘을 처음 배울 때 우리는 대개 "이게 빨라요? 느려요?" 같은 직관에서 출발하지만, 컴퓨터과학은 이 질문을 훨씬 더 차가운 형태로 다시 묻는다. 입력이 두 배가 되었을 때 시간은 몇 배가 되는가...

이 시각화에 영감을 준 레퍼런스

  • R7. Wheat & Chessboard Problem — 지수 폭발의 정전(canon) 메타포. "후반부에서 상상이 끊어지는" 체감을 차용.
  • R5. XKCD #399 "Travelling Salesman" — "복잡도가 인간 시간을 짓누른다" 의 정서. 같은 시간 예산 위에 클래스를 줄 세우는 기조.
  • R1. samwho.dev "Big O" — "한 단계 = 한 번의 터치" 라는 시간 단위 추상화. 본 데모에서는 "한 단계 = 연료 한 단위" 로 변환.
회피한 표준 레퍼런스: bigocheatsheet.com (색띠 표 + 겹친 곡선), earthinspoon (슬라이더로 곡선 키우기) — B타입은 표준 곡선 그래프를 무대에서 치운다.
Big-O 표기법은 알고리즘이 입력 크기 n 이 커질 때 실행 시간이나 자원 사용이 어떻게 자라는지를 함수의 상한으로 묶어서 부르는 점근 표기법이다. 우리는 1초·1분 같은 절대 시간이 아니라, "같은 작업을 두 배 큰 입력에 시켰을 때 몇 배가 더 걸리는가" 의 성질만을 본다. O(1) 은 입력이 아무리 커져도 시간이 거의 그대로이고, O(n^2) 부터는 입력 두 배에 시간 네 배로 폭발하기 시작하며, O(2^n) 과 O(n!) 은 사실상 풀 수 없는 영역으로 들어간다.
Big-O 는 시계가 아니라 운명을 말한다. 같은 연료 한 통을 일곱 척의 우주선에 똑같이 나눠 주었다. 클래스마다 다른 별 - O(1), O(log n), O(n), O(n log n), O(n^2), O(2^n), O(n!). 같은 연료로 어디까지 가는지가 그 클래스의 운명이다. n 을 키워 보라 - 어떤 우주선이 먼저 떨어지는가.
n = 16 작은 n 에선 운명이 뒤집힐 수도

트랙 우측 끝에 박힌 별일수록 같은 연료로 더 큰 n 을 처리한다. n 을 키우면 좌측에서 별들이 차례로 회색이 되며 떨어져 나간다 - 이것이 점근 클래스의 운명이다. 별을 클릭하면 그 클래스가 처리할 수 있는 n 의 한계가 인간 시간 단위로 환산되어 나타난다.
