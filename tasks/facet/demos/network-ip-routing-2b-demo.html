…앞 본문에서 IP 라우팅의 큰 그림을 짚었다. 패킷은 라우터들 사이를 한 칸씩 던져지며 목적지에 도달하고, 매 라우터는 자기 안의 라우팅 테이블에서 가장 잘 맞는 규칙 하나를 골라 다음 라우터로 보낸다. 이 단락에서는 그 결정 절차의 본질을 풍경으로 다시 만난다…

이 시각화가 참고한 레퍼런스

  • Kurose & Ross 동반 애니메이션 — 라우팅 테이블 lookup 의 step-by-step 결정 절차를 학습자가 한 번에 인지하는 데 필요한 "결정 호흡" 의 개념적 출발점.
  • Practical Networking — Subnetting/Routing — prefix 길이 비교의 시각적 처리(비트 자) 에 대한 영감. 본 데모에서는 "자" 대신 "골짜기 깊이" 로 변환해 차원을 비틀었다.
  • 국토지리정보원 지형도 / 자연 분수계(watershed) — 빗방울이 한 자리의 경사만으로 다음 골짜기를 찾는 "국소적·즉응적 결정" 의 원형. 이 데모의 핵심 메타포 그 자체.
  • Opte Project — Internet Map — 인터넷의 자기조직성을 풍경으로 보는 미적 태도. 풀스크린 우주는 차용하지 않으나, "한 장의 풍경에 결정이 응축된다" 는 자세를 빌렸다.
의도적으로 회피한 표준: Cisco Packet Tracer 의 장비 룸 + 봉투 메타포, GNS3/NetAnim 의 노드+엣지 그래프, BGPlay 의 AS path 다이어그램, 그리고 "라우팅 테이블 = 옆 표" 의 사이드 패널 규약. 이들은 "그래프 위 경로 탐색" 이라는 학습자 기대를 강화한다 — 본 데모는 그 기대를 한 번 끊고 자연 풍경으로 갈아탄다.
인터넷의 패킷은 한 라우터에서 다음 라우터로 한 칸씩 던져지며 목적지에 도달한다. 각 라우터는 패킷의 목적지 IP 만 보고, 자기 안의 라우팅 테이블에서 가장 잘 맞는 규칙(가장 긴 prefix 가 매치되는 행, Longest Prefix Match) 을 골라 다음 라우터로 보낸다. 출발 라우터는 끝까지의 경로를 알지 못한다 — 매 hop 마다 그 라우터가 자기만의 결정을 새로 내린다(Hop-by-hop forwarding). 패킷 머리에 든 TTL 은 hop 마다 1 씩 줄어들어, 길을 잃은 패킷이 영원히 떠도는 일을 막는다.
빗방울은 자기 발 밑 경사만 본다. 봉우리=라우터, 등고선의 깊이=라우팅 규칙의 구체성(prefix 길이). 가장 깊은 골짜기가 이긴다 = longest prefix match.
현재 봉우리: — ↓ 더 깊은 V = 더 구체적인 prefix = 이긴다
목적지 IP 192.168.42.7 TTL

봉우리 위에 빗방울이 머무는 1.2 초가 라우터의 결정이 일어나는 시간이다 — 후보 등고선 여럿이 동시에 빛나다, 가장 좁고 깊은 하나가 빗방울을 자기 골짜기로 데려간다. 빗방울 부피가 줄어드는 정도가 곧 TTL 이다.

…이어지는 본문에서는 같은 결정이 IPv6 (/48, /64) 에서 어떻게 더 깊은 골짜기로 표현되는지, 그리고 한 봉우리의 가장 깊은 골짜기가 메워지면(=라우팅 테이블 행 삭제) 빗방울이 어디로 빠지는지 — 즉 misconfiguration 과 블랙홀 — 를 살펴본다…
