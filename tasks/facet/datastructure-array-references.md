## 2. 레퍼런스 요약

### R1. VisuAlgo — Array (Resize-able Array)

- 출처: visualgo.net (NUS Steven Halim 그룹의 사실상 표준 시각화 도구. 2024년 1월에 Array 모듈이 신규 추가됨)
- URL: https://visualgo.net/en/array , https://visualgo.net/en/array/print
- 한 줄 정체성: 배열을 "용량(M) 안에서 N개 칸이 채워진 압축 셀 띠" 로 그리고, get / search / insert / remove / append / resize 를 같은 화면에서 단계별로 보여주는 표준 인터랙티브 도구.
- 시각 객체: 가로로 늘어선 직사각형 셀 M개. [0..N-1] 은 값으로 채워지고 [N..M-1] 은 회색·빈 칸. 각 셀 위에 인덱스 번호. N(size) 와 M(capacity) 를 별도 라벨로 노출.
- 시간 표현: Step / Play / Reverse 미디어 컨트롤 + 키보드(스페이스 재생, 방향키 단계, +/- 속도). 한 클릭/단계 = 한 셀의 색 변화 + 인덱스 포인터 이동.
- 메타포: "고정 길이 차고 안의 자동차들" — 회색 칸은 빈 자리, 채워진 칸은 점유. 압축(no-gap) 규칙이 강제됨.
- 강조: get(i) 의 즉각성(O(1) 점프) vs insert/remove(i) 의 우/좌 시프트 파동(O(N)). 더블링 resize 가 일어날 때 새 띠로 항목들이 한꺼번에 복사되는 시각.
- 인상적 디테일: 압축 배열 원칙을 시각적으로 강제 — 빈 칸이 가운데 생기지 않는다. resize 시 회색 영역이 두 배로 늘어나는 단일 사건이 amortized 비용을 직관화.
- 약점: 1차원만 다룬다. 메모리 주소·캐시·다차원 배치는 다루지 않음. 셀 간격이 균일해 "연속 메모리" 의 물리성보다 "논리 셀" 인상에 가까움.

### R2. USFCA Galles — Data Structure Visualization (Array-based Stack / Indexed Array Search)

- 출처: cs.usfca.edu/~galles/visualization (DSA 시각화의 고전. 거의 모든 한국 대학 강의 노트가 인용)
- URL: https://www.cs.usfca.edu/~galles/visualization/Algorithms.html , https://www.cs.usfca.edu/~galles/visualization/StackArray.html
- 한 줄 정체성: 가로 셀 + 인덱스 라벨 + 단일 포인터(top / current) 라는 가장 단순한 1990년대식 도식으로 배열-인덱스 관계를 못박은 클래식 도구.
- 시각 객체: 일정 너비의 가로 박스 셀들 + 셀 아래 인덱스 숫자 + 위쪽에 강조 화살표(top / pointer). 강조된 셀은 색이 바뀜.
- 시간 표현: 버튼 클릭 = 한 연산. 이산적 단계. 애니메이션 트위닝은 짧은 색 페이드 + 화살표 이동.
- 메타포: "고정 칸 + 카운터 화살표". 추상에 가까움.
- 강조: 인덱스 → 셀 위치 매핑이 항상 보인다. 인덱스 0 부터 시작하는 관습을 일관되게 강제.
- 인상적 디테일: 동일 ADT(예: 스택) 가 배열 구현과 연결 리스트 구현으로 두 페이지로 분리되어, 학생이 "추상 인터페이스 vs 메모리 모양" 을 구분할 수 있게 함.
- 약점: UI 가 90년대 자바 애플릿 스타일. 색감/모션이 평면적이고 메모리 주소·strides·다차원 표현은 없음. 모바일 미지원.

### R3. freeCodeCamp — Data Structures 101: Arrays — A Visual Introduction for Beginners

- 출처: freecodecamp.org / Estefania Cassingena Navone (시각 입문 글의 정착작)
- URL: https://www.freecodecamp.org/news/data-structures-101-arrays-a-visual-introduction-for-beginners-7f013bcc355a/
- 한 줄 정체성: 컴퓨터 메모리를 "격자(grid)" 로 그리고 그 위에 인접 셀들을 점유하는 띠로 배열을 표현한 입문 시각 해설.
- 시각 객체: 손그림풍 그리드 다이어그램 + 인덱스 0-기반 셀 박스 + 삽입 시 우측 화살표로 시프트 표현 + 삭제 시 좌측 화살표로 압축 표현.
- 시간 표현: 정적 다이어그램 시퀀스(Before / After 두 컷). 단계마다 별도 그림.
- 메타포: "메모리 격자 위에 옆집끼리 손잡고 사는 셀들". 인접성(adjacency) 자체가 메타포.
- 강조: O(1) 임의 접근의 근거가 "주소 = 시작 + 인덱스 × 크기" 라는 산술이라는 점. 끝에서의 push 는 싸고 중간/앞에서의 insert 는 비싸다는 비대칭.
- 인상적 디테일: "왜 인덱스가 0부터 시작하는가" 를 메모리 오프셋 그림으로 자연스럽게 설명.
- 약점: 정적 그림이라 운동감 없음. 동적 배열 resize 의 시각적 사건을 별도로 다루지 않음.

### R4. CS50 (Harvard) — Lecture 3 / Week 2: Arrays as Lockers

- 출처: cs50.harvard.edu (David Malan, 컴퓨팅 교육 표준 강의)
- URL: https://cs50.harvard.edu/x/weeks/2/ , https://cs50.harvard.edu/x/2026/notes/3/
- 한 줄 정체성: 배열을 "강의실 앞에 줄지어 선 빨간 사물함 7개" 로 비유하고, "한 번에 한 칸만 열 수 있다" 는 물리 제약을 선형 검색의 동기로 직결시키는 강의용 도식.
- 시각 객체: 무대 위 실제 빨간 사물함 소품 + 슬라이드의 셀 그림. 사물함 각각에 인덱스 0..6 라벨. 강사가 직접 한 칸씩 문을 여는 퍼포먼스.
- 시간 표현: 강의 시간에 맞춰 강사가 손으로 한 칸씩 여는 라이브 진행. 슬라이드는 정적이지만 행위가 시간을 만든다.
- 메타포: 사물함 — 물리적 칸막이, 잠금, 한 번에 하나만 열림.
- 강조: "임의 접근은 사실 인덱스를 알아야만 빠르다. 모르면 하나씩 열어봐야 한다" — 배열 자체의 O(1) 와 검색의 O(N) 을 분리.
- 인상적 디테일: 물리 소품을 통해 "옆에 붙어 있다(contiguous)" 는 추상이 즉각적인 시각 사실로 변환됨.
- 약점: 삽입/삭제·동적 크기·다차원은 이 비유로 잘 확장되지 않음(고정 사물함이 모순). 셀 내부의 값 변경(write) 비유는 약함.

### R5. Eli Bendersky — Memory Layout of Multi-Dimensional Arrays

- 출처: eli.thegreenplace.net (시스템 프로그래밍 해설의 표준)
- URL: https://eli.thegreenplace.net/2015/memory-layout-of-multi-dimensional-arrays
- 한 줄 정체성: 다차원 배열을 "선형 메모리 띠 위에 어떤 순서로 펼치느냐" 의 두 가지 답(row-major / column-major) 으로 이원화해 그린 시스템 시각.
- 시각 객체: 위쪽에 2D 그리드(논리 형태) + 아래쪽에 그 그리드를 풀어 펼친 1D 메모리 띠 + 회색 셀에 메모리 주소 라벨 + 파란 화살표로 접근 순서 표시.
- 시간 표현: 정적 다이어그램 시퀀스. 행우선 vs 열우선을 대비 컷으로 보여줌.
- 메타포: "책장에서 책을 어느 방향으로 꽂느냐" — 같은 책장(논리) 라도 꽂는 방향(메모리) 이 다르면 옆 책을 집는 비용이 달라짐.
- 강조: 인덱스 (i,j) → 메모리 오프셋 = i × stride_row + j × stride_col 라는 산술의 시각화. 캐시 친화 vs 비친화 접근 패턴.
- 인상적 디테일: 같은 그리드 위에 두 가지 화살표 경로(가로 스캔 / 세로 스캔) 를 겹쳐 그려 "어느 패턴이 띠 위에서 점프하는가" 를 즉각 보여줌.
- 약점: 정적·1D-2D만 다룸. 인터랙션 없음. 1차원 배열만 학습하는 입문자에게는 과한 추상.

### R6. Python Tutor — Code Visualizer (List / Array 모드)

- 출처: pythontutor.com (Philip Guo, 컴퓨팅 교육 표준 도구)
- URL: https://pythontutor.com/visualize.html
- 한 줄 정체성: 배열·리스트 변수를 "이름표가 달린 셀 묶음 + 화살표로 연결되는 객체" 로 그리고, 각 코드 줄 실행 직후의 메모리 상태를 단계 이동으로 보여주는 코드 시각화 도구.
- 시각 객체: 좌측 코드 패널 + 현재 줄 화살표, 우측 프레임 영역에 변수 이름 + 인덱스 0..n-1 가 붙은 셀 박스. 객체 참조는 화살표로 표현.
- 시간 표현: Step Forward / Back 양방향. 코드 한 줄 = 셀 하나의 변경 사건.
- 메타포: "강사가 칠판에 그리는 변수 그림" — 교실 판서를 디지털화.
- 강조: 같은 셀의 값이 여러 시점에 어떻게 갱신되는지(write 의 시간성), 슬라이싱·할당의 참조 의미.
- 인상적 디테일: 시간 양방향 — 잘못된 인덱스 할당으로 망가진 셀을 되돌리며 인과를 뒤집어 학습.
- 약점: 인덱스 접근의 O(1) "점프" 인상이 약하다(모든 셀이 한꺼번에 보임). 동적 resize 가 사건으로 시각화되지 않음.

### R7. Wikipedia — Dynamic Array

- 출처: en.wikipedia.org (표준 백과)
- URL: https://en.wikipedia.org/wiki/Dynamic_array
- 한 줄 정체성: 동적 배열의 핵심 사건인 "용량 두 배 재할당" 을 정적 다이어그램과 거북이 아이콘으로 명료화한 표준 도식.
- 시각 객체: 가로 셀 띠 + 채워진 칸(파란/색칠) + 회색 예비 칸 + 일부 단계에 거북이 아이콘(느린 삽입). size 라벨과 capacity 라벨이 별도로 표시.
- 시간 표현: 삽입 시퀀스의 정적 컷 다수 — i번째 삽입에서 띠가 두 배로 늘어나는 사건이 한 컷으로 강조됨.
- 메타포: "이사" — 헌 집(작은 띠) 의 모든 짐을 새 집(두 배 띠) 으로 옮기고 헌 집을 버린다.
- 강조: 대부분의 push 는 빠르고(상수), 가끔 한 번이 매우 느리며(선형), 평균하면 상수(amortized) 라는 비용 구조. 회색 영역이 곧 미래의 성장 여지.
- 인상적 디테일: 거북이 아이콘은 시각적 사건성으로 amortized 분석의 직관적 근거를 만든다. "느린 삽입을 본 학생은 다음에 빠른 삽입의 행운을 누린다."
- 약점: 정적이라 복사 운동감이 약함. capacity 가 두 배가 되는 그 순간의 시간 압축이 단일 컷으로만 표현됨.

- 약점: 1D 배열이 아닌 항상 2D 격자라 1차원 학습에는 차원 잉여. 삽입·삭제는 "행 삽입/삭제" UX 로 흡수되어 셀-시프트 비용이 잘 안 보인다(스프레드시트가 자동 처리).

### R9. Jay Alammar — A Visual Intro to NumPy and Data Representation

- 출처: jalammar.github.io (시각 머신러닝 교육의 표준)
- URL: https://jalammar.github.io/visual-numpy/
- 한 줄 정체성: 1D / 2D / 3D NumPy 배열을 "색칠된 사각 블록의 적층" 으로 그려, 배열을 모양과 연산이 있는 시각 객체로 정착시킨 해설.
- 시각 객체: 1D는 가로 셀 띠, 2D는 격자, 3D는 격자의 적층. 각 셀에 숫자. 브로드캐스팅·슬라이싱은 색깔 영역의 합성/잘림으로 표현.
- 시간 표현: 정적 다이어그램 시퀀스. 연산 전 / 연산 후 두 컷.
- 메타포: "도형 블록 (shape) — 같은 모양은 자유로이 합쳐지고, 다르면 정렬해야 한다".
- 강조: shape 와 axis 가 곧 배열의 정체성. 인덱스 접근은 "어느 축의 어느 칸" 으로 분해됨.
- 인상적 디테일: 색을 통해 데이터의 흐름·집계·축소를 시각적 사건으로 보여줌. 1D → 2D → 3D 의 자연스러운 확장.
- 약점: 정적·과학 계산 맥락 위주. push/insert/delete 같은 자료구조 ADT 연산은 거의 다루지 않음.

### R10. 한국어 자료 — 나무위키 배열 / opentutorials 자료구조 배열

- 출처: namu.wiki + opentutorials.org (한국어 검색 시 상위 노출)
- URL: https://namu.wiki/w/%EB%B0%B0%EC%97%B4 , https://opentutorials.org/module/1335/8677
- 한 줄 정체성: 한국어 학습자가 배열을 처음 만나는 "정의 + 인덱스 + 시간복잡도" 텍스트 정리. 그림은 보조 다이어그램 한두 장.
- 시각 객체: 가로 셀 박스 + 인덱스 라벨. 삽입/삭제 시 화살표로 shift 표시. 한국어 캡션.
- 시간 표현: 정적 그림. 인터랙션 없음.
- 메타포: 본문 비유 부재. "연속된 공간" 이라는 표현만 반복.
- 강조: 임의 접근 O(1) / 삽입·삭제 O(N) / 크기 고정 같은 성능 표가 본문의 중심.
- 인상적 디테일: 한국 학습자가 "첨자(index) 는 0부터" 라는 관습을 한국어 문장으로 처음 명시되는 자료.
- 약점: 시각적 사건성이 거의 없다. 메모리 주소·다차원·resize 모두 텍스트로만 다룸. 동적 인터랙션 부재.

## 3. 공통 요소

거의 모든 레퍼런스가 공유하는 시각 언어:

1. **연속된 가로 셀 띠** — 동일 너비의 직사각형 박스를 좌→우로 인접 배치. "옆에 붙어 있다" 는 contiguity 자체가 시각 자명. (R1, R2, R3, R5, R7, R8, R9, R10)
2. **인덱스 라벨 0..N-1** — 각 셀 아래·위에 0부터 시작하는 숫자. 인덱스가 셀의 정체성을 결정한다는 약속. (전 레퍼런스)
3. **인덱스 → 셀 직접 점프** — 임의 접근을 표현할 때 화살표가 중간 셀을 건드리지 않고 곧장 목표 셀로 이동. O(1) 의 시각적 근거. (R1, R2, R3, R6, R8)
4. **삽입/삭제의 시프트 파동** — insert(i) / remove(i) 시 i 이후 셀들이 한 칸씩 우/좌로 도미노처럼 밀린다. O(N) 의 시각적 근거. (R1, R2, R3, R7, R10)
5. **size vs capacity 의 색 분리** — 점유 셀은 채색, 예비 셀은 회색/빈 칸. 두 개념이 같은 띠 위에서 분리되어 보임. (R1, R7)
6. **인덱스 접근의 단일 화살표 / 포인터** — top / current / i 라는 단일 포인터가 셀 위에 떠다니며 "지금 만지는 자리" 를 명시. (R1, R2, R6)
7. **버튼 기반 입력 반응형 인터랙션** — get(i) / insert(i,v) / remove(i) 버튼 + 인덱스/값 입력 필드. 한 클릭 = 한 연산. (R1, R2)
8. **2D 확장 시 격자(grid)** — 1차원이 부족할 때 항상 행×열 격자로 확장. 좌표 (i,j) 로 셀 지정. (R5, R8, R9)

학습자에게 형성된 기대:
- 배열은 가로로 늘어선 같은 모양의 칸들이며, 각 칸은 0부터 매겨진 번호로 호명된다.
- 인덱스만 알면 어느 칸이든 한 번에 갈 수 있다(O(1)).
- 중간을 비우거나 끼워 넣으면 옆 칸들이 줄줄이 밀린다(O(N)).
- 칸이 다 차면 더 큰 새 띠로 통째로 옮겨간다(드물지만 비싸다).

## 4. 공백

기존 레퍼런스에서 약하거나 비어있는 영역:

1. **"연속 메모리" 의 물리성** — 대부분의 시각화는 셀 사이에 균등한 간격을 두어 셀을 "논리 칸" 으로만 그린다. 셀이 물리적으로 붙어 있고 그 옆에는 다른 변수의 영역이 곧장 시작된다는 contiguity 의 압박감을 강조한 인터랙티브 도구가 거의 없다(R5 의 정적 다이어그램이 유일).
2. **인덱스 점프의 산술적 근거** — "인덱스 i 가 곧 base + i × stride" 라는 주소 산술을 셀 위에 동적으로 그린 시각이 부족하다. R5 만 정적으로 시도. 학생은 "왜 O(1) 인가" 를 산술이 아니라 마법으로 받아들이게 된다.
3. **시프트 파동의 비용 누적** — insert/remove 의 도미노가 한 번 일어날 때는 잘 보이지만, "여러 번 반복하면 비용이 어떻게 누적되는가" 를 누적 막대·열량계 같은 보조 시각으로 보여준 사례가 없다(R7 의 거북이 아이콘이 가장 근접).
4. **resize 사건의 시간 압축** — 더블링 시 전체 항목 복사가 단일 컷으로만 그려진다(R7). 회색 영역이 두 배로 늘어나는 그 순간의 운동(이사 메타포) 을 단계별 운동으로 풀어낸 인터랙티브가 거의 없다.
5. **두 시간축의 동시 강조** — 외부 입력 시간(사용자의 연산 호출 순서) 과 내부 시간(셀의 색·위치 변화) 을 한 화면에서 직교 운동으로 보여주는 사례 부재. 입력 반응형 모델의 본질이 이 두 축의 매핑인데도 시각화가 약하다.
6. **빈 셀 vs 미할당 영역의 구분** — 회색 셀이 "할당됐지만 비었다" 인지 "아예 없다" 인지가 도구마다 다르고 모호하다. R1 은 명시하지만 R3/R10 은 둘을 합쳐 그린다.
7. **임의 접근 vs 순차 검색의 대비** — get(i) 의 직접 점프와 search(v) 의 순차 스캔이 같은 화면에서 운동 대비로 보여지는 사례가 드물다. CS50(R4) 는 이 대비를 강의 퍼포먼스로만 만들고 도구화하지 않았다.
8. **"빈 칸이 가운데 못 생긴다" 는 압축 규칙의 정서** — VisuAlgo(R1) 만 명시적으로 강제하나, 학생이 "왜 못 만드는가" 를 시각적 사건(셀들이 자동으로 모인다) 으로 체감하기 어렵다. 강제 압축의 시각적 동기가 약하다.
9. **다차원과 1차원의 연결** — R5/R9 가 다차원 메모리 펼침을 그리지만, 1차원 배열을 처음 배우는 학습자가 "내 1D 배열도 사실 메모리 위의 띠다" 를 느끼게 잇는 다리가 부족하다.
10. **메타포의 통합 모델** — 사물함(R4) / 메모리 격자(R3) / 스프레드시트(R8) / NumPy 블록(R9) 가 모두 같은 "인덱스 + 연속" 이라는 사실을 한 화면에서 토글하며 보여주는 자료가 없다. 각 비유가 각자의 페이지에서 따로 다뤄짐.

## 5. 대상 개념 요약

> 배열은 같은 종류의 값들을 메모리 위에 옆자리끼리 붙여 늘어놓고, 각 자리에 0부터 시작하는 번호(인덱스) 를 매겨 호명할 수 있게 만든 자료구조다. 인덱스만 알면 시작 주소에 인덱스를 더하는 산술 한 번으로 어느 칸이든 즉시 도달하므로 임의 접근이 O(1) 이다. 그러나 중간에 끼워 넣거나 빼낼 때는 옆 칸들이 줄줄이 밀려 O(N) 의 비용이 들고, 칸이 다 차면 더 큰 띠로 모든 항목을 통째로 옮겨야 한다. 이 한 점에서 임의 접근의 빠름과 구조 변경의 느림이 동시에 나오며, 배열은 "위치를 알면 빠르고, 위치를 바꾸면 비싸다" 는 비대칭의 원형이 된다.

학습자가 시각화와 함께 읽을 짧은 캡션:
- "옆자리끼리 붙어 있다." (연속 메모리)
- "번호만 알면 한 번에 간다." (O(1) 임의 접근)
- "가운데를 끼우면 옆 사람이 다 밀려난다." (O(N) 시프트)
- "칸이 다 차면 두 배 큰 새집으로 이사간다." (resize / amortized)

용어 풀어쓰기:
- 인덱스(index) → "0부터 매겨진 자리 번호"
- size → "지금 채워진 칸의 수"
- capacity → "지금 띠가 가진 전체 칸의 수(미래의 성장 여지 포함)"
- 임의 접근 → "번호만 알면 옆 칸들을 거치지 않고 곧장 도달한다"
- shift → "한 칸을 끼우거나 빼내려고 옆 칸들을 일제히 한 자리씩 옮기는 운동"
- resize / 더블링 → "다 찬 띠를 버리고 두 배 큰 띠로 모든 항목을 옮기는 이사"
