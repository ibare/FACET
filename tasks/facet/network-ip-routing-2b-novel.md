# 네트워크 계층 IP 라우팅 — 새로운 방식 기획 (등고선 유역 메타포)

## 0. 기본 정보

- **대상 이름**: 네트워크 계층 IP 라우팅 (IPv4 forwarding)
- **도메인/카테고리**: CS 기초 / 네트워크
- **대상의 본질**: 패킷이 라우터마다 목적지 IP 만 보고 라우팅 테이블에서 longest prefix match 로 다음 홉을 결정해 hop-by-hop 으로 전달되며, TTL 이 매 홉마다 1 줄어 0 이 되면 폐기되는 분산적 결정 과정.
- **이 기획의 접근**: 새로운 방식
- **선택한 출발점**: 보고서가 후보로만 언급하고 깊이 파지 않은 "강물의 유역(watershed)" 메타포. 보고서 §4 의 "가장 가까운 골짜기로 흐르는 물 = longest prefix match" 한 줄을 출발점으로 삼아, **등고선 패치(contour patch)** 라는 새 시각 언어로 발전시킨다.
- **참조한 레퍼런스**:
  - 영감을 준 것: 보고서 §4 강물 유역 메모, 지형 등고선 지도(국토지리정보원 지형도), 모래 위에 물 한 방울이 갈라지며 흐르는 자연 현상, 산악 분수계(divide) 개념.
  - 적극적으로 회피한 것: Cisco Packet Tracer / GNS3 / Kurose 의 "노드+엣지+움직이는 점", BGPlay 의 AS 그래프, Practical Networking 의 비트 자(ruler) 정렬 격자 (이쪽은 공통 강화 에이전트가 다듬고 있을 가능성). Opte/CAIDA 의 풀스크린 우주 풍경(임베드 폭 부적합).

## 1. 시그니처 행동

본질은 표준과 동일하다.
1. 패킷이 출발 라우터에 도착한다.
2. 라우터가 목적지 IP 만 보고 라우팅 테이블에서 longest prefix match 를 찾는다.
3. 매치된 다음 홉으로 패킷을 던진다. TTL 이 1 줄어든다.
4. 도착한 라우터에서 1~3 을 반복한다.
5. 목적지 도착 또는 TTL=0 으로 폐기.

시각화 방식만 다르다. 라우터를 동그라미로 그리지 않는다. 패킷을 점으로 그리지 않는다. 라우팅 테이블을 표로 옆에 두지 않는다. 대신 **등고선 패치 한 장**으로 모든 것을 그린다.

## 2. 시각적 은유

**메타포: 빗방울이 등고선 지형 위를 흘러내린다**

출처:
- 자연 현상의 분수계(watershed divide). 한 방울의 빗물이 어느 골짜기로 떨어지는가는 그 자리의 지형 경사가 결정한다. 한 라우터가 한 패킷을 어느 다음 홉으로 보내는가는 그 라우터의 라우팅 테이블이 결정한다 — 둘 다 **국소적이고 즉각적인 결정**이다.
- 등고선 지도. 등고선의 깊이는 prefix 의 길이를 표현한다. 더 깊은 골짜기 = 더 긴 prefix = longest prefix match 의 승자.

**왜 이 메타포가 본질을 왜곡하지 않는가**:
- 빗방울은 출발 시점에 "어디로 흘러갈지" 전체 경로를 알지 못한다. 매 분수계에서 그 자리의 경사가 결정한다 → **hop-by-hop** 의 정확한 비유.
- 빗방울이 보는 것은 "발 밑의 경사" 뿐, 빗방울이 어디서 왔는지 지형은 모른다 → **목적지 기반(자기 위치 기반) 결정**, 출발지 무관.
- 등고선이 깊을수록 좁고 구체적이다 → **prefix 가 길수록 매치 범위가 좁고 구체적**. longest prefix match = 가장 깊이 파인 골짜기를 따라간다.
- 빗방울은 무한히 흐르지 않는다. 햇볕에 증발한다 → **TTL 감소**. 시각적으로 빗방울의 크기/투명도가 줄어들다 0 이 되면 사라진다.

**임베드 폭(640~720px) 작동 검증**:
- 등고선 지형은 한 장의 평면 패치. 가로 660px × 세로 360px 안에 5~7 개 라우터(=분수계 봉우리) 와 그 사이 골짜기들을 충분히 그린다.
- 스크롤이나 풀스크린 가정 없음. 마크다운 본문 한 단락 옆에 자연스럽게 들어가는 정사각형에 가까운 가로 직사각형.
- 핵심 비교는 한 화면 안에 동시에 보여야 하는 "같은 지점에서 만나는 여러 길이의 등고선" 인데, 이는 격자가 아니라 동심원형 패턴이라 폭에 더 너그럽다.

## 3. 결정적 순간

새 시각 언어 안에서의 결정적 순간 셋.

1. **빗방울이 분수계 마루(crest) 위에 정확히 떨어졌을 때**
   - 표준 시각화의 "라우터에 도착해 잠깐 멈춤" 에 해당.
   - 이 순간 빗방울 주변에 **여러 등고선 곡선이 동시에 빛난다** — 이 자리에서 매치되는 prefix 후보들이다.
   - 가장 좁고 깊은 등고선이 천천히 부풀어 빗방울을 그쪽 골짜기로 끌어당긴다. 다른 후보 등고선은 흐릿해진다.
   - = **longest prefix match 의 시각적 결착**. 비트 자나 표 행 highlight 가 아니라 "가장 깊은 골짜기가 이긴다" 는 지형적 직관.

2. **다음 봉우리에 도착해 다시 결정이 일어나는 순간**
   - 첫 번째 결정과 똑같은 애니메이션이 반복된다.
   - 이것이 핵심: 표준 시각화는 "라우터를 통과한다" 처럼 보이지만, 등고선 메타포에서는 **각 봉우리마다 빗방울이 새로 떨어지는 것처럼** 결정 절차가 처음부터 다시 일어난다.
   - = **hop-by-hop 의 본질**. 출발 라우터가 끝까지 알지 못한다는 사실이 시각적으로 강제된다.

3. **빗방울이 증발해 사라지는 순간**
   - 골짜기를 헤매다 TTL=0 이 되면 빗방울이 마지막 봉우리 위에서 작아지다 ~ 한 점의 김으로 흩어지며 사라진다.
   - = **TTL 의 의미적 시각화**. 숫자가 줄어드는 것이 아니라 **물방울 자체의 부피가 줄어든다**.

## 4. 다른 대상과의 구별점

같은 분야(라우팅/포워딩) 의 다른 대상에 이 메타포를 적용하면 결과가 명확히 달라진다.

- **출발지 기반 라우팅(source routing)** 이라면? 빗방울이 떨어진 지점이 아니라 "이전에 어느 골짜기를 거쳐 왔는가" 가 다음 흐름을 결정해야 한다. 등고선 지형 자체로는 표현 불가능 — 빗방울에 깃발(이력) 을 달아야 하는데, 그 순간 메타포가 깨진다. 즉 IP 의 **목적지 기반 결정** 이 이 메타포에 깔끔히 들어맞는다는 강한 증거.
- **회로 교환(circuit switching)** 이라면? 출발 시점에 전체 경로(파이프) 가 미리 깔린다. 등고선 지도 위에 **미리 파인 운하**가 그려져야 한다 — 정적 운하 vs 동적 등고선의 대비로 자연스럽게 구별된다.
- **Distance Vector / Link State 같은 라우팅 알고리즘** 이라면? 등고선 지형 자체가 "어떻게 이 모양으로 형성되었는가" 가 강조된다 — 라우팅 알고리즘은 등고선을 새기는 지질학적 과정이고, IP 포워딩은 그 위에 떨어진 빗방울이다. 즉 이 메타포는 자연스럽게 control plane(등고선 형성) 과 data plane(빗방울 흐름) 을 분리한다.
- **MPLS 라벨 스위칭** 이라면? 빗방울이 "나는 골짜기 17 번을 따른다" 는 라벨을 들고 있어, 그 라벨이 분수계마다 새 라벨로 바뀐다. 등고선 자체가 무관해진다 — 메타포의 주인공이 바뀐다.

## 5. 달성해야 할 시각 요소

1. **등고선 패치(contour patch)** — 라우터 = 봉우리(분수계), 라우팅 테이블 = 그 봉우리에서 갈라지는 등고선들의 깊이/방향. 표준의 "노드+엣지+표 사이드패널" 을 한 장에 융합.
2. **흐르는 빗방울** — 점이 아니라 **부피 있는 물방울**. 크기 = TTL 잔량. 진행 경로는 등고선의 골을 따라 굽이친다(직선 엣지 아님).
3. **결정 펄스의 깊이 비교** — 봉우리에 닿는 순간 후보 등고선들이 동시에 깜빡이며 깊이를 비교한다(가장 깊은 게 이긴다). 표가 아닌 **3D 단면 단축 표현** 으로 보여 줄 수 있음(작은 단면도 인셋).
4. **목적지의 위치** — 지형 우측 상단의 빛나는 별. 빗방울은 별을 보지 않는다(자기 발 밑 경사만 본다) — 별은 학습자에게만 보이는 정답 표시.
5. **증발 게이지 없음** — 명시적 TTL 숫자 게이지는 두지 않는다. 빗방울 크기 자체가 게이지다. 인스펙터에서만 숫자 노출.

표준 레퍼런스의 시각 요소(원+선 그래프, 점-패킷, 사이드 표) 와 명확히 다르다.

## 7. 레이아웃

- **640~720px 폭** 기본. 880px 확장 시 등고선 패치만 가로로 늘어남.
- 등고선 패치가 면적의 70% 점유.
- 봉우리(라우터) 는 가로로 약하게 사인 곡선 형태로 배치 — 출발지(좌하) → 목적지(우상). 일직선 배치는 그래프 메타포로 회귀하므로 회피.
- 봉우리 사이 골짜기는 **반드시 곡선**. 직선이면 엣지처럼 보여 표준 토폴로지로 환원된다.
- 단면 인셋은 등고선 패치 하단에 가로로 길게(660px × 80px). 현재 빗방울이 서 있는 봉우리의 단면을 옆에서 본 그림 — 세로축이 깊이(=prefix 길이), 횡축이 좌→우 분기. 가장 깊은 V 자가 longest prefix match.

## 8. 진행과 이벤트

표준의 "play / pause / step" 은 유지하되, **시간 표현 자체를 바꾼다**.

- **선형 시간이 아니라 호흡 시간**. 빗방울이 봉우리 위에 도달하면 "결정 호흡" 이 일어난다 — 등고선들이 들숨처럼 동시에 부풀고, 가장 깊은 것이 날숨처럼 빗방울을 데려간다. 이 호흡이 한 hop 단위.
- 봉우리 사이 흐름은 빠르게(0.6초), 봉우리 위 호흡은 천천히(1.2초). **결정에 시간을 더 준다** — 표준의 "라우터를 슉 지나간다" 와 정반대.
- step 모드는 호흡 단위로 끊는다. 한 번 누르면 한 봉우리 결정 + 다음 골짜기 흐름까지.
- 이벤트 종류:
  - `raindrop_arrives(routerId)` — 빗방울이 봉우리에 도달.
  - `contours_pulse(candidates: prefix[])` — 후보 등고선들 동시 펄스.
  - `deepest_wins(prefix, nextHop)` — 가장 깊은 등고선 선택.
  - `flow_to(nextHop)` — 다음 봉우리로 흐름.
  - `ttl_evaporate(remaining)` — 빗방울 부피 감소.
  - `dropped_ttl_zero` / `arrived_destination`.

## 9. 초기 상태와 파라미터

- **지형**: 5 개 봉우리(라우터). 각 봉우리에 3~5 개 등고선 prefix.
  - R1 (출발): `0.0.0.0/0 → R2`, `192.168.0.0/16 → R2`, `10.0.0.0/8 → R3`.
  - R2: `0.0.0.0/0 → R4`, `192.168.0.0/16 → R3`, `192.168.42.0/24 → R5`.
  - R3, R4: 일부 prefix 와 default route.
  - R5 (목적지 인접): `192.168.42.0/24 → host`.
- **빗방울**: 목적지 IP `192.168.42.7`, TTL=8.
- **등고선 깊이 매핑**: prefix 길이 0(default) = 가장 얕은 등고선 1 단, /8 = 2 단, /16 = 3 단, /24 = 4 단, /32 = 5 단. 시각적으로 깊이 차이가 분명히 보일 것.
- **목적지 별** ★ 은 R5 쪽에 빛나지만 빗방울 자체에는 영향 없음 — 단지 학습자에게 정답 표시.
- 인터랙션:
  - 다른 목적지 IP 입력 → 같은 지형 위 다른 흐름 경로.
  - 한 봉우리의 가장 깊은 등고선을 사용자가 "메우면"(클릭으로 비활성화) → 빗방울이 그 봉우리에서 다른 골짜기로 빠진다(블랙홀 시연).
  - TTL 시작값 슬라이더(2~16). 작게 두면 도착 전에 증발.

## 10. 본질 보존 검증

다섯 본질 개념 각각이 새 표현에서 어떻게 보존되는가, 그리고 잘못 표현될 위험과 방지.

### 목적지 기반 결정
- **보존**: 빗방울이 봉우리 위에 떨어진 자리 = 그 봉우리에서 본 목적지 IP. 빗방울 자체에 출발지 정보가 없음. 등고선은 빗방울이 어디서 왔는지 모르고 그 자리의 깊이만으로 결정한다.
- **위험**: 빗방울에 "꼬리" 같은 이전 경로 흔적을 그리면 source-routing 처럼 보인다 → 꼬리 그리지 않음. 빗방울 뒤의 흐름선은 학습자 보조용으로만 가늘게 fade out.
- **방지**: 인스펙터에서 빗방울을 클릭하면 "내가 보는 것: 목적지 IP 만" 명시.

### Hop-by-hop
- **보존**: 봉우리마다 결정 호흡이 처음부터 다시 일어난다. 출발 봉우리 R1 의 등고선은 R2 의 등고선과 무관하게 그려진다. R1 시점에 R2~R5 의 등고선은 흐릿해서 잘 안 보인다(현재 봉우리만 선명).
- **위험**: 학습자가 등고선 지형 전체를 한눈에 보면 "전체 경로가 미리 결정되어 보이는" 착각이 생길 수 있음.
- **방지**: 현재 빗방울이 위치한 봉우리의 등고선만 진하게, 나머지 봉우리는 회색조. 빗방울이 이동할 때마다 진한 영역이 옮겨 간다. = "빗방울은 자기 발 밑만 본다" 의 시각적 강제.

### 라우팅 테이블 lookup
- **보존**: 봉우리에서 동시에 빛나는 후보 등고선들이 곧 라우팅 테이블의 매치 후보 행들. 즉석 계산이 아니라 **이미 새겨진 등고선** 을 읽는다는 사실이 강조된다.
- **위험**: 등고선이 자연 풍경처럼 보이면 "결정이 어디서 왔는지" 가 흐려진다.
- **방지**: 인스펙터에서 봉우리 위 등고선들을 prefix 텍스트(`192.168.0.0/16` 등) 로 라벨링. 단면 인셋에서는 깊이 옆에 prefix 길이 숫자(`/16`, `/24`) 를 명기.

### Longest Prefix Match
- **보존**: 단면 인셋에서 V 자 골짜기들의 깊이 비교가 본질. 가장 깊은 V 가 빗방울을 끌어당긴다. 길이 비교가 아니라 **깊이 비교** 로 차원이 변환되었지만 의미는 동일(prefix 가 길수록 더 구체적, 등고선이 깊을수록 더 좁고 구체적).
- **위험**: 학습자가 "왜 더 깊은 게 이기는가" 를 모를 수 있음.
- **방지**: 단면 인셋에 짧은 라벨 — "더 좁고 깊은 골짜기 = 더 구체적인 규칙 = 이긴다". 메타포 안내 텍스트에 한 줄 포함.

### TTL 감소
- **보존**: 빗방울의 부피가 매 hop 마다 줄어든다. TTL=8 → 8 단계의 부피 차이. 인스펙터 호버 시 숫자도 노출.
- **위험**: 부피가 너무 천천히 줄면 "줄어드는 중" 이 안 느껴짐.
- **방지**: 부피 감소를 비선형으로 — 마지막 3 hop 에서 가속. 0 도달 시 한 번 펄스 후 김으로 흩어지는 명시적 애니메이션.

## 11. 표준 대비 가치

표준 레퍼런스(Packet Tracer, Kurose) 가 주지 못하는 통찰을 새 메타포가 준다.

1. **결정의 국소성을 풍경으로 본다**. 표준은 "테이블을 본다 → 다음 홉을 고른다" 의 절차를 강조한다. 등고선 메타포는 "발 밑 경사가 즉각 결정한다" 를 풍경 자체로 전달한다 — 라우팅이 **자연 현상에 가까운 즉응적 결정** 이라는 감각.
2. **prefix 길이의 의미가 시각화된다**. 표준의 "/24 가 /16 보다 더 길다 → 더 구체적" 은 학습자가 외워야 한다. 등고선의 깊이는 즉시 보인다 — 더 깊은 골짜기로 빗방울이 빠진다는 사실은 자연 직관이다.
3. **control plane 과 data plane 의 분리가 자연스럽다**. 등고선이 어떻게 새겨졌는지(라우팅 알고리즘) 와 빗방울이 어떻게 흐르는지(포워딩) 가 같은 그림 안에서 분리된다. 표준은 둘을 같은 노드+엣지 그래프에 섞어 그린다.
4. **TTL 의 의미가 단순한 숫자가 아니다**. "수명 = 부피 = 증발" 의 자연 비유는 "왜 TTL 이 있는가(루프 패킷이 영원히 떠도는 것 방지)" 를 말로 설명하지 않아도 느끼게 한다.
5. **블랙홀과 misconfiguration 의 직관**. 한 봉우리의 가장 깊은 골짜기를 메우면 빗방울이 다른 곳으로 빠진다 — 이 대화형 시연은 표준의 "라우팅 테이블 행을 지운다" 보다 훨씬 즉각적이다.

## 12. 글 흐름 안에서의 위치

- **글의 위치**: IP 라우팅 챕터의 본문 중 "라우팅 테이블이란 무엇인가, 왜 longest prefix match 인가" 단락 직후. 학습자가 개념을 한 번 텍스트로 본 직후에 풍경으로 다시 만난다.
- **시각화 안에 포함될 짧은 설명 텍스트**(보고서 §6 활용, 28px 영역에 1~2 줄로 압축):
  > 인터넷의 패킷은 매 라우터마다 목적지 IP 만 보고 다시 결정된다. 이 그림에서 빗방울 = 패킷, 봉우리 = 라우터, 등고선의 깊이 = 라우팅 규칙의 구체성(prefix 길이). 가장 깊은 골짜기가 이긴다 — 이게 longest prefix match.
- **메타포 안내**(처음 보는 학습자용, 단면 인셋 위 또는 호버 툴팁):
  > 빗방울은 자기 발 밑 경사만 본다. 어디서 왔는지, 끝까지 어디로 갈지 모른다. 매 봉우리에서 그 자리의 등고선이 다음 골짜기를 정한다. 이것이 hop-by-hop forwarding 의 본질이다.

## 부록: 기획 메모

### 자가 점검 결과

- **공통 요소 회피**: 노드+엣지 그래프, 표 사이드패널, 점-패킷 모두 버림.
- **본질 왜곡 없음**: 다섯 본질 각각의 보존 메커니즘을 §10 에 명시.
- **결정적 순간 강조**: 봉우리 위 결정 호흡이 가장 시각적으로 두드러지는 순간.
- **다른 대상에 적용 시 결과 다름**: source routing / circuit switching / MPLS 각각 등고선 메타포에서 다르게 깨지거나 다르게 표현됨(§4).
- **임베드 폭**: 660×460px 기본, 880px 확장. 풀스크린 가정 없음.
- **이해 가능성**: 자연 현상 비유라 첫 만남에서도 직관 작동. 단면 인셋이 longest prefix match 를 explicit 하게.
- **시각화 안 설명 포함**: §12 의 짧은 텍스트.
- **표준 대비 가치**: §11 의 5 가지 통찰.
- **다르기 위한 다름인가**: 등고선 깊이 = prefix 구체성, 빗방울 부피 = TTL 의 자연 비유. 단순히 모양만 바꾼 것이 아니라 **차원 변환**(2D 노드+엣지 → 2.5D 등고선 풍경) 이 이루어졌다.

### 위험과 사용자 평가 시 주목할 포인트

1. **등고선 그래픽의 구현 난이도** — 본 기획에서는 "구현 가능성 미고려" 원칙이지만, FACET 의 SVG 기반 Layout 자산으로 자연스러운 등고선을 만드는 것이 표준 격자/노드 자산보다 어려울 수 있다. 정적 등고선 패스 5~7 세트 + 빗방울 따라가는 베지에 곡선으로 충분히 표현 가능하리라 본다.
2. **"지형 전체가 보이는" 착시** — §10 의 hop-by-hop 보존 방어책(현재 봉우리만 선명) 이 충분히 강하지 않으면 학습자가 "전체 경로가 미리 정해진 것" 으로 오해할 수 있다. 회색조 처리의 강도가 중요한 디자인 변수.
3. **prefix 길이 = 등고선 깊이 매핑의 자의성** — /16, /24, /32 가 이산적인데 등고선 깊이는 연속적이다. 이산-연속 차이가 직관에 거슬릴 수 있음. 단면 인셋이 이 차이를 보정한다(이산 V 자 골짜기 명시).
4. **표준 비교 학습자에게의 낯섦** — Packet Tracer 등으로 이미 학습한 사람에게는 "왜 그래프가 아니지?" 의 저항이 있을 수 있다. 챕터 도입부에 표준 그래프 다이어그램을 한 번 보여 준 뒤, 이 메타포를 "다른 각도" 로 제시하는 글 흐름 권장.
5. **단면 인셋의 정보 밀도** — 80px 높이에 5 개 봉우리의 후보 등고선 단면을 다 보여 주려면 좁다. **현재 봉우리 1 개의 단면만** 보여 주는 것으로 좁히는 것이 옳다(이미 그렇게 설계).

### 사용자 평가 시 주목할 포인트

- 비전공자 학습자가 "왜 가장 깊은 골짜기가 이기는가" 를 별도 설명 없이 이해하는가 — 이게 메타포의 핵심 시험.
- §10 의 "회색조 처리"(현재 봉우리만 선명) 가 hop-by-hop 의 본질을 정말 전달하는가, 아니면 그저 시각적 장식인가.
- 단면 인셋이 등고선 풍경의 "예술적 모호함" 을 깨고 정확한 prefix 길이 비교를 책임지는가.
- 등고선 메타포가 IPv6 (128비트 prefix, /48, /64 등) 에도 자연스럽게 확장되는지 — 깊이 차원이 더 연속적이 되어 오히려 더 어울릴 수 있다는 가설.
