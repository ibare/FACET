# 컨텍스트 스위칭 핵심 추출

> 입력 — 분류: type=systemBehavior, 진행 모델=시간 진행형, 핵심 학습 질문="CPU 가 한 실행 흐름의 상태를 어떻게 저장하고 다른 실행 흐름의 상태를 복원해야, 두 흐름이 각자 멈춘 지점부터 정확히 이어서 실행될 수 있는가". 레퍼런스 9건 (OSTEP, Silberschatz, Tanenbaum, MIT 6.S081, Wikipedia, GeeksforGeeks, Computerphile, Process Scheduling Visualizer, 한국어 wikidocs/위키백과) 의 공통 시각 언어와 7개 공백을 종합.

## 1. 반드시 다뤄야 할 본질

학습자가 빠뜨리면 시각화가 컨텍스트 스위칭을 거짓으로 보여주게 되는 것.

1. **단 하나의 무대 — 한 시점에 오직 하나의 흐름만 CPU 위에 있다.** 두 개 이상의 실행 흐름이 동시에 그려져 있더라도, 그중 단 하나만이 굵은 선·채워진 띠·살아 있는 색으로 진행 중이고 나머지는 모두 정지된 상태로 보여야 한다. "CPU 자원 = 단일" 이라는 사실이 매 순간 시각적으로 강제되지 않으면 스위칭의 존재 이유 자체가 무너진다.
2. **컨텍스트의 외부 보관소 — 흐름 바깥에 자기 자리 한 칸씩이 있다.** 각 실행 흐름에는 자기만의 보관 칸(PCB·커널 스택 등 어떤 형태든)이 흐름 본체와 분리된 자리로 그려져, 거기에 register 묶음이 들고 나는 것이 시각적 단위가 된다. 보관소가 흐름 안에 그려지면 "꺼냈다 넣는다" 의 운동이 사라진다.
3. **저장-복원의 한 쌍이 한 번의 스위치를 만든다.** 한 흐름의 상태를 보관소에 떠 넣는 운동과 다른 흐름의 보관소에서 상태를 끌어와 CPU 자리에 채우는 운동이 반드시 짝으로, 둘 사이에 짧은 정지 구간을 두고 보여야 한다. 두 운동 중 하나만 그리거나 둘을 한 화살표로 합쳐 그리면 사건의 구조가 거짓이 된다.
4. **이어서 시작 — 멈춘 자리에서 정확히 재개된다.** 복원이 끝난 직후 다시 굵어지는 띠가 새로운 시작이 아니라 "그 흐름이 이전에 멈춘 그 지점에서 이어진다" 가 시각적으로 식별되어야 한다. 두 번째 등장이 처음과 시각적으로 연속됨이 보이지 않으면 컨텍스트를 저장하는 이유가 사라진다.
5. **외부 트리거 — 스위치는 흐름 자신이 결정하지 않는다.** 진행 중인 흐름을 끊는 사건이 흐름 바깥에서 들어오는 표식(번개·외부 화살표·별표 등 어떤 어휘든 외래성)으로 그려져, "흐름은 스스로 멈추지 않는다" 가 형상으로 보여야 한다. 트리거가 흐름 내부 사건처럼 보이면 선점의 본질이 무너진다.

## 2. 강화해야 할 요소

표준 레퍼런스가 약하게 다루는 것을 이 시각화의 차별점으로 끌어올린다.

1. **저장과 복원의 비대칭 운동.** 저장은 "지금 CPU 자리에 있는 값을 그대로 떠내는" 일방적 캡처지만, 복원은 "보관소에 있는 값을 의도된 자리에 정확히 되돌려 넣는" 정렬 운동이다. 두 운동의 시각 어휘가 거울처럼 대칭이 아니라 다른 무게로 보여야, 둘이 같은 일이 아님이 즉시 인지된다.
2. **사이의 빈 시간 — 누구도 일하지 않는 구간.** 저장이 끝난 시점부터 복원이 시작되기까지 두 흐름이 모두 정지해 있는 짧은 빈 구간이 시각적으로 자리를 차지해야 한다. 그 구간이 길어질수록 오버헤드가 커진다는 사실이 막대 길이나 숫자가 아니라 "아무것도 진행되지 않는 시간의 폭" 자체로 보인다.
3. **사건의 내부 단계 — 한 점이 아니라 세 박자.** 표준 자료는 스위치를 한 개의 화살표나 한 개의 짧은 막대로 압축한다. 이 시각화는 트리거 도착 → 저장 → (스케줄러 결정) → 복원 → 재개의 세 박자가 한 사건 안에서 순차로 식별되도록 펼쳐, 스위치가 사건이지 점이 아님이 운동의 형태로 보인다.
4. **컨텍스트 양의 차이 — 무엇이 옮겨지고 무엇이 남는가.** 같은 프로세스 안의 두 스레드 사이를 옮길 때 보관소로 떠내는 묶음이 얇고, 서로 다른 프로세스 사이를 옮길 때 떠내는 묶음이 그보다 두꺼워, 박스 부피·이동량의 차이로 두 스위치가 같은 비용이 아님이 시각적으로 즉시 구분된다. 표준 자료는 둘을 같은 도식으로 그려 차이를 지운다.
5. **권한 영역의 전환 — 사용자 모드와 커널 모드.** 흐름이 진행하는 동안의 배경과 스위치 구간 동안의 배경이 시각적으로 다른 영역으로 식별되어, "스위치는 사용자 흐름 안에서 일어나지 않고 잠시 다른 권한 영역으로 넘어가서 일어난다" 가 띠 위치 자체로 보여야 한다. 표준 자료는 같은 평면에 모든 사건을 그려 권한 전환을 잃는다.

## 3. 공백 활용 후보

레퍼런스의 공백 중 흥미로우나 베이스에는 채택하지 않는 것. 미래 창의 사이클의 출발점.

1. **스위치 빈도와 quantum 의 트레이드오프 시각.** 사용자가 시간 조각 길이를 줄이면 스위치 사건이 빈번해지고 빈 구간의 비율이 화면 전체에서 출렁이며 늘어나는 트레이드오프 모드. "더 자주 바꿀수록 더 적게 일한다" 가 운동 비율로 직접 보인다.
2. **캐시·TLB 무효화의 사후 효과.** 스위치 직후 한동안 새 흐름의 진행 속도가 느려지고 점차 회복되는 "워밍업" 운동. 본문 텍스트로만 언급되는 사후 비용이 스위치 직후의 띠 두께·속도 변화로 시각화된다.
3. **멀티코어 — 코어 간 마이그레이션.** 단일 무대가 아니라 여러 무대가 동시에 돌아가는 화면에서, 어느 흐름이 어느 무대에 올라가는가의 결정과 무대 간 이동 비용이 운동으로 보인다. 단일 CPU 가정의 한계를 넘는 시각.
4. **트리거의 다양성을 분기로.** 타이머·시스템 호출·I/O 완료·페이지 폴트가 모두 같은 외래 표식으로 그려지지 않고, 트리거의 종류에 따라 외부 화살표의 출발지·색·운동이 분기되어 "왜 스위치가 일어났는가" 가 시각적으로 구별되는 모드.
5. **scheduler 라는 가운데 무대.** 두 흐름이 서로의 코드를 직접 부르지 않고 가운데에 자리한 별도 흐름(스케줄러)을 거쳐서만 만나는 양단 구조. 저장과 복원 사이에 끼어 있는 결정의 자리가 사건의 한 단계로 명시된다.
6. **베껴 쓰기의 물리성.** Computerphile 의 손글씨 메타포처럼 register 묶음이 한 번에 옮겨지는 묶음이 아니라 칸 하나하나가 순차로 옮겨지는 운동으로 그려져, "복사" 가 추상이 아닌 물리 행위임이 사건의 길이로 체감되는 모드.
7. **트리거 없는 자발적 양보 — 협조형 스위치와의 대조.** 외부 트리거 없이 흐름이 스스로 양보하는 두 번째 스위치 유형이 한 화면에서 비교되어, 선점형과 협조형이 같은 "저장-복원" 골격을 공유하면서도 트리거의 출처만 다르다는 사실이 대조 시각으로 드러난다.
