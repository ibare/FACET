# Pub/Sub 핵심 추출

> 입력 — 분류: type=systemBehavior, 진행 모델=메시지 시퀀스형, 도메인=messaging, 핵심 학습 질문="발행자와 구독자가 서로를 모르는 채로 토픽을 매개로 메시지가 어떻게 다대다 비동기로 전달되는가". 레퍼런스 10건 (EIP, Azure, AWS SNS/SQS, Aiven Kafka, SoftwareMill Kafka, RabbitMQ Fanout, Redis Pub/Sub & MQTT, GoF Observer UML, 우아한형제들 카프카, 한국어 위키/블로그) 의 공통 시각 언어와 10개 공백을 종합. 베이스의 메시지 흐름은 가장 표준적인 **단일 토픽 + 다중 publisher + 다중 subscriber** 를 기준으로 고정한다 — 토픽 격리는 강화 5번에서 두 토픽 병렬의 보조 시각으로만 짧게 다룬다. 결정 근거: 단일 broker · 단일 토픽은 학습 부하가 가장 낮으면서 "익명성 + 팬아웃 + 시간 디커플링" 세 본질을 동시에 운반할 수 있는 최소 무대고, 다대다 publisher 측을 함께 그려야 패턴의 정체가 거짓이 되지 않는다.

## 1. 반드시 다뤄야 할 본질

학습자가 빠뜨리면 시각화가 Pub/Sub 을 거짓으로 보여주게 되는 것.

1. **broker 가 사이를 막는다 — publisher 와 subscriber 사이의 직접 화살표 부재.** 메시지가 publisher 에서 subscriber 로 직행하는 선이 한 줄도 보이지 않아야 한다. 모든 흐름은 가운데 broker(토픽) 박스를 반드시 거치고, 그 박스의 좌측 경계와 우측 경계가 두 세계를 분리한다는 사실이 위치만으로 강제되어야 한다. publisher 는 우측의 누구도 만질 수 없고, subscriber 는 좌측의 누구도 만질 수 없다.
2. **한 발행 → N 사본의 팬아웃.** publisher 한 명이 토픽에 메시지를 한 번 던지는 사건이, 토픽을 거쳐 모든 subscriber 에게 동시 다발로 사본을 뿌리는 운동으로 펼쳐져야 한다. 화살표 한 가닥이 들어와 부채꼴 N 가닥으로 갈라지는 형상이 한 사건의 시각적 핵이며, 사본은 각 subscriber 에게 한 번씩 도착한다 — 누구는 받고 누구는 못 받는 것이 아니다.
3. **시간 디커플링 — 발행 시각과 수신 시각의 분리.** publisher 가 던진 순간과 각 subscriber 가 받는 순간이 서로 다른 시점이라는 사실이 한 메시지의 운동 안에서 시각으로 보여야 한다. broker 안에 잠시 머무는 정지 구간, subscriber 마다 다른 도착 지연이 정적인 동시성 그림이 아니라 시간 위에 펼쳐진 비동기로 운반되어야 한다.
4. **다대다 — publisher 도 N, subscriber 도 N.** 발행자가 한 명이 아니라는 사실이 그림에 명시되어야 한다. 여러 publisher 가 같은 토픽에 메시지를 섞어 쏘고, 여러 subscriber 가 그 흐름을 모두 받는 구조가 한 화면에 있어야 한다. publisher 1 → subscriber N 만 그리면 패턴의 절반이 사라진다.
5. **토픽 라벨이 라우팅을 정한다 — 라벨로만 연결된다.** broker 가 가운데 있는 이유는 토픽이라는 라벨 한 줄이 "누구에게 갈지" 를 결정하기 때문이다. publisher 도 subscriber 도 상대의 신원이 아니라 토픽 이름만 알고 행동한다는 사실이, 양측 박스에서 broker 로 향하는 연결의 라벨이 토픽 이름이라는 사실로 보여야 한다. 신원이 아니라 라벨이 매개다.

## 2. 강화해야 할 요소

표준 레퍼런스가 약하게 다루는 것을 이 시각화의 차별점으로 끌어올린다. 레퍼런스 보고서의 10개 공백 중 베이스에 적합한 5개를 선별한다.

1. **두 시간축의 직교 배치 — 가로는 참여자, 세로는 시간.** 대부분의 표준 broker 도식이 가로 한 축에 모든 시간을 압축하는 한계를 깨고, UML 시퀀스 다이어그램의 세로 시간축을 broker 가 있는 도식 위에 결합한다. 위쪽에 publisher / broker / subscriber 박스가 가로로 늘어서고, 각자 아래로 라이프라인이 뻗으며, 발행과 수신이 가로 화살표로 시간 위를 가로지른다. broker 도식과 시퀀스 도식의 통합은 레퍼런스 거의 부재.
2. **"서로를 모른다" 의 시각적 강제 — 좌·우 두 영역의 가시적 차단.** broker 박스의 존재만으로 익명성을 함의하지 않고, publisher 영역과 subscriber 영역 사이를 불투명 벽·점선 띠·다른 배경 톤으로 명시 분리한다. 학습자의 시선이 좌측에서 우측으로 직접 건너뛸 수 없게 만들고, 모든 화살표는 반드시 broker 라이프라인 위의 한 점을 통과한다. 익명성이 그림 자체의 위상으로 강제된다.
3. **메시지 정체성의 보존 — 같은 색·같은 번호로 사본 추적.** 한 메시지가 N 명에게 팬아웃될 때 모든 사본이 같은 색·같은 번호 라벨을 유지해 "같은 메시지의 사본" 임이 시각적으로 추적된다. 다중 publisher 가 섞어 쏠 때도 publisher 별 색 차이로 출처가 구분되며, 학습자는 "이 칩과 저 칩이 같은 발행" 인지 한눈에 안다. 단순 색 칩 N 개의 갈라짐만으로는 사본 의미가 약하다는 공백의 보완.
4. **구독자의 동적 합류·이탈 운동.** 시뮬레이션 도중 새 subscriber 가 우측 영역에 등장해 토픽으로 가는 구독선을 그어 운동에 합류하고, 기존 subscriber 가 구독선을 끊고 사라지는 운동이 사건으로 발화한다. 합류 이전의 메시지는 새 subscriber 가 받지 않는다는 사실이 시간축 위에서 자연스럽게 드러난다. 정적 N 명 도식의 한계를 깨는 강화.
5. **다중 publisher × 다중 subscriber 의 다대다 운동, 토픽 격리는 보조.** 베이스는 단일 토픽 위에서 publisher M 명이 메시지를 섞어 쏘고 subscriber N 명이 모두 받는 다대다 운동을 정면으로 그린다. 보조 시각으로 화면 일부에 두 토픽이 병렬로 놓여 한 메시지가 자기 토픽의 구독자에게만 가고 다른 토픽 구독자에게는 가지 않는 격리를 짧게 대비한다. 깊은 토픽 라우팅·와일드카드는 별도 사이클로 미루고, 베이스에서는 다대다와 격리의 모양만 명시한다.

## 3. 공백 활용 후보

레퍼런스의 공백 중 흥미로우나 베이스에는 채택하지 않는 것. 미래 창의 사이클의 출발점.

1. **fanout vs 라운드로빈 (Pub/Sub vs 메시지 큐) 의 동시 운동 비교.** 같은 입력 메시지 시퀀스를 팬아웃 토픽과 경쟁 컨슈머 큐에 동시에 흘려, "모두에게 사본" 과 "한 명에게만" 의 차이가 출력 운동의 차이로 즉시 드러나는 반사실 시각.
2. **at-least-once / at-most-once / exactly-once 의 시각화.** 같은 도식 위에서 보장 수준 토글에 따라 중복 칩 등장, 손실 칩 사라짐, 중복 제거 X 마크 같은 사건이 운동으로 펼쳐진다. 보장 수준이 텍스트가 아닌 메시지 칩의 운명으로 운반되는 시각.
3. **백프레셔와 미처리 누적의 정서적 압박.** 구독자가 못 따라잡을 때 broker 안에 메시지가 쌓이며 색·두께·진동으로 "broker 가 답답해진다" 를 운반하는 시각. 임계 근방의 긴장감을 사건성으로.
4. **토픽 계층과 와일드카드.** MQTT 식 `home/livingroom/temperature` 같은 슬래시 트리 위에서 `home/+/temperature` 또는 `home/#` 구독이 어떤 메시지를 잡고 어떤 메시지를 흘려보내는지를 트리 운동으로 보여주는 별도 시각.
5. **메시지 페이로드의 변환 사슬.** broker 와 subscriber 사이에 변환 단계가 끼어 페이로드가 형태를 바꿔 가는 운동(이벤트 변환·필터·집계). 단순 전달이 아닌 흐름 처리로의 외연 확장.
6. **파티셔닝과 순서 보장.** 단일 토픽 안에 파티션이 나뉘어 같은 키의 메시지가 같은 파티션 위에서만 순서가 보장되는 운동. SoftwareMill 의 오프셋 슬라이딩을 더 깊게 끌어오는 별도 사이클.
7. **미수신 구독자의 사건성.** 오프라인 구독자가 영영 받지 못하는 메시지가 X 마크 한 글자가 아니라 "이 자리에 도착할 수 없었다" 의 정서적 시각으로 운반되는 보강. 거꾸로 보관형의 "기다리는 메시지의 답답함" 도 함께.
