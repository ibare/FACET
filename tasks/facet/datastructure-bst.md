# 이진 탐색 트리 (BST) 시각화 기획

### 0. 기본 정보

- **대상 이름**: 이진 탐색 트리 (Binary Search Tree, BST)
- **도메인/카테고리**: 컴퓨터 과학 기초 / 자료구조 / 순서 있는 연관 자료구조
- **대상의 본질**: 모든 노드에 대해 **"좌측 서브트리의 모든 값 < 노드 값 < 우측 서브트리의 모든 값"** 이라는 단 하나의 공간적 불변식으로, 값의 크기 관계를 구조의 좌우 위치에 영속적으로 새긴 자료구조.

### 1. 시그니처 행동

루트에서 출발해 **매 노드마다 세계의 절반을 포기한다** — 검색 키가 노드 값보다 작으면 우측 서브트리 전체를, 크면 좌측 서브트리 전체를 포기하고 남은 절반 속으로만 내려간다. 이 "반을 버리는" 단절적 선택의 연쇄가 트리 높이만큼 반복되면, 정렬된 배열을 한 줄로 훑는 대신 접힌 공간을 대각선으로 꿰뚫는 경로가 남는다.

### 2. 시각적 은유

- **갈림길에서 반대편을 접는 종이지도** — 일상 경험. 갈 곳이 정해지면 안 가는 쪽은 접어 덮어 버리고, 남은 반만 펼쳐 들여다본다. "반을 버린다" 는 BST 의 심장을 가장 직설적으로 표현.
- **수평선 위에 세로로 접힌 정렬된 수열** — 수학적 은유. 중위순회(inorder) 하면 정렬된 수열이 되는 성질을 "2D 평면 위로 접혀 있는 1D 정렬 배열" 로 형상화. 트리 아래에 바닥 정렬선을 깔아 이 접힘을 즉시 보이게 한다.
- **저울의 기울기** — 기계적 동작. 각 노드를 지탱하는 좌/우 서브트리의 무게 차이가 커질수록 트리가 *기울어진다*. 편향(skew) 의 경고음을 시각으로 번역.

### 3. 결정적 순간

**비교 1회, 반을 접어 버리는 "폴드(fold)" 순간.**

- **전**: 현재 커서가 어떤 노드에 도착해 있고, 좌/우 서브트리가 **둘 다 정상 채도**로 살아 있다. 도착한 노드와 검색/삽입 키가 상단 비교창에 나란히 놓인다.
- **중**: 비교 결과가 한 tick 안에 판가름 난다. 패배한 쪽 서브트리 전체가 **세로축을 중심으로 안쪽으로 살짝 말리며**(가로 scale 0.92, 저채도 + 알파 0.35) 한 단계 뒤로 밀려나고, 그 위에 옅은 회색 사선 해치가 덮여 "버려짐" 을 명시한다. 승리한 쪽 서브트리는 제자리에 남지만 경로 간선이 선두 색으로 한 번 pulse.
- **후**: 커서는 승리한 방향의 자식 노드로 한 단계 이동. 직전에 접힌 서브트리는 접힌 채로 남아, 탐색이 진행될수록 화면의 절반 이상이 점진적으로 어둡게 "포기 영역"으로 쌓인다. 검색이 끝난 시점의 프레임 한 장만 봐도, **살아남은 경로 + 접혀 버린 다수의 서브트리**라는 BST 고유의 모습이 드러난다.

시각적 강조: "fold" 는 **한 tick 에 완료되는 짧은 140ms curl 애니메이션** 이다. 서서히 페이드 아웃이 아니라, 종이접기처럼 *한 번에 딱* 접히는 것이 핵심. 이 단호함이 BST 가 매 비교에서 정확히 절반을 확정적으로 버린다는 사실을 몸으로 전달한다.

### 4. 다른 대상과의 구별점

- **vs 일반 이진트리 (structural binary tree)**: 일반 이진트리는 좌/우의 *의미가 없다*. BST 는 좌우에 **크기 대소** 의미가 올라탄다. 시각적으로 BST 는 (a) 좌 서브트리 영역 배경에 차가운 푸른 기(hue~210), 우 서브트리 영역 배경에 따뜻한 주황 기(hue~25) 의 옅은 음영을 깔아 "좌소우대(左小右大)" 를 공간으로 주장하고, (b) 캔버스 하단에 inorder 바닥선을 항상 띄워 "트리에서 왼쪽인 것은 바닥선에서도 왼쪽" 이라는 일관성을 증명한다. 일반 이진트리 시각화에는 둘 다 없다.
- **vs Heap (이진 힙)**: Heap 은 부모-자식 사이의 대소만 말하고 *형제 간엔 침묵한다*. 또한 Heap 은 구조적으로 항상 거의-완전 이진트리로 레이아웃이 고정된다. BST 는 (a) 형제/서브트리 전체에 대소가 걸리므로 inorder 바닥선으로 "모든 노드의 순위가 좌→우로 배치됨"이 드러나야 하고 — Heap 에는 이 바닥선이 없다. (b) 삽입 순서에 따라 트리 모양이 크게 달라져 **기울기 경고 배지** 가 의미를 갖는다 — Heap 은 언제나 꽉 찬 삼각형이라 기울기가 존재하지 않는다.
- **vs AVL / Red-Black 트리**: AVL/RB 는 회전으로 높이를 강제로 균형 맞춘다 — 그들의 시각화는 "rotation 애니메이션" 이 주연이다. BST 에는 rotation 이 없다. 그래서 BST 시각화의 주연은 **포기되는 서브트리의 누적**과 **편향이 일어나는 순간의 기울기** 다. 편향 preset 을 일부러 제공해 "이래서 AVL/RB 가 태어났다" 는 맥락을 체감시킨다.

### 5. 달성해야 할 시각 요소

- **이름**: 좌소우대 색지(色地)
  - **설명**: 트리의 모든 노드가, 자신이 어느 서브트리에 속해 있는지에 따라 배경의 색지(色地)로부터 "왼쪽이면 작다, 오른쪽이면 크다" 라는 약속을 자동으로 입는다.
  - **구현 힌트**: 각 노드가 루트로부터 내려온 방향 시퀀스(L/R 문자열)를 계산. 서브트리 영역의 바운딩 폴리곤을 노드 좌표들로부터 볼록 외곽(convex hull) 으로 추정해 매우 옅은(alpha 0.05~0.1) HSL 음영으로 채운다 — 좌 서브트리 계열은 hue 200~220 한색, 우 서브트리 계열은 hue 20~35 난색. 깊이가 깊어져 좌/우가 중첩된 영역에서는 색지끼리 자연스럽게 섞이도록 multiply blend. 노드 원 자체 색과는 독립된 **배경 레이어**.

- **이름**: 반을 접는 폴드 (fold-out)
  - **설명**: 비교에서 패배한 서브트리가 종이접기처럼 **한 번에 딱 접혀** 무대 뒤로 물러난다.
  - **구현 힌트**: algorithm 이 비교 결과를 `compare-step` 이벤트로 emit(현재 노드 id + 승리 방향 `'L' | 'R'`). projector 는 승리 방향의 반대 서브트리에 속한 모든 노드/간선 id 를 수집해 한 번에 CSS `transform: scaleX(0.92) translateX(±offset)` + `filter: saturate(0.3) brightness(0.7)` + 해치 오버레이 SVG `<pattern>` 를 한 프레임에 적용. 지속 시간 140ms ease-in-out. 서서히가 아니라 단호하게. 이후 접힌 상태는 누적돼 **끝날 때까지 유지** — 경로가 끝난 뒤 한 장면으로 결과를 읽을 수 있어야 한다.

- **이름**: 경로 조명 (path spotlight)
  - **설명**: 루트에서 커서가 내려온 비교 경로만이 밝게, 그 외 살아남은 미방문 영역은 평소 채도보다 낮게 유지된다.
  - **구현 힌트**: 현재까지 방문한 노드 집합을 projector 가 보유. 방문 노드 원의 stroke 를 선두 색(예: hue 45 골드) 으로 2px 굵게, 방문 간선은 같은 색 실선. 방문하지 않았지만 **접히지도 않은** 노드 — 즉 아직 고려 대상인 절반 — 은 평소 색에서 채도 -20% 로 한 단계 낮춰 "무대 밖 후보" 임을 표시. 접힌 서브트리와 혼동되지 않도록 해치 패턴은 사용하지 않음(해치는 "접힘" 전용).

- **이름**: 정렬된 바닥선 (inorder mirror)
  - **설명**: 캔버스 아래의 가로 띠에 현재 트리의 중위순회 결과가 항상 왼→오 정렬된 수열로 재현된다. 트리의 구조적 질서가 곧 1D 정렬 질서임이 실시간으로 증명된다.
  - **구현 힌트**: 노드가 추가/삭제될 때마다 inorder walk 를 다시 돌려 바닥선을 갱신. 바닥선의 각 셀은 트리 캔버스의 해당 노드와 **수직 점선 가이드** 로 연결. 검색/삽입 진행 중에는 폴드로 버려진 구간의 바닥선 셀을 같은 해치/저채도로 동기화해, "트리에서 반을 접으면 바닥선에서도 같은 구간이 접힌다" 는 등가성을 시각적으로 드러낸다. 중복 값은 입력 자체를 금지(삽입 시 reject + 메트릭 증가)해 시각적 모호성을 원천 차단.

- **이름**: 기울기 배지 (skew gauge)
  - **설명**: 트리가 한쪽으로 편향될 때, 시각적으로 돌출된 외형과 함께 숫자 지표가 "얼마나 균형에서 멀어졌는지" 를 말해 준다.
  - **구현 힌트**: 상단 좌측에 `h / log₂(n+1)` 비율 게이지(1.0 에 가까우면 균형, 2.0 에 가까우면 편향, ≥ 3.0 이면 사다리). 배경 바의 색이 초록 → 노랑 → 빨강으로 부드럽게 전이. 트리가 한쪽으로 기울면 레이아웃 자체도 루트를 중심에서 반대쪽으로 약간 평행이동(최대 ±60px) 시켜 **비대칭을 기하적으로 체감**. 편향 preset("정렬된 입력 그대로 삽입") 을 돌리면 이 배지가 단번에 붉어지며 레이아웃이 사다리로 늘어나는 장면이 바로 관찰된다.

### 6. 시각 구성

- **중심 시각**: 위→아래로 뻗는 이진 트리 캔버스. 루트는 상단, 리프는 하단. 각 노드는 값이 찍힌 원. 간선은 부모-자식 직선. 배경에는 **좌소우대 색지** 가 깔려 있어 트리가 아직 비어 있을 때도 "왼쪽은 작고 오른쪽은 크다" 라는 약속이 공간적으로 먼저 선언된다.
- **보조 요소들**:
  - **정렬된 바닥선**: 캔버스 하단 가로 띠. 중위순회 결과를 왼→오 순서로 표시. 트리 캔버스의 노드와 수직 점선 가이드로 1:1 연결.
  - **비교 헤드업**: 상단 중앙의 작은 창에 현재 비교가 `[키 48] ? [노드 35]` 형태로 표시되고, `<` 또는 `>` 결과에 따라 화살표가 즉시 좌/우로 튀어 나간다. 폴드 애니메이션의 **발화점** 역할.
  - **기울기 게이지**: 좌상단. `h / log₂(n+1)` 비율 + 편향 경고 색 전이.
  - **연산 콘솔**: 우측 또는 상단에 `insert / search / delete` 버튼과 키 입력창. 현재 진행 중 연산에는 버튼이 비활성.
  - **코드 패널** (우측 선택): BST 의 재귀/반복 구현 중 선택 가능. 현재 실행 라인은 비교 헤드업 + 폴드 애니메이션과 동기 하이라이트.
- **요소 간 관계**: 비교 헤드업이 판단을 내리는 순간, 그 결과는 (a) 트리 캔버스의 폴드 애니메이션, (b) 바닥선의 해치 동기화, (c) 코드 패널의 다음 줄 하이라이트, (d) 경로 조명 확장 — 네 곳에 **한 tick 안에 동시에 반영**된다. 이 동시성이 "비교 1회 = 절반 포기" 의 단호함을 학습자에게 각인시킨다.

### 7. 레이아웃

- 상단 ~10%: 비교 헤드업 + 기울기 게이지 + 연산 콘솔(키 입력, insert/search/delete 버튼).
- 중단 ~60%: 트리 캔버스. 루트는 상단 중앙. 좌/우로 공간을 대칭적으로 확보하되, 편향이 심해지면 ±60px 까지 평행이동 허용.
- 하단 ~15%: 정렬된 바닥선(inorder mirror). 트리 노드와 수직 점선으로 연결.
- 하단 ~10%: 재생/단계/속도 컨트롤 + 메트릭(비교 횟수, 현재 높이, 노드 수).
- 우측(선택): 코드 패널. 접어 두었다가 펼 수 있음.
- 캔버스 배경: 좌소우대 색지 — 루트 아래 좌/우 영역이 아주 옅은 한/난색 음영으로 양분되어, 트리가 비어 있어도 "공간의 약속" 이 먼저 보인다.

### 8. 진행과 이벤트

주요 진행 단계(검색/삽입 공통):

1. **시작**: 키 입력 + 버튼 클릭. 키가 루트 위 상공에 작은 "드롭 마커" 로 떠오른다. 루트에 도착 전에 비교 헤드업에 `[키] ? [루트값]` 이 채워진다.
2. **노드 도착**: 커서가 현재 노드 원 위에 정착. 원에 활성 stroke 가 입혀지고, 비교 헤드업의 물음표가 `<` 또는 `>` 로 해소된다.
3. **폴드(결정적 순간)**: 패배한 서브트리가 140ms curl 애니메이션으로 접히고, 바닥선의 해당 구간이 동시에 해치 저채도로 동기화, 코드 패널의 선택된 분기 라인이 하이라이트.
4. **이동**: 커서가 승리 방향의 자식으로 한 단계 내려간다. 해당 간선이 경로 조명에 편입. 2~4 반복.
5. **리프 도달(분기)**:
   - **검색 hit**: 커서가 값이 일치하는 노드에 도달하면 원이 한 번 pulse(확대 1.2 → 1.0), 경로 전체가 골드로 잠시 환해진다.
   - **검색 miss**: 자식이 없는 방향으로 내려가려는 순간, 그 빈 자리에 **회색 점선 유령 원** 이 한 번 찍혔다가 사라진다. 비교 헤드업에 "found = false".
   - **삽입**: miss 상황에서 유령 원이 사라지지 않고 그대로 **실체화**(stroke 실선 + 활성 색 pulse)되고, 바닥선에 새 셀이 올바른 정렬 위치로 슬라이드 인.
6. **삭제(별도 흐름)**:
   - **leaf**: 노드 원이 아래로 떨어지는 짧은 애니메이션 후 소실. 바닥선의 해당 셀이 자리를 비우며 양옆 셀이 간격을 메꾼다.
   - **자식 1개**: 남은 자식 서브트리 전체가 한 단계 위로 승격. 기존 노드 자리로 자식의 루트가 슬라이드 업.
   - **자식 2개(인오더 후계자 서브루틴)**: 별도 **보조 커서**(다른 색, 예: 보라) 가 우측 서브트리의 루트에서 출발해 왼쪽 자식을 따라 내려가는 미니 탐색을 수행 — 이 서브루틴 동안에는 폴드 애니메이션을 쓰지 않고 **보조 커서의 경로 조명만** 사용해 "삭제 본류" 와 혼동되지 않도록 한다. 후계자 노드의 값이 삭제 대상 노드로 이사(값 텍스트가 날아가듯 이동)하고, 원래 후계자 자리는 case 1 또는 case 2 의 보조 삭제로 정리된다.
7. **종료**: 연산이 끝나면 경로 조명/폴드는 기본 800ms 여운 뒤 해제되고, 다음 연산을 받을 수 있는 기본 상태로 복귀. 단, 사용자가 "pin" 을 누르면 마지막 경로/폴드 상태를 화면에 고정해 스냅샷으로 남긴다.

사용자 제어 지점:

- **재생/일시정지**.
- **단계 전진 해상도**: (a) `tick` (한 번의 비교), (b) `operation` (insert/search/delete 하나가 끝날 때까지), (c) `batch` (콘솔 큐에 쌓인 여러 연산 연속 실행).
- **속도 슬라이더**.
- **preset 드롭다운**: "균형 잡힌 삽입(기본)" / "정렬된 입력 → 편향 트리" / "무작위 삽입 + 검색 섞기".
- **연산 수동 입력**: 키 입력창 + insert/search/delete 버튼.

기본 단계 해상도는 `operation` — BST 는 연산 하나 단위가 학습의 자연스러운 호흡이다. 비교 하나하나를 보고 싶은 학습자는 `tick` 으로 전환.

### 9. 초기 상태와 파라미터

- **초기 데이터**:
  - 초기 트리: 중간 높이의 균형 잡힌 시드 — 예) 루트 50, 이후 30·70·20·40·60·80·10·35·55·75 순으로 삽입된 상태.
  - 시나리오(자동 재생 시): `search 35` → `search 42`(miss) → `insert 42` → `insert 65` → `delete 30`(자식 2개 — 후계자 서브루틴 시연) → `delete 80`(leaf) → `insert 5`.

  이 시나리오는 (a) hit / miss / insert / leaf 삭제 / 후계자 삭제 의 다섯 경로 유형을 한 바퀴에 모두 관찰할 수 있도록 의도적으로 배열한다.

- **파라미터**:
  - `preset`: `balanced`(기본) / `skewed-sorted`(편향) / `random-mix`.
  - `step mode`: `tick` / `operation`(기본) / `batch`.
  - `allow duplicates`: false 고정(중복 키는 삽입 시 reject 하고 `rejected-duplicate` 메트릭만 증가 — 시각적 모호성 제거).
  - `code paradigm`: `recursive` / `iterative`(코드 패널 토글).

## 부록: 기획 메모 (메타 서술)

- **수렴 방지 포인트**: BST 를 AI 로 생성하면 "노드에 숫자 박고 insert 시 회색 화살표가 내려가는" 평범한 d3 tree 시각화로 수렴한다. 본 기획은 그 지점을 버리고 **(1) 반을 버리는 폴드 = 단호한 140ms curl**, **(2) 좌소우대 = 공간의 약속을 배경 색지로 선언**, **(3) 공간적 순서 = 정렬된 바닥선 + 수직 가이드**, **(4) 편향의 경고 = 기울기 게이지 + 레이아웃의 물리적 기울기**, 네 축으로 BST 를 재정의했다. 이 네 축 중 하나라도 빠지면 일반 이진트리나 Heap 시각화와 구별되지 않는다.
- **Heap / AVL / 일반 이진트리와의 대비**는 각 요소의 *존재 유무* 로 설계되었다. 정렬된 바닥선은 Heap 시각화에 없다. 기울기 배지는 AVL 시각화에서 의미가 없다(항상 균형). 좌소우대 색지는 일반 이진트리에서 거짓말이 된다. 나란히 놓여도 BST 로만 읽혀야 한다.
- **구현 단계로 넘길 판단 사항(의도적으로 열어 둠)**:
  - (a) 좌소우대 색지를 **볼록 외곽 polygon** 으로 계산할지, 더 단순한 **서브트리 바운딩 박스** 로 근사할지 — 성능/가독성 트레이드오프.
  - (b) inorder 바닥선의 재렌더링을 **매 연산 끝 1회** 로 할지 **폴드 단위로 점진적** 으로 할지 — 학습자가 혼란을 느끼지 않는 경계.
  - (c) 후계자 서브루틴의 **보조 커서** 색을 어디까지 화려하게 할지 — 본류 폴드와 구별되면서도 시각 소음이 되지 않는 균형.
  - (d) 코드 패널의 재귀/반복 토글이 실시간 phase 라인 하이라이트를 둘 다 유지하려면, 두 IR 이 같은 phase 어휘를 공유해야 한다 — 이건 `rules/concerns/C3-phase-sync.md` 의 트리거 대상이라 구현 기획에서 먼저 phase 목록을 맞출 것.
  - (e) "레이아웃 평행이동으로 편향 체감" 기능은 첫 구현에서 비활성으로 두고, 기본 기울기 게이지만 먼저 안정화한 뒤 켜는 것이 안전.
- **새 View 가 필요한가?**: 트리 캔버스 본체는 기존 `tree-layout` 재사용 가능성이 높지만, **좌소우대 색지 + 폴드 애니메이션 + inorder 바닥선 + 기울기 게이지** 가 한 묶음으로 붙어야 BST 의 정체성이 선다. 이 묶음을 `tree-layout` 의 features 로 쌓을지, 새 `bst-stage` View 로 분리할지는 구현 단계의 **첫 번째 설계 결정** 으로 올릴 것(principles.md §6: "둘 이상의 facet 이 공유할 수 있을 때만 새 View").
