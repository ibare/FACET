# FACET 시각화 기획

사용자가 지시한 시각화 대상에 대한 기획서를 4단계 서브에이전트 파이프라인으로 생성한다. 메인은 오케스트레이터로 동작하고 직접 작업하지 않는다.

기획서는 시각화의 본질만 담는다. 구현 아키텍처·메커니즘·코드 어휘 금지. 영구 자산이며 아키텍처가 바뀌어도 같은 기획서로 재구현 가능해야 한다.

---

## 시각화 환경

- 마크다운 글에 임베드되는 형태. 풀스크린 아님.
- 가용 폭 640~720px (확장 880px)
- 시각화 안에 짧은 개념 설명(2~4문장) 영역 확보
- 좌우 4~6 패널은 깨짐. 핵심 + 보조 1~2개가 적정.

---

## 파이프라인

```
Stage 0  분류        → classification.md  (작업 자료)
Stage 1  레퍼런스    → references.md       (작업 자료)
Stage 2  핵심 추출   → essence.md          (영구)
Stage 3  기획서      → <대상>.md           (영구, 최종)
```

---

## Stage 0 — 분류

```
시각화 대상을 분류한다. 검색·기획 안 한다.

## 결정

1. type: algorithm | procedure | protocol | dataStructure | systemBehavior | concept | transformation
2. 진행 모델: 시간 진행형 | 입력 반응형 | 메시지 시퀀스형 | 정적 | 혼합
3. 도메인: 영문 소문자 kebab-case
4. 핵심 학습 질문: 한 문장

## 출력

저장: tasks/facet/<도메인>-<대상>-classification.md

섹션:
## 1. 대상 정보
## 2. type
## 3. 진행 모델
## 4. 핵심 학습 질문

채팅에는 경로 + 결정 한 줄.
```

---

## Stage 1 — 레퍼런스 분석

```
시각화 대상에 대한 세상의 시각화 사례를 조사한다.
분류 보고서를 입력으로 받아 검색에 반영한다.
기획·코드는 작성 안 한다.

## 작업

1. 검색 — 5~10개 레퍼런스
   - 학술/표준, 인기 해설, 독창적 시각화, 다른 분야 시각 표현
   - 영어·한국어 모두
   - URL 필수

2. 각 레퍼런스 요약
   - 출처, URL, 한 줄 정체성
   - 시각 객체, 시간 표현, 메타포, 강조점, 인상 디테일, 약점

3. 공통 요소 — 거의 모든 레퍼런스가 공유하는 시각 언어

4. 공백 — 레퍼런스가 다루지 않는 측면

5. 대상 개념 요약 — 2~4문장

## 출력

저장: tasks/facet/<도메인>-<대상>-references.md

섹션:
## 1. 검색 결과
## 2. 레퍼런스 요약
## 3. 공통 요소
## 4. 공백
## 5. 대상 개념 요약

채팅에는 경로 + 3줄 요약.

## 입력

분류: {{stage0_경로}}
대상: {{대상명}}
```

---

## Stage 2 — 핵심 추출

```
레퍼런스 분석을 정제하여 시각화의 정체성을 확정한다.
출력은 영구 자산이며 기획서의 직접 입력이다.

## 핵심 원칙

베이스 시각화는 표준 강화 방향이다. 창의적 메타포는 베이스에 넣지 않는다.
LLM이 학습 콘텐츠에서 이 시각화를 호출할 때 추가 설명 부담 없이 사용 가능해야 한다.
창의 버전은 별도 사이클에서 추가될 영역이다.

## 추출 항목

### 1. 반드시 다뤄야 할 본질 (3~5개)
학습자가 반드시 이해해야 할 것. 빠지면 시각화가 거짓이 되는 핵심.

### 2. 강화해야 할 요소 (3~5개)
표준 레퍼런스가 약하게 다루는 것. 이 시각화의 차별점.

### 3. 공백 활용 후보 (창의 사이클용)
레퍼런스 분석의 공백 중 흥미로운 것.
**베이스에는 채택하지 않는다.** 미래 창의 버전의 출발점.

본질·강화 요소는 "시각적으로 무엇이 보여야 하는가"로 작성. 구현 힌트 아님.

## 출력

저장: tasks/facet/<도메인>-<대상>-essence.md (영구)

섹션:
## 1. 반드시 다뤄야 할 본질
## 2. 강화해야 할 요소
## 3. 공백 활용 후보

채팅에는 경로 + 본질·강화 핵심 한두 줄.

## 입력

분류: {{stage0_경로}}
레퍼런스: {{stage1_경로}}
```

---

## Stage 3 — 기획서

```
essence 기반으로 최종 기획서를 작성한다.
정식 구현의 직접 입력이며 영구 자산이다.

## 원칙

- essence의 본질을 모두 다루고 강화 요소를 모두 반영한다.
- 공백 활용 후보는 채택하지 않는다.
- 임베드 폭 640~720px.
- 구현 아키텍처·메커니즘·코드 어휘 금지.
- 표준 시각 언어를 디테일과 완성도로 강화한다. 창의적 메타포 만들지 않는다.

## 기획서 구조

### 0. 기본 정보
- 대상 이름, 도메인, type, 진행 모델
- 대상의 본질 한 문장

### 1. 시그니처 행동
대상을 그것답게 만드는 핵심 동작 한 문장.

### 2. 시각적 은유
표준 메타포 채택·정교화.

### 3. 결정적 순간
"이게 그 대상이구나" 알아차리는 장면.
전/중/후 시각 상태, 강조 장치.

### 4. 다른 대상과의 구별점
가장 가까운 대상 2개 이상과의 시각 차이.

### 5. 달성해야 할 시각 요소 (3~5개)
essence의 본질과 강화 요소를 모두 포함.
각 항목: 이름 / 설명 / 시각 표현.

### 6. 시각 구성
중심 시각, 보조 요소, 요소 간 관계.

### 7. 레이아웃
640~720px 안에서의 배치.

### 8. 진행과 인터랙션
진행 모델에 따라:
- 시간 진행형: 주요 단계, 단계별 시각 변화, 제어 지점
- 입력 반응형: 입력 → 시각 갱신 매핑
- 메시지 시퀀스형: 참여자, 메시지 흐름
- 정적: 무엇이 어떻게 표시되는가
- 혼합: 조합

### 9. 초기 상태와 파라미터

### 10. 시각화 안 텍스트
- 대상 개념 설명 2~4문장 (references의 요약 다듬음)
- 학습자가 무엇을 보는지 안내

### 11. 참고 레퍼런스
시각화 UI에 표시될 레퍼런스 정보.
references의 레퍼런스 중 영감을 준 2~4개 선별.
각 항목: 이름 + URL + 한 줄 정체성.

## 자가 점검

- essence의 본질·강화 요소가 모두 반영되는가
- 시그니처 행동이 다른 대상과 혼동되지 않는가
- 결정적 순간이 시각적으로 강조 가능한가
- 임베드 폭에서 깨지지 않는가
- 시각화 안 짧은 설명이 포함되는가
- 참고 레퍼런스 항목이 명시되는가
- 구현 아키텍처 어휘가 들어가지 않았는가

## 출력

저장: tasks/facet/<도메인>-<대상>.md (영구, 최종)
- 섹션 제목 ### 1. 시그니처 행동 형태 유지 (구현 파서가 의존)
- 파일 이미 존재 시 사용자 확인 후 -v2 접미사

채팅에는 경로 + 시그니처 행동·결정적 순간·핵심 시각 요소 각 한 줄.

## 입력

분류: {{stage0_경로}}
레퍼런스: {{stage1_경로}}
essence: {{stage2_경로}}
```

---

## 메인 오케스트레이터

서브에이전트 4개를 순차 호출. 각 단계 완료 후 산출물 파일 존재 확인.

진행 보고: "Stage N ..." 한 줄씩.
각 서브에이전트의 채팅 응답은 그대로 전달.
본문 풀어쓰지 않음.

최종 보고:
- 영구 자산 2개 경로 (essence, 기획서)
- 작업 자료 2개 경로 (분류, 레퍼런스)

서브에이전트 실패 시 1회 재시도, 실패 시 사용자 보고.
